# Drizzle ORMで「型安全なDB設計」を実現する思想

TypeScriptエコシステムにおいて、ORMの選択は長らく「PrismaかTypeORMか」という二択だった。しかし近年、Drizzle ORMが急速に支持を集めている。Next.js App RouterやEdge Runtimeとの相性、そして「SQLに近い書き心地」が評価されているようだ。

本稿では、Drizzle ORMの設計思想と、それがNext.jsアプリケーションのDB設計に何をもたらすかを考える。

## ORMの「抽象化」とその代償

まず、ORMが解決しようとしている問題を整理しよう。

素のSQLは強力だが、TypeScriptとの橋渡しが難しい。クエリ結果の型が`any`になりがちで、カラム名のタイポがランタイムエラーになる。テーブル定義とアプリコードが乖離しやすい。

従来のORMはこれを「オブジェクト指向のモデル定義」で解決しようとした。テーブルをクラスとして定義し、クエリをメソッドチェーンで表現する。これは一定の問題を解決したが、別の問題を生み出した。

**N+1問題の隠蔽**: ORMが自動的にJOINを避けて別クエリを発行し、気づかないうちにN+1問題が発生する。

**SQLの喪失**: 複雑なクエリをORMの記法で表現しようとすると、SQLを直接書くより難しくなることがある。

**ランタイムの重さ**: 大きなORMランタイムはサーバーレス・Edge環境では不利だ。

## Drizzleの哲学「SQLに近く、型安全に」

Drizzle ORMは異なるアプローチを取る。スローガンは「SQL-like」——SQLに近い記法を保ちながら、TypeScriptの型安全性を実現する。

スキーマ定義を見るとその思想がよく分かる。

```typescript
import { pgTable, serial, text, timestamp, integer } from 'drizzle-orm/pg-core';

export const posts = pgTable('posts', {
  id: serial('id').primaryKey(),
  title: text('title').notNull(),
  content: text('content'),
  authorId: integer('author_id').notNull(),
  createdAt: timestamp('created_at').defaultNow(),
});
```

これはTypeScriptのオブジェクトでありながら、テーブル定義を正確に表現している。カラムの型、NULL可否、デフォルト値——すべてが型情報として扱われる。

クエリも同様にSQLライクだ。

```typescript
const userPosts = await db
  .select()
  .from(posts)
  .where(eq(posts.authorId, userId))
  .orderBy(desc(posts.createdAt))
  .limit(10);
```

SQLを書いたことがある人なら、このクエリが何をするか直感的に読める。そしてこの結果は適切に型付けされており、`userPosts`は`typeof posts.$inferSelect[]`型になる。

## PrismaとDrizzleの設計思想の違い

Prismaとの比較が参考になる。両者は同じ問題（TypeScript × DB）に対して、異なる哲学でアプローチしている。

Prismaは**宣言的スキーマ**を中心に置く。`.prisma`ファイルにスキーマを定義し、生成ツールがTypeScriptクライアントを生成する。クエリはPrismaの独自DSLで書く。抽象化のレベルが高く、SQLを書かなくても多くのことができる。

Drizzleは**TypeScriptファースト**だ。スキーマ定義からクエリまで、すべてがTypeScriptのコードとして存在する。外部ファイルなし、コード生成なし。クエリはSQLと対応関係が明確で、生成されるSQLが予測しやすい。

どちらが優れているかではなく、思想の違いだ。Prismaは「SQLの複雑さから解放されたい」チームに向いている。Drizzleは「SQLをコントロールしたい、型安全性も欲しい」チームに向いている。

## Next.js App RouterとDrizzleの相性

Next.js App Routerとの相性でDrizzleが注目される理由のひとつが、Edge Runtimeへの対応だ。

PrismaのクライアントはNode.js専用のバイナリを使用するため、Cloudflare WorkersやVercel Edge Functionsでは動作しない（Prisma Accelerateを使えば別だが）。Drizzleはランタイムに依存しない設計のため、Edge Runtimeでも動作する。

また、Server ComponentsでのDB直接呼び出しとも相性が良い。

```typescript
// app/posts/page.tsx (Server Component)
import { db } from '@/lib/db';
import { posts } from '@/lib/schema';

export default async function PostsPage() {
  const allPosts = await db.select().from(posts);
  return <PostList posts={allPosts} />;
}
```

型安全なクエリ結果がそのままコンポーネントのpropsに流れ込む。`any`を使わずに済み、カラム名のタイポは型エラーとして事前に検出される。

## マイグレーションの考え方

スキーマの変更管理（マイグレーション）もDrizzleの重要な機能だ。

Drizzleのマイグレーションは2つのアプローチがある。

**drizzle-kit generate**: 現在のスキーマ定義と前回の状態の差分からSQLマイグレーションファイルを自動生成する。生成されたSQLを確認・編集してから適用できる。

**push（開発時）**: スキーマの変更を直接DBに反映する。開発中のイテレーションを速くするためのもので、本番環境には不適切だ。

生成されるマイグレーションは素のSQLファイルだ。ORMの魔法ではなく、実際にどんなDDLが実行されるかが目で見える。これはPrismaのマイグレーションと同様の思想だが、Drizzleはさらにそのファイルを手動で編集することも容易だ。

## 型安全性が生む開発体験

Drizzleの型安全性が実際の開発でどう効くか、具体的に考えてみよう。

スキーマに`notNull()`を指定したカラムをinsert時に省略すると、TypeScriptエラーになる。カラム名を`title`から`headline`に変更すれば、そのカラムを参照しているすべての箇所でコンパイルエラーが出る。クエリ結果の特定のカラムだけを選択すれば、返り値の型も自動的に絞り込まれる。

これは「実行して確かめる」ではなく「書いた瞬間に分かる」開発体験だ。特に複数人でコードを書くチームでは、DBスキーマの変更が全体に与える影響がコンパイル時に明確になることの価値は大きい。

## いつDrizzleを選ぶか

新規プロジェクトでDrizzleを選ぶ基準を整理しよう。

**Drizzleが向いている場面**:
- SQLを書くことに慣れたチームで、型安全性も欲しい
- Edge RuntimeやサーバーレスでのDB接続が必要
- 生成されるSQLを細かく制御したい
- 外部ファイルや生成ステップを減らしたい

**Prismaが依然として向いている場面**:
- SQLよりも高レベルな抽象化を好むチーム
- Prisma Studioなどのエコシステムを活用したい
- 大規模なPrismaの実績・コミュニティを重視する

ツールの選択は思想の選択でもある。Drizzleが示す「SQLに近く、型安全に」という哲学は、TypeScriptとデータベースの関係の新しいあり方だ。

## まとめ

- DrizzeはSQL-likeな記法でTypeScriptの型安全性を実現するORM
- スキーマ・クエリがすべてTypeScriptコードとして存在し、外部ファイル・コード生成が不要
- PrismaとはDB抽象化の哲学が異なる（低抽象 vs 高抽象）
- Edge RuntimeやNext.js App Routerとの相性が高い
- マイグレーションは素のSQLファイルで管理され、予測可能
- 型安全性がDBスキーマ変更の影響をコンパイル時に可視化する
