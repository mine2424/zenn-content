# Next.jsにPasskeys認証を組み込む——パスワードレス時代のWebAuthn設計思想

パスワードは「知っているもの」だ。そして人間は秘密を管理するのが苦手だ。

使い回し、弱いパスワード、フィッシングによる奪取——パスワードベースの認証が持つリスクは、20年以上のインターネットの歴史が証明している。Passkeys（パスキー）はこの問題に対するアーキテクチャレベルの回答だ。

この記事では、WebAuthnとPasskeysの設計思想を整理し、Next.jsアプリケーションへの組み込み方を概観する。実装の詳細よりも「なぜこの設計なのか」という理解を深めることを目的とする。

---

## パスワードの何が根本的に問題か

パスワードの問題は「人が覚えられる範囲」と「安全性の要件」が相反することにある。

安全なパスワードは長く複雑で、サービスごとに異なるものを使う必要がある。これは人間の記憶容量を超えている。だからパスワードマネージャーが必要になり、マスターパスワードという新たなSPOFが生まれる。

そして最大の問題は「フィッシング」だ。偽サイトに本物そっくりのUIを作れば、どんなに教育されたユーザーでも騙されうる。パスワードは「見た目が正しければ入力してしまう」という人間の認知的限界に付け込まれやすい。

Passkeysはこの問題を「パスワードという秘密情報を人間が管理する」という構造そのものを変えることで解決しようとしている。

---

## WebAuthnとPasskeysの関係

用語を整理しておく。

**FIDO2**: FIDOアライアンスが策定したパスワードレス認証の仕様群。

**WebAuthn**: W3Cが標準化したWebブラウザ向けのAPIで、FIDO2の一部。ブラウザが公開鍵暗号認証を扱えるようにする。

**Passkeys**: GoogleとAppleとMicrosoftが推進する、WebAuthnをデバイス間で同期可能にした実装。iCloudキーチェーンやGoogleパスワードマネージャーを通じてバックアップ・同期される。

Passkeysは「WebAuthnをより使いやすく」したものだと理解すればよい。

---

## 公開鍵暗号認証の仕組み

Passkeysの核心は公開鍵暗号だ。

登録時：
1. デバイスがサーバーからchallenge（ランダムな値）を受け取る
2. デバイスで公開鍵・秘密鍵のペアを生成する
3. 公開鍵をサーバーに送る。**秘密鍵はデバイスを絶対に離れない**
4. 生体認証（Face ID、指紋等）でユーザーの存在を確認して完了

認証時：
1. サーバーがchallengeを送る
2. ユーザーが生体認証を行う
3. デバイスが秘密鍵でchallengeに署名して送る
4. サーバーが登録時の公開鍵で署名を検証する

フィッシングが無効化される理由がここにある。署名はWebAuthnのAPIが自動的に「どのドメインに対する認証か」を埋め込む（`rpId`というフィールド）。偽サイトからのchallengeに署名しても、本物のサーバーがそれを正しいと認めることは数学的に不可能だ。

## Next.jsへの組み込みアプローチ

Next.jsでPasskeys認証を実装する方法は主に2つある。

**1. ライブラリを使う（推奨）**

`@simplewebauthn/browser`（クライアント）と`@simplewebauthn/server`（サーバー）の組み合わせが代表的だ。WebAuthnの仕様を直接実装するのは複雑なので、ライブラリへの依存は合理的な判断だ。

```typescript
// Server Actions（または API Route）での登録開始
import { generateRegistrationOptions } from '@simplewebauthn/server';

export async function startRegistration(userId: string) {
  const options = await generateRegistrationOptions({
    rpName: 'My App',
    rpID: 'myapp.com',
    userID: userId,
    userName: 'user@example.com',
    attestationType: 'none',
    authenticatorSelection: {
      residentKey: 'required',     // Passkeys必須の設定
      userVerification: 'required', // 生体認証を要求
    },
  });
  
  // challengeをセッションに保存する（検証時に必要）
  await saveChallenge(userId, options.challenge);
  
  return options;
}
```

**2. 認証プロバイダーに任せる**

ClerkやAuth0はPasskeysを組み込みでサポートしている。実装の複雑さを完全に外部委託できる。プロバイダーに依存することのトレードオフを許容できるなら、この選択は合理的だ。

## 登録・認証フローの設計

フロー設計で押さえるべきポイントをいくつか整理する。

**challengeの管理**

WebAuthnはリプレイ攻撃を防ぐため、チャレンジ・レスポンス方式を使う。サーバーが発行したchallengeを適切な場所（Redis・DB・セッション）に保存し、検証後は必ず削除することが必要だ。有効期限（5分程度）も設定する。

**ユーザー体験の設計**

Passkeysはまだ全ユーザーが持っているわけではない。移行期には「Passkeysで続ける」と「パスワードでログイン」を並立させる設計が必要だ。パスワードを使うユーザーにはPasskeysへの移行を促すUXを設計する。

**エラーハンドリング**

ユーザーが生体認証をキャンセルした、デバイスに該当するPasskeysが存在しない、など様々なエラーケースが存在する。エラーメッセージは「何が起きたか」より「どうすればよいか」を伝える設計にする。

---

## Passkeysのデバイス同期とバックアップ

従来のFIDOセキュリティキー（YubiKey等）は物理デバイスに秘密鍵が紐付けられていた。デバイスを無くすと認証手段を失う。

Passkeysはこれを解決するために、秘密鍵をプラットフォームのKeychain/Keyvaultに保存し、**クラウド経由で同期する**。

- Apple: iCloudキーチェーン経由でAppleデバイス間で同期
- Google: Googleパスワードマネージャー経由でAndroid・Chrome間で同期
- Microsoft: Windowsキーに保存

この設計は利便性とセキュリティのバランスを変えた。「デバイスを失うと認証手段を失う」という問題を解決した代わりに、「クラウドアカウントが侵害されたらPasskeysも危険になる」という新たな考慮点が生まれた。

サーバー側では、ユーザーが複数のPasskeysを登録・管理できる仕組み（「このデバイスを削除する」）が重要だ。

---

## セキュリティの観点からのPasskeys評価

**強み**

- フィッシング耐性が構造的に保証される
- サーバー側に秘密情報を保存しない（データ漏洩リスクの低減）
- 生体認証による本人確認が自動的に含まれる
- パスワード管理の認知的負担がゼロ

**考慮すべき点**

- プラットフォームのエコシステムへの依存
- ブラウザ・OS対応状況の確認が必要（2024年時点で主要ブラウザは対応済み）
- 企業環境での管理ポリシーとの整合

---

## パスワードを「廃止する」のか「補完する」のか

Passkeysの普及は、パスワードが不要になる世界への移行を意味する。しかしその移行は段階的だ。

Next.jsアプリケーションにPasskeysを組み込む際は、まず「Passkeysを使えるユーザーへの提供」から始め、段階的にパスワードへの依存を減らしていく設計が現実的だ。

「ユーザーの安全性を上げながら、体験もシンプルにする」——これがPasskeysの本質的な価値だ。認証はただのログイン画面ではなく、信頼の基盤だ。その設計に真剣に向き合うことが、現代のWebアプリケーション開発者に求められている。
