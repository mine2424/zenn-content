# Next.jsアプリのモニタリング設計——Sentryで「何が壊れたか」を捉える

アプリは壊れる。

どんなに丁寧にテストを書いても、本番環境では予期しない問題が起きる。ユーザーの環境、データのパターン、ネットワークの揺らぎ——これらすべてを事前に網羅することは不可能だ。

だからこそ「壊れた瞬間に気づき、原因を特定できる」仕組みが必要になる。この記事では、Next.jsアプリにおけるモニタリングの設計思想と、Sentryを中心とした実践的なアプローチを整理する。

## モニタリングの「3つの問い」

モニタリングを設計するとき、まず答えるべき問いがある。

**1. 何が壊れたか（What）**
エラーの種類、スタックトレース、影響を受けたコードパス。「TypeError: Cannot read properties of undefined」のような情報だ。

**2. いつ壊れたか（When）**
エラーの発生時刻、頻度、トレンド。「デプロイ後から急増した」「深夜に集中している」といったパターンが見える。

**3. 誰に・どこで壊れたか（Who/Where）**
影響を受けたユーザー数、使用ブラウザ・OS、地域、URLパス。「特定のブラウザだけで起きている」「/checkout ページで集中している」といった文脈だ。

これら3つの問いに答えられない状態は「モニタリングしていない」と同じだ。

## Next.jsにおけるエラーの発生場所

Next.jsアプリのエラーは、複数の異なる場所で発生し得る。それぞれ捕捉方法が異なる。

| エラーの場所 | 特徴 | 捕捉方法 |
|------------|------|---------|
| Client Component | ブラウザで発生 | Error Boundary + Sentry |
| Server Component | サーバーで発生、クライアントに不透明 | Sentry Node SDK |
| API Routes / Route Handlers | HTTPエンドポイント | Sentry Node SDK |
| Edge Middleware | Edgeランタイムで発生 | Sentry Edge SDK |

この「エラーが4つの異なる領域に散らばる」という特性が、Next.jsのモニタリングを少し複雑にしている。

## Sentryのセットアップ設計

SentryはNext.jsを第一級サポートしており、`@sentry/nextjs`パッケージひとつで上記すべての領域をカバーできる。

```typescript
// sentry.client.config.ts
import * as Sentry from "@sentry/nextjs";

Sentry.init({
  dsn: process.env.NEXT_PUBLIC_SENTRY_DSN,
  environment: process.env.NODE_ENV,
  
  // エラーのサンプリングレート（全件収集は高コスト）
  sampleRate: 1.0,
  
  // パフォーマンストレーシングのサンプリング
  tracesSampleRate: process.env.NODE_ENV === "production" ? 0.1 : 1.0,
  
  // セッションリプレイ（ユーザー操作を録画）
  replaysOnErrorSampleRate: 1.0,
  replaysSessionSampleRate: 0.01,
});
```

`sentry.server.config.ts`と`sentry.edge.config.ts`も同様に設定し、それぞれの環境のエラーを捕捉する。

### ソースマップの重要性

本番環境のコードはminify・bundleされているため、スタックトレースが難読化される。Sentryにソースマップをアップロードすることで、元のTypeScriptコードの行番号が表示されるようになる。

これは「何が壊れたか」を特定する上で非常に重要だ。ソースマップなしのスタックトレースは、原因特定にかかる時間を何倍にもする。

## アラート設計：「何に対して通知するか」

モニタリングを導入しただけでは意味がない。問題が起きたときに**適切な人が適切なタイミングで気づける**ことが重要だ。

### 通知疲れを避ける

すべてのエラーを通知すると、通知が多すぎて重要なものが埋もれる。「通知疲れ（Alert Fatigue）」だ。

設計の原則は、**「行動が必要なエラーだけを通知する」** こと。

- **新規エラー（初めて発生したエラー）**: 注目すべき問題の可能性が高い
- **エラーの急増（スパイク検知）**: デプロイによるリグレッションの可能性
- **影響ユーザー数が閾値を超えた**: ビジネスへの実際のインパクト

一方、以下は通知頻度を下げるか抑制を検討する。

- 既知の問題として登録済みのエラー
- ネットワーク由来のユーザー起因エラー
- クローラーやボットからのエラー

### エラーの優先度付け

SentryのIssueには優先度をつけられる。「高頻度かつ多ユーザーに影響するエラー」と「稀にしか発生しないエラー」では、対応の優先度が異なる。

定期的に「未解決の高優先度エラー」をレビューするリズムをチームに作ることが、モニタリングの実効性を高める。

## パフォーマンスモニタリングの位置付け

エラー監視と並んで重要なのが、パフォーマンスのモニタリングだ。

Vercel Analytics（または`@vercel/analytics`）を使うと、Core Web Vitals（LCP、INP、CLSなど）をリアルユーザーデータで計測できる。これは「壊れていないが遅い」状態を検知するためのものだ。

エラー監視（Sentry）とパフォーマンス監視（Vercel Analytics）は補完関係にある。

- エラー監視：「落ちた・壊れた」
- パフォーマンス監視：「遅い・重い」

両方を組み合わせることで、ユーザー体験の全体像が見えてくる。

## カスタムコンテキストを付ける

デフォルトのエラー情報だけでは原因特定が難しいことがある。Sentryにカスタムコンテキストを付けることで、デバッグ速度が大幅に上がる。

```typescript
import * as Sentry from "@sentry/nextjs";

// ユーザー情報を紐付ける
Sentry.setUser({ id: userId, email: userEmail });

// 処理の文脈情報を付ける
Sentry.setContext("order", {
  orderId: order.id,
  totalAmount: order.total,
  itemCount: order.items.length,
});
```

これにより「このエラーは、どのユーザーが、どの注文処理中に発生した」という文脈付きで問題を調査できるようになる。

## まとめ

モニタリングは「壊れないようにする」技術ではなく、「壊れたとき素早く気づいて直す」技術だ。

Next.jsアプリでは、クライアント・サーバー・エッジの各領域でエラーが発生し得る。Sentryを使ってすべての領域を網羅し、適切なアラート設計で通知疲れを避け、カスタムコンテキストでデバッグ速度を上げる。

「何が壊れたか」が30秒でわかる状態を作ることが、本番品質の第一歩だ。
