## はじめに

マイクロサービスやサーバーレスが当たり前になった現代のシステムでは、「なぜアプリケーションが遅くなったのか」「どのサービスでエラーが発生したのか」を特定するのが難しくなっている。

ログを見ても、メトリクスを見ても、単一サービスの断片的な情報しか得られない。リクエストがどのサービスをどの順番で通過し、どこで時間がかかったのかが「見えない」のだ。

この「見えない」問題に取り組むのが**観察可能性（Observability）**であり、その標準的な実装基盤がOpenTelemetryだ。

---

## 観察可能性の3本柱

観察可能性は、一般的に3種類のシグナルで構成される。

**ログ（Logs）**: 何が起きたかの記録。従来から使われているが、分散環境では単一サービスのログだけでは不十分。

**メトリクス（Metrics）**: 数値で測定できるシステムの状態。レイテンシ、スループット、エラーレートなどの集計値。

**トレース（Traces）**: リクエストが複数のサービスを横断する様子を時系列で記録したもの。これが分散システムの「見える化」のカギだ。

OpenTelemetryは、この3つのシグナルを統一的に収集・伝搬するためのオープンスタンダードだ。特定のベンダーに依存せず、JaegerやZipkin、Datadog、Grafana Tempoなど多様なバックエンドと組み合わせられる。

---

## 分散トレースが解決する問題

例えば、次のようなシナリオを想像してほしい。

```
ユーザー → APIゲートウェイ → 認証サービス → 商品サービス → 在庫サービス → レスポンス
```

このフローのどこかで500msの遅延が発生しているとする。ログには「リクエストが遅い」という情報はあるが、どのサービスで遅延が発生しているか分からない。

分散トレースを使うと、リクエスト全体の流れが1つの「トレース」として可視化される。各サービスで処理した「スパン（Span）」が階層的につながり、どのスパンで何ミリ秒かかったかが一目でわかる。

```
Trace: リクエスト全体（total: 520ms）
 └─ APIゲートウェイ (10ms)
    ├─ 認証サービス (15ms)
    └─ 商品サービス (480ms)  ← ここが遅い
       └─ 在庫サービス (30ms)
```

これにより、「商品サービスに問題がある」と即座に特定できる。

## OpenTelemetryの基本概念

### Trace と Span

**Trace**は、1つのリクエストのライフサイクル全体を表す。一意の`traceId`を持つ。

**Span**は、Trace内の個々の作業単位だ。サービスAがサービスBを呼び出すとき、それぞれのサービスで1つのSpanが作られ、親子関係でつながる。

各Spanには以下の情報が含まれる：
- `spanId`（一意識別子）
- `parentSpanId`（親Spanへの参照）
- 開始時刻・終了時刻
- ステータス（OK / ERROR）
- 属性（キーバリューのメタデータ）
- イベント（ポイントインタイムの出来事）

### Context Propagation

分散トレースの肝は**コンテキスト伝搬**だ。サービスAからサービスBを呼び出す際に、HTTPヘッダー（`traceparent`など）にトレース情報を埋め込んで伝搬させる。これにより、異なるサービスで記録されたSpanが同じTraceに束ねられる。

```
traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01
             └─ バージョン
                └─ traceId（32文字16進数）
                                          └─ parentSpanId（16文字）
                                                               └─ フラグ
```

この伝搬がなければ、各サービスのSpanはバラバラの孤立したデータとして記録されるだけで、つながりが見えない。

## Node.jsでのOpenTelemetry実装例

```javascript
// instrumentation.js
const { NodeSDK } = require("@opentelemetry/sdk-node");
const { OTLPTraceExporter } = require("@opentelemetry/exporter-trace-otlp-http");
const { getNodeAutoInstrumentations } = require("@opentelemetry/auto-instrumentations-node");

const sdk = new NodeSDK({
  traceExporter: new OTLPTraceExporter({
    url: "http://otel-collector:4318/v1/traces",
  }),
  instrumentations: [getNodeAutoInstrumentations()],
});

sdk.start();
```

`getNodeAutoInstrumentations()` を使うと、Express、HTTP、gRPC、データベースクライアントなどの一般的なライブラリへの自動計装が有効になる。コードを大幅に書き換えることなく、トレース情報の収集が始まる。

カスタムSpanを手動で追加したい場合：

```javascript
const { trace } = require("@opentelemetry/api");
const tracer = trace.getTracer("my-service");

async function processOrder(orderId) {
  return tracer.startActiveSpan("processOrder", async (span) => {
    span.setAttribute("order.id", orderId);
    try {
      const result = await doWork(orderId);
      span.setStatus({ code: SpanStatusCode.OK });
      return result;
    } catch (err) {
      span.recordException(err);
      span.setStatus({ code: SpanStatusCode.ERROR });
      throw err;
    } finally {
      span.end();
    }
  });
}
```

## テスト戦略との関係

観察可能性は、QA戦略とも密接に関係する。

**テスト環境での活用**: ステージング環境にOpenTelemetryを導入しておくと、パフォーマンステスト中にどのサービスがボトルネックになっているかをリアルタイムで把握できる。k6などの負荷テストツールと組み合わせることで、「負荷がかかるとどこが劣化するか」を精密に測定できる。

**障害注入テスト（Chaos Engineering）との組み合わせ**: 意図的に障害を注入したとき、システムが期待通りに振る舞うかを検証する際にも、分散トレースが活躍する。どこで障害が検出・処理されたかをトレースで確認できる。

**アラートの精度向上**: 「エラーレートが上昇した」というメトリクスアラートに対して、「どのトレースがエラーだったか」をすぐに掘り下げられる。アラートから根本原因までの調査時間（MTTR）が大幅に短縮される。

## OpenTelemetry Collectorの役割

実際のシステムでは、アプリケーションが直接バックエンドにデータを送るのではなく、**OpenTelemetry Collector**を中間に置くことが推奨される。

```
アプリ → OTel Collector → Jaeger / Grafana Tempo / Datadog など
```

Collectorには以下のメリットがある：

- **バッファリング**: バックエンドが停止してもデータロストを防げる
- **フィルタリング・サンプリング**: 全トレースを保存するとコストが高いため、エラートレースは100%保存、正常トレースは1%などのサンプリングを設定できる
- **変換・エンリッチ**: データを変換・加工してから複数のバックエンドに送れる

サンプリングは特に重要で、本番環境では適切なサンプリング戦略なしにOpenTelemetryを運用するとデータ量・コストが爆発する。

## 何を計装すべきか

「全部計装すればいい」は正しくない。重要なのは**境界を意識した計装**だ。

計装すべき重要な場所：
- **外部サービス呼び出し**（HTTPリクエスト、gRPC、メッセージキュー）
- **データベースクエリ**（特に遅いクエリの検出に有効）
- **ビジネスロジックの重要な処理**（注文処理、決済など）
- **非同期処理のキュー投入・取り出し**

逆に、ループの内部や細粒度すぎる処理まで計装すると、ノイズが増えてトレースが読みにくくなる。「このSpanを見てデバッグできるか」という視点で判断する。

---

## まとめ

OpenTelemetryは、分散システムの「見えない問題」を「見える化」するための業界標準だ。ログ・メトリクス・トレースという3つのシグナルを統一的に扱えることで、システムの状態をより立体的に把握できる。

QAの文脈では、テスト・負荷試験・カオスエンジニアリングと組み合わせることで、単なるpass/failを超えた「なぜ失敗したか」の洞察を得られる。

「テストが通った」だけでなく「本番で何が起きているか」を継続的に観察できる体制を作ることが、現代のソフトウェア品質保証の本質的な課題だ。OpenTelemetryはその基盤として、今後ますます重要になるだろう。
