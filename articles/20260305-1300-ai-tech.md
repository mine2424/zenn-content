## 「どちらか一方」では足りない

LLMを使ったRAG（Retrieval-Augmented Generation）システムを設計するとき、最初にぶつかる壁が「検索の精度」だ。ベクトル検索を使えば意味的な類似度で文書を引ける。全文検索を使えばキーワードの一致で確実に文書を引ける。それぞれが強みを持っているのに、なぜどちらか一方を選ぼうとしてしまうのか。

Hybrid Searchとは、この2つを組み合わせる手法だ。ただし「両方やれば精度が上がる」という単純な話ではない。それぞれの検索が「何を得意とし、何を苦手とするか」を正確に理解した上で設計しなければ、複雑さだけが増して精度は変わらない結果になる。

本記事では、Hybrid Searchの設計思想と、実際に組み合わせを機能させるための考え方を整理する。

## ベクトル検索が「意味」を扱う仕組み

ベクトル検索の中心にあるのは、エンベディング（Embedding）と呼ばれる技術だ。テキストを高次元のベクトル空間に写像し、意味的に近いテキスト同士がベクトル空間でも近い位置に配置されるようにする。

この特性は強力だ。「犬」と「dog」、「機械学習」と「ML」のように、表記が異なっても意味が近ければ高いスコアが付く。ユーザーが曖昧なクエリや言い換えを使っても、それなりに意図した文書を引ける。

一方で弱点もある。ベクトル検索は「意味の近さ」を測るが、固有名詞や型番、バージョン番号といった具体的なキーワードには弱い。「v3.2.1のバグ修正ノート」を探そうとしたとき、バージョン番号がベクトル空間で正確に表現されているとは限らないからだ。

## 全文検索が「正確なキーワード」に強い理由

全文検索（Lexical Search）の代表的な手法がBM25だ。TF-IDF（単語の出現頻度と逆文書頻度）をベースに、クエリと文書の間のキーワード一致度をスコアリングする。

全文検索の強みは、キーワードの一致に確実性があることだ。特定の商品名、人名、関数名、エラーコードを検索するとき、全文検索はそのキーワードが含まれる文書を高精度で返す。

弱点は意味の揺れへの耐性だ。「実行速度を上げる方法」というクエリに対して、「パフォーマンス最適化の手順」という文書は全文検索ではマッチしにくい。クエリと文書に共通するキーワードが少なければスコアが下がるからだ。

## Hybrid Searchが解決する問題

2つの検索を組み合わせることで、それぞれの弱点を補い合える。具体的には次のような場面でHybrid Searchは効果を発揮する。

**キーワードと意味の両方が重要なクエリ**：「Pythonで非同期処理を最適化する方法」というクエリは、「Python」「非同期」というキーワードの一致（全文検索）と、「最適化」「パフォーマンス改善」の意味的な類似（ベクトル検索）の両方が必要だ。

**専門用語が混在するドメイン**：医療、法律、エンジニアリングの文書では、固有の専門用語と一般的な説明文が混在する。全文検索が専門用語を正確に捉え、ベクトル検索が意味的な文脈を補う。

**多言語・翻訳が絡むケース**：ベクトル検索は言語をまたいだ類似度を測れるモデルがある。全文検索はその言語のキーワードに特化する。組み合わせることで多言語コーパスに対する検索品質を高められる。

## スコアの統合——どう組み合わせるか

Hybrid Searchの設計で最も重要な問いが「2つのスコアをどう統合するか」だ。

**線形結合（Weighted Sum）**

最もシンプルな方法が線形結合だ。ベクトル検索のスコア `v_score` と全文検索のスコア `l_score` に重みをかけて足し合わせる。

```python
hybrid_score = alpha * v_score + (1 - alpha) * l_score
```

`alpha` はドメインやクエリの傾向によってチューニングする。意味的な検索が重要なドメインでは `alpha` を高く、キーワード検索が重要なドメインでは低くする。

問題はスコアのスケールが異なることだ。ベクトル検索のコサイン類似度は-1〜1の範囲に収まるが、BM25スコアは文書コーパスに依存して上限が変わる。単純に足し合わせると、一方のスコアに引っ張られやすい。

**Reciprocal Rank Fusion（RRF）**

スケールの違いを回避するために広く使われるのがRRF（相互順位融合）だ。スコアの値ではなく、各検索結果の「順位」を使ってスコアを計算する。

```
RRF_score = Σ 1 / (k + rank_i)
```

`k` は定数（一般に60が使われる）、`rank_i` は各検索手法での順位だ。順位ベースなのでスコールのスケール差を気にしなくていい。実装がシンプルで安定した結果を出すことから、実務では最初の選択肢になることが多い。

**Cross-Encoder Reranking**

RRFで得た上位候補を、さらにCross-Encoderで並べ替える手法もある。Cross-Encoderはクエリと文書のペアを同時に入力として精度の高い関連度スコアを出す。計算コストが高いため全件には適用できないが、上位10〜20件を絞り込んだ後の精緻化として有効だ。

## 実装プラットフォームの選択

Hybrid SearchをサポートするVectorDB・検索エンジンは増えている。

**Weaviate**: ベクトル検索とBM25をネイティブに組み合わせるHybrid Searchを標準機能として提供する。RRFによるスコア統合も組み込まれており、設定が少なく始められる。

**Elasticsearch / OpenSearch**: 長年の全文検索の実績にベクトル検索（kNN）を追加した。既存のElasticsearchインフラがあるなら統合しやすい。

**Qdrant**: ベクトル検索特化型だが、Sparse Vector（スパースベクトル）を使ったBM25相当の検索をサポートし始めている。

**pgvector + PostgreSQL**: PostgreSQLにpgvectorを追加し、全文検索（tsvector）と組み合わせる方法もある。インフラをシンプルに保ちたいチームに向く。

## チャンキングとHybrid Searchの関係

RAGシステムではHybrid Searchの前にチャンキング（文書の分割）がある。チャンキングの設計がHybrid Searchの精度に直結することを見落としやすい。

全文検索はキーワードが文書内に出現することが前提だ。チャンクが小さすぎてキーワードが分断されると、全文検索のスコアが下がる。ベクトル検索は長すぎるチャンクで意味が希薄になる。HybridSearchのためのチャンクサイズは、両方の特性を考慮した調整が必要だ。

Sentence Window Retrievalという手法では、小さいチャンクでベクトル検索し、実際に返す文脈は前後のチャンクを含めて広げる。検索精度と文脈の豊かさを両立する設計として参考になる。

## どこに力を入れるべきか

Hybrid Searchを設計するとき、「RRFで統合すれば終わり」と思わないほうがいい。実際の精度向上の多くは次の部分から来る。

- クエリ前処理（同義語展開、クエリリライト）
- チャンキング戦略の見直し
- ドメイン固有のエンベディングモデルの選択
- Rerankerの導入タイミングとモデル選択

Hybrid Searchはあくまでパイプラインの一要素だ。「2つの検索を組み合わせる」ことよりも、「どのクエリでどの検索が失敗しているか」を観察し、改善するプロセスこそが精度向上の本質だ。検索ログを取り、失敗パターンを分析する文化がなければ、どれだけ精巧な統合ロジックを組んでも精度の天井が見えにくい。

## まとめ

Hybrid Searchは、ベクトル検索と全文検索のそれぞれの弱点を補い合う設計思想だ。スコアの統合方法としてはRRFが実務での出発点として扱いやすく、さらなる精度を求めるならCross-Encoder Rerankingを組み合わせる。しかし最終的には、どの検索手法を組み合わせるかよりも、「自分のシステムがどのクエリでどう失敗するか」を観察し続けることが、長期的な精度改善の基盤になる。
