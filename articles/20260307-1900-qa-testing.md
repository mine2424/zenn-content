# ミューテーションテストで「テストの品質」を問い直す——テストをテストする発想

「テストカバレッジ80%を達成しました」という報告は、一見安心感をもたらす。しかし、カバレッジが高くてもバグを見逃すテストは存在する。問題は、テストの「量」ではなく「質」だ。

**ミューテーションテスト（Mutation Testing）**は、テスト自体の品質を測るための手法だ。テストを書く行為を疑い、「このテストは本当にバグを捕まえられるか」という問いを自動化する。

---

## カバレッジが測れないもの

コードカバレッジは「テストがどのコードを通過したか」を測る。行カバレッジ、分岐カバレッジ、ステートメントカバレッジ——様々な粒度があるが、共通の盲点がある。

それは「テストがアサーションを適切に行っているか」を測れない点だ。

```javascript
function add(a, b) {
  return a + b;
}
```

このコードに対して、こんなテストを書いたとしよう。

```javascript
test("add function is called", () => {
  add(1, 2); // アサーションなし
});
```

このテストはカバレッジを100%にするが、何も検証していない。`return a - b` に変わっても、このテストは通過し続ける。カバレッジはこの問題を検出できない。

---

## ミューテーションテストの仕組み

ミューテーションテストは「コードに意図的なバグを注入し、テストがそのバグを検出できるか」を検証する手法だ。

注入される変更を**ミュータント（mutant）**と呼ぶ。代表的なミュータントの種類を見てみよう。

- `+` を `-` に変える（算術演算子の変更）
- `>` を `>=` に変える（比較演算子の変更）
- `&&` を `||` に変える（論理演算子の変更）
- `true` を `false` に変える（定数の変更）
- `return value` を `return undefined` に変える（戻り値の変更）

ミューテーションテストツールは、これらの変更を自動的にコードに適用し、それぞれのミュータントに対してテストスイートを実行する。

**ミュータントが「Kill」された**とは：テストスイートが失敗したこと。つまりテストがバグを検出できた。  
**ミュータントが「Survive」した**とは：テストスイートが通過したこと。つまりバグを見逃した。

ミュータントをKillできた割合を**ミューテーションスコア**と呼ぶ。これがテストの実質的な品質指標になる。

---

## JavaScript/TypeScriptでの実践：Stryker

JavaScriptエコシステムで広く使われているミューテーションテストツールが**Stryker Mutator**だ。

```bash
npm init stryker-config
```

設定ファイルを作成し、`npx stryker run` を実行するだけで、既存のJestやVitestのテストをそのまま使ってミューテーションテストが始まる。

Strykerは実行後にHTMLレポートを生成する。レポートを開くと、どのミュータントがSurviveしたか（つまりどのバグがテストで見逃されたか）がコードと対応して表示される。

これが直感的に強力で、「ここのif文に対するテストが弱い」「この比較の境界値が検証されていない」という発見が、視覚的に得られる。

---

## ミューテーションテストから見えてくること

ミューテーションスコアを改善しようとすると、テストの書き方が変わる。

**境界値に対するアサーション**が増える。`>` と `>=` を区別するテストは、境界値を明示的にテストしないと書けない。境界値のミュータントをKillするには、`n === threshold` というケースを明示的に追加する必要がある。

**エラーケースのカバレッジ**が自然に増える。正常系だけテストしている場合、「戻り値をundefinedに変える」ミュータントがSurviveし続ける。例外やエラー状態を検証するテストが必要になる。

**副作用の検証**が丁寧になる。副作用（状態の変化、外部呼び出し）に対してアサーションがないと、そこに注入されたミュータントがSurviveする。`expect(mock).toHaveBeenCalledWith(...)` の重要性が実感できる。

---

## コストと向き合い方

ミューテーションテストの課題は実行コストだ。ミュータントの数はコードの規模に比例し、数百〜数千になることがある。それぞれに対してテストスイートを実行するため、実行時間が長くなる。

現実的な運用としていくつかのアプローチがある。

**対象を絞る**：全コードではなく、ビジネスロジックのコアモジュールにだけ適用する。UIコンポーネントや設定ファイルは対象外にする。

**CIに統合する際は差分のみ**：プルリクエストで変更されたファイルに対してのみミューテーションテストを走らせる。Strykerは `--since` オプションで差分実行をサポートしている。

**スコア目標を設定しすぎない**：ミューテーションスコア100%を追うのは非現実的で、コストが見合わないケースも多い。「コアロジックは80%以上」「全体の基準は60%以上」といった段階的な目標が現実的だ。

---

## カバレッジとミューテーションスコアの関係を整理する

カバレッジとミューテーションスコアは補完関係だ。

| 状態 | 意味 |
|------|------|
| カバレッジ高・ミューテーションスコア低 | テストがコードを通過するだけでアサーションが弱い |
| カバレッジ低・ミューテーションスコア高 | テストしている範囲は質が高いが、範囲が足りない |
| 両方高 | 良いテストスイートの証拠 |

理想は「カバレッジが高く、ミューテーションスコアも高い」状態だが、まず現状のギャップを把握することに価値がある。

---

## まとめ：テストを疑う文化を持つ

ミューテーションテストの本質は、「テストを書いたこと」への安心感を疑う文化だ。

テストは書いただけでは意味をなさない。バグを検出できるテストかどうか——この問いに自動で答えてくれるのがミューテーションテストだ。

ツールを使えば既存のテストを書き直す必要はない。今のテストスイートをそのまま評価できる。まずはコアロジックに対してStrykerを走らせ、Surviveしているミュータントのレポートを眺めてみることをおすすめする。そこには「テストが守れていなかった場所」が可視化されているはずだ。
