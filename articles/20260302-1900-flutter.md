## はじめに

Flutterを学び始めて最初の壁として多くの人が挙げるのが「状態管理」です。Provider、Riverpod、BLoC、Cubit、GetX……選択肢が多すぎて何を選べばいいかわからず、それぞれを学ぶたびに新しい概念が出てきて混乱する。

しかし、状態管理の難しさは「ライブラリが複雑だから」ではありません。本質的な難しさは「**状態とは何か**」「**どこで状態を持つべきか**」という設計の問題にあります。

この記事では、ライブラリの使い方よりも先に理解すべき「状態管理の思想」について解説します。

## 「状態」とは何か

まず「状態（State）」という言葉を正確に定義しましょう。状態とは、**「時間とともに変化し、UIの見た目に影響するデータ」**のことです。

例えば：

- ユーザーが入力したテキスト（フォームの内容）
- APIから取得したユーザーのリスト
- ダークモード・ライトモードの設定
- 現在選択されているタブのインデックス

これらは全て状態です。一方で、定数やアプリ起動時に1度だけ読み込んで変わらないデータは状態ではありません。

この区別が重要な理由は、**状態の変化をUIに反映させる仕組み**がFlutterの「StatefulWidget」や各種状態管理ライブラリの核心だからです。「変わらないものを状態として管理する」のは無駄なコストです。

## 状態の「スコープ」という概念

状態管理を難しくしている大きな原因の1つが、**「この状態はどこで管理すべきか」**という問題です。これを「状態のスコープ」と呼びます。

状態のスコープには大きく3つのレベルがあります。

**ローカル状態（Local State）**

1つのウィジェット内だけで使われる状態。例えばテキストフィールドにフォーカスが当たっているかどうか、ExpandableなUIが開いているかどうかなど。

これは`StatefulWidget`の`setState`で十分です。わざわざRiverpodやBLoCで管理する必要はありません。

**共有状態（Shared State）**

複数のウィジェットが参照・更新する状態。例えば「ショッピングカートの中身」は、商品一覧画面でも、カートアイコンのバッジ数でも参照されます。

この場合、「どちらかが更新したら、もう一方も自動で更新される」仕組みが必要になります。これがProviderやRiverpodが解決しようとしている問題の核心です。

**グローバル状態（Global State）**

アプリ全体にわたって存在する状態。認証状態（ログイン中かどうか）、テーマ設定、言語設定など。ほぼ全ての画面から参照される可能性があるものです。

**重要な原則：状態のスコープはできるだけ小さくする。**

グローバル状態が増えすぎると、どこで状態が変化したかを追跡するのが難しくなり、バグの原因になります。「この状態、本当にグローバルに必要か？」を常に問い直す習慣が大切です。

## 状態の「流れ」を一方向にする

Flutterに限らず、モダンなUIフレームワークに共通する重要な設計原則として「単方向データフロー（Unidirectional Data Flow）」があります。

考え方は単純です。

1. **状態**が存在する（State）
2. 状態からUIが生成される（View = f(State)）
3. ユーザーの操作がアクションを発生させる（Action）
4. アクションが状態を更新する（State Update）
5. 更新された状態から新しいUIが生成される（View再描画）

このサイクルを常に1方向に保つことが、バグを減らし、デバッグを容易にする鍵です。

**双方向バインディングの落とし穴**

「UIが状態を直接変更できる」設計は便利に見えますが、状態の変更経路が複数になるとデバッグが困難になります。「どこで状態が変わったのか」を追うのが難しくなるからです。

RiverpodのNotifierやBLoCのEventは、「状態を変更できる経路を明示的に定義する」ための仕組みです。この制約こそが、複雑なアプリでも状態の変化を追跡しやすくする設計です。

## 非同期状態の管理：最も難しい問題

Flutterの状態管理で最も難しいのは、**非同期処理（APIコールなど）の状態管理**です。

非同期処理には常に3つの状態があります：

- **Loading（処理中）**: ローディングインジケーターを表示
- **Success（成功）**: データを表示
- **Error（失敗）**: エラーメッセージを表示

これを適切に管理しないと、「ローディング中なのにエラーUIが表示されている」「前回のデータが残ったまま新しいデータが表示される」などの奇妙なバグが発生します。

Riverpodの`AsyncValue`はこの3状態を型安全に扱うための概念です。

```dart
// AsyncValueを使うと、3状態のハンドリングを網羅的に書ける
ref.watch(userProvider).when(
  data: (user) => UserProfile(user),
  loading: () => CircularProgressIndicator(),
  error: (e, st) => ErrorWidget(e),
);
```

`when`メソッドを使うことで、コンパイル時に「loading状態の時の処理を書き忘れた」というミスを防げます。これは「状態のハンドリング漏れ」という非常によくあるバグを型システムで防ぐ、優れた設計です。

## ライブラリ選択の前に考えること

「RiverpodとBLoCどちらがいいですか？」という質問をよく見かけます。正直なところ、**どちらを選んでも大きな問題にはなりません**。重要なのは、ライブラリの選択よりも前に考えるべきことです。

- 状態のスコープを適切に設計できているか
- 単方向データフローを守れているか
- 非同期状態の3ステートを正しく扱えているか

これらの設計思想を理解した上でライブラリを選べば、どのライブラリでも意図通りに使えます。逆に、これを理解せずにライブラリのAPIだけ覚えても、複雑なアプリになった途端に破綻します。

個人開発や小規模チームにはRiverpod、大規模チームでイベントとステートを明示的に分離したい場合はBLoCが向いている、というのが現実的な指針ですが、これはあくまで参考程度に考えてください。

## まとめ

Flutterの状態管理を正しく理解するための思想をまとめます。

- 状態とは「変化し、UIに影響するデータ」——変わらないものを状態にしない
- 状態のスコープを小さく保つ：ローカル→共有→グローバルの順に検討する
- 単方向データフローを守ることで、状態の変化を追跡しやすくする
- 非同期処理は必ずLoading/Success/Errorの3状態を正しく管理する
- ライブラリは設計思想の理解の後に選ぶ

状態管理の難しさの本質は、ライブラリの複雑さではなく設計の問題です。思想を理解することで、どのライブラリを選んでも自信を持って使えるようになります。
