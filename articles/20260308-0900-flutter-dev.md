## ディープリンクはUXの「見えないインフラ」だ

ユーザーがメールやSNSのリンクをタップして、アプリが開いた直後に目的のコンテンツが表示される——これは当たり前のように見えて、実装が複雑な機能だ。逆に、リンクをタップしたのにアプリのトップ画面が開くだけだとしたら、ユーザーは苛立ちを感じ、もう一度目的の画面を探すことになる。

**ディープリンク（Deep Link）**は、アプリの特定の画面に直接遷移するためのURLの仕組みだ。ECアプリで「この商品ページを友人に共有する」「メールで送った注文確認リンクから注文詳細画面を開く」といった体験を支える。

Flutterでは複数のディープリンク方式があり、プラットフォームごとの設定も必要だ。本稿ではその仕組みと設計思想を整理する。

## ディープリンクの種類を理解する

Flutter（モバイル）のディープリンクには大きく3つの方式がある。

### カスタムURLスキーム

`myapp://products/123` のような、アプリ固有のURLスキームを使う方式だ。設定がシンプルで、AndroidとiOSともにサポートされている。

ただし重大な欠点がある。**ブラウザからは開けない**し、**他のアプリに同じスキームを登録されると競合する**。セキュリティの観点からも、悪意あるアプリが同じカスタムスキームを登録して起動されるリスクがある。

### Universal Links（iOS）/ App Links（Android）

`https://example.com/products/123` という通常のHTTPS URLをそのままアプリで処理する方式だ。アプリがインストールされていればアプリで開き、インストールされていなければWebブラウザで開く。

これがモダンなディープリンクの標準だ。URLがHTTPS形式なので、Webにもフォールバックでき、SEOにも活用できる。ただし**ドメインの所有証明が必要**だ（iOSはASAファイル、Androidはassetlinksファイルを特定のURLに配置する）。

### Firebase Dynamic Links（廃止予定）

FirebaseのDynamic Linksは長らく使われてきたが、2025年にサービス終了が予告された。既存のコードベースで使っている場合は移行が必要だ。

## iOSとAndroidそれぞれのプラットフォーム設定

Universal Links（iOS）を有効にするには、Webサーバーの `https://example.com/.well-known/apple-app-site-association` にJSONファイルを配置する必要がある。このファイルでアプリのBundleIDとパスのマッピングを宣言する。

Android App Linksでは、`https://example.com/.well-known/assetlinks.json` に同様のファイルを配置し、アプリのパッケージ名とSHA-256証明書フィンガープリントを記述する。

どちらも「このドメインに対してこのアプリが権限を持つ」ことをサーバー側で宣言する仕組みだ。この設定が正しくないと、リンクはブラウザで開かれてしまう。

開発環境でのデバッグには `adb` コマンドやXcodeのシミュレーターを使ったテストが有効だが、本番と同じドメインに設定ファイルが配置されていないとテストできないため、ステージング環境の準備が重要になる。

## go_routerとのセット設計

FlutterでのURLルーティングには `go_router` が事実上の標準だが、ディープリンクとの統合を設計する際に考えるべきことがある。

ディープリンクからアプリが起動された場合、URLをパースして適切なルートに遷移させる必要がある。go_routerはURLベースのルーティングを前提に設計されているため、`initialLocation` の設定や `redirect` ロジックでディープリンクを処理できる。

```dart
final router = GoRouter(
  routes: [
    GoRoute(path: '/products/:id', builder: (context, state) {
      final productId = state.pathParameters['id']!;
      return ProductDetailPage(productId: productId);
    }),
  ],
);
```

ただしディープリンクで起動した場合、**認証状態の確認**が必要になる。未ログインユーザーが `https://example.com/orders/456` を開こうとしたとき、いきなり注文詳細画面は見せられない。ログイン画面にリダイレクトし、ログイン完了後に元のURLに戻る「ディープリンク→認証→リダイレクト」のフローを設計する必要がある。

go_routerの `redirect` コールバックと `extraParam` を使って、ログイン前のターゲットURLをセッション中に保持するのが一般的なパターンだ。

## 「アプリがインストールされていない場合」のフォールバック設計

Universal Links / App Linksの重要な特性は、アプリが未インストールの場合にWebにフォールバックすることだ。これはUXの観点から大きなメリットだ。

フォールバック先のWebページは、単なるエラーページではなく、アプリをインストールするよう誘導するランディングページが理想的だ。さらに理想を言えば、アプリインストール後に元のコンテンツへの遷移を実現する**Deferred Deep Link**を検討する価値がある。

Deferred Deep Linkは「リンクタップ → アプリなし → ストアへ → インストール完了 → アプリ起動時にリンクのURLを復元」というフローだ。Firebase Dynamic Linksが提供していたこの機能は、サービス終了に伴いAppsFlyer、Branch、AdjustなどのMMPサービスか、自前実装に移行する必要がある。

## テスト戦略

ディープリンクは「設定ファイル + アプリコード + Webサーバー」の3つが正しく連携して初めて動く。テストも多層的に行う必要がある。

**Unitレベル**: URLのパースロジックをユニットテストする。`/products/123` が `ProductDetailPage` に正しくマッピングされるか。

**Integrationレベル**: `app_links` や `uni_links` パッケージを使ったリンク受信ロジックをモックでテストする。

**手動テスト**: 実機または実機に近い環境で、実際にリンクをタップしてアプリが正しく開くかを確認する。特にアプリを**フォアグラウンドで開いている場合**・**バックグラウンドにある場合**・**完全に終了している場合**の3パターンで動作が異なることがあるため、全ケースを確認する。

## ディープリンクは「ユーザーを迷子にしない」設計だ

ディープリンクは技術的には設定ファイルとルーティングコードの話だが、本質はUXの問題だ。ユーザーはリンクを受け取ったとき、「このリンクを踏んだら目的のコンテンツにたどり着ける」と期待している。その期待を裏切らないことが、アプリへの信頼に直結する。

特にプッシュ通知、メールマーケティング、SNSシェア、QRコード——これらすべてがディープリンクを起点にする。コンバージョン率やリテンション率に直接影響する「見えないインフラ」として、ディープリンク設計に適切な投資をする価値がある。

設定の複雑さはあるが、一度正しく設計できれば、アプリとWebの境界を意識させないシームレスな体験をユーザーに届けられる。
