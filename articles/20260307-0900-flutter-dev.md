# LottieとRiveでFlutterのアニメーションを「デザイナーの意図」のまま動かす

アニメーションはアプリの「感触」を決める。

ボタンを押した瞬間の微妙な反応、ローディング中の動き、成功時のフィードバック——これらが自然であるとき、ユーザーはアプリを「いい」と感じる。逆に、ぎこちないアニメーションや動きのなさは、機能的に完璧なアプリでも「なんか違う」という印象を与えてしまう。

FlutterのAnimationControllerを使えば様々なアニメーションが実装できるが、「デザイナーがAfter Effectsで作ったあの動き」をコードで再現しようとすると、多大な時間とすり合わせが発生する。LottieとRiveはこの「デザインとコードの翻訳コスト」を根本から削減するための仕組みだ。

---

## LottieとRive：何が違うのか

両者はどちらも「外部ツールで作ったアニメーションをFlutterで再生する」という目的を持つが、思想が異なる。

**Lottie**

After Effectsで作成したアニメーションをBodymovingプラグインでJSON形式（`.json`）にエクスポートし、そのまま再生する。アニメーションの制御は「再生・停止・ループ・速度変更」程度で、インタラクティブな状態遷移は苦手だ。

一方向の「見せるアニメーション」に特化している。ローディングインジケーター、オンボーディングのイラスト、成功・失敗のフィードバックアニメーションなど、「決まった動きを美しく再生する」用途に向いている。

**Rive**

Rive（旧Flare）は独自のオーサリングツール（rive.app）を持ち、アニメーションに「ステートマシン」を組み込める。「タップされたら状態Aから状態Bへ遷移する」「数値に応じてアニメーションが変化する」といったインタラクティブな表現が得意だ。

「反応するアニメーション」と言えばRiveの領域だ。キャラクターの表情、インタラクティブなボタン、ゲームのUIなど、状態に連動した複雑なアニメーションに向いている。

## Lottieの実装

`lottie`パッケージをpubspec.yamlに追加することで使える。

```yaml
dependencies:
  lottie: ^3.0.0
```

基本的な使い方はシンプルだ。

```dart
// assetsからの読み込み
Lottie.asset('assets/animations/loading.json')

// ネットワークからの読み込み
Lottie.network(
  'https://assets.lottiefiles.com/packages/lf20_xxxx.json',
)
```

より細かい制御が必要な場合は`AnimationController`と組み合わせる。

```dart
class LoadingAnimation extends StatefulWidget {
  @override
  State<LoadingAnimation> createState() => _LoadingAnimationState();
}

class _LoadingAnimationState extends State<LoadingAnimation>
    with SingleTickerProviderStateMixin {
  late AnimationController _controller;

  @override
  void initState() {
    super.initState();
    _controller = AnimationController(vsync: this);
  }

  @override
  Widget build(BuildContext context) {
    return Lottie.asset(
      'assets/animations/loading.json',
      controller: _controller,
      onLoaded: (composition) {
        _controller
          ..duration = composition.duration
          ..repeat();
      },
    );
  }

  @override
  void dispose() {
    _controller.dispose();
    super.dispose();
  }
}
```

`onLoaded`コールバックでアニメーションの長さを受け取り、controllerにセットする点がポイントだ。Lottieファイルによって長さが異なるため、ハードコードより動的に設定する方が安全だ。

## Riveの実装とステートマシン

Riveの強みはステートマシンにある。rive.appでアニメーションを作る際、「idle」「hover」「pressed」のような状態と遷移条件を設定できる。Flutterはそのステートマシンを「入力値を送る」だけで制御できる。

```yaml
dependencies:
  rive: ^0.13.0
```

```dart
class InteractiveButton extends StatefulWidget {
  @override
  State<InteractiveButton> createState() => _InteractiveButtonState();
}

class _InteractiveButtonState extends State<InteractiveButton> {
  SMIBool? _pressInput;

  void _onRiveInit(Artboard artboard) {
    final controller = StateMachineController.fromArtboard(
      artboard,
      'ButtonStateMachine', // Riveで設定したステートマシン名
    );
    
    if (controller != null) {
      artboard.addController(controller);
      _pressInput = controller.findInput<bool>('isPressed') as SMIBool?;
    }
  }

  @override
  Widget build(BuildContext context) {
    return GestureDetector(
      onTapDown: (_) => _pressInput?.change(true),
      onTapUp: (_) => _pressInput?.change(false),
      child: SizedBox(
        width: 200,
        height: 60,
        child: RiveAnimation.asset(
          'assets/button.riv',
          onInit: _onRiveInit,
        ),
      ),
    );
  }
}
```

`SMIBool`はステートマシンへのBoolean型の入力だ。タップイベントをこの入力に接続することで、Flutterのコードは「押された・離された」を伝えるだけで、アニメーションの具体的な動きはRiveファイルが管理する。

ここにRiveの設計思想が現れている。**アニメーションの表現とアプリのロジックの責務を分離する**。デザイナーはRiveでアニメーションを磨き、エンジニアは状態を管理する——このように役割分担できる。

## パフォーマンス上の考慮

両ライブラリを使う際、パフォーマンスに関して意識すべきことがある。

**不要な再ビルドを避ける**

アニメーションが含まれるWidgetが親の状態変化で再ビルドされると、アニメーションがリセットされたり、コントローラーが再生成されて余分な処理が発生する。アニメーションWidgetはできる限りツリーの下層に、小さなWidgetとして配置する。

**メモリとファイルサイズ**

Lottieのjsonファイル、Riveの.rivファイルはそれなりのサイズになりうる。使わないアニメーションをアプリバンドルに含めないよう、必要なものだけを`assets`に追加する。頻繁に使う場合はキャッシュも考慮する。

**RepaintBoundary**

アニメーションは毎フレーム再描画される。`RepaintBoundary`でアニメーションWidgetを囲むことで、アニメーションの再描画が親Widgetに伝播するのを防ぎ、不要な再描画を削減できる。

```dart
RepaintBoundary(
  child: Lottie.asset('assets/animations/loading.json'),
)
```

## どちらを選ぶか

選択の判断軸をシンプルにまとめる。

**Lottie**:
- デザイナーがAfter Effectsを使っている
- 一方向の演出アニメーション（ローディング・空状態・オンボーディング）
- インタラクションは不要

**Rive**:
- アニメーションが状態に連動して変化する
- インタラクティブなUI要素（ボタン・キャラクター・データ可視化）
- デザイナーがRiveツールを使える

両方を使うプロジェクトも珍しくない。「ローディングはLottie、ボタンのインタラクションはRive」という組み合わせも自然だ。

## デザインとコードの橋渡し

LottieとRiveの本質的な価値は「技術的な実装能力」ではなく「デザイナーとエンジニアの協力のあり方」を変えることにある。

「このアニメーション、コードで再現するのは難しい」という諦めや、「エンジニアに実装できる範囲に合わせてデザインを妥協する」という制約から解放される。デザイナーの意図をそのままコードに持ち込める。

アプリの「感触の良さ」は、こういった小さな摩擦の積み重ねを減らすことから生まれる。ツールの選択は技術的な問いである前に、チームがどう協働するかという問いでもある。
