# マルチエージェントシステムの「役割分担」をどう設計するか

ひとつのLLMにすべてを任せる時代は、少しずつ変わりつつある。複数のエージェントが協調して動く「マルチエージェントシステム」は、特定のタスクにおいて単一エージェントより高い精度と柔軟性を発揮する。

しかし、マルチエージェントはそれ自体が「複雑さ」を持ち込む技術でもある。設計を誤ると、エージェント間の責任が曖昧になり、デバッグ不能なカオスが生まれる。

この記事では、マルチエージェントシステムにおける「役割分担」の設計思想を整理する。

## なぜシングルエージェントでは足りないのか

まず前提として、シングルエージェントの限界を理解しておく必要がある。

単一のLLMに複雑なタスクを渡すと、いくつかの問題が顕在化する。

**コンテキストウィンドウの限界**。複雑なタスクは多くの情報を必要とする。調査・計画・実行・検証を一つのエージェントが担うと、コンテキストが膨大になり、精度が落ちる。

**役割の曖昧さ**。ひとつのエージェントが「考える係」と「実行する係」を兼任すると、どちらの精度も中途半端になりやすい。人間でも、同じ人が設計と施工を同時に担うと品質が下がるのと同じ論理だ。

**並列処理の困難**。独立したサブタスクが複数ある場合、シングルエージェントは直列にしか処理できない。マルチエージェントなら並列実行が可能で、全体のスループットが向上する。

## マルチエージェントの基本トポロジー

マルチエージェントシステムには、大きく3つの構造パターンがある。

### オーケストレーター型（中央制御）

一人の「指揮者エージェント」が全体の計画を立て、専門エージェントたちに指示を出すパターン。

```
Orchestrator Agent
├── Research Agent（情報収集）
├── Analysis Agent（分析）
└── Writer Agent（文章生成）
```

指揮者は「何を・誰に・いつ頼むか」だけを考え、実際の作業は専門エージェントに委ねる。オーケストレーターのコンテキストは常にスリムに保たれ、全体の流れを把握することに集中できる。

**向いているケース**: タスクに明確な順序がある、全体の品質管理が重要、エラーハンドリングを中央集権にしたい。

### ピアツーピア型（分散協調）

エージェント同士が直接コミュニケーションし、必要に応じて相互に助け合うパターン。中央制御がないため、単一障害点がない。

**向いているケース**: タスクが並列に分解可能、エージェント間の依存が少ない、スケーラビリティが求められる。

### レビュー型（批評と改善）

ひとつのエージェントが生成し、別のエージェントがレビューする往復型のパターン。

```
Generator Agent → Critic Agent → Generator Agent（修正）→ ...
```

これはLLM特有の「自己評価の弱さ」を補う構造だ。同じモデルが自分の出力を評価しても精度が低いが、別のエージェント（時には別のモデル）が評価することで精度が上がる。

**向いているケース**: 出力品質が最重要、ハルシネーションを減らしたい、文章や計画の改善ループが必要。

## 「役割」の粒度をどう決めるか

設計で最も難しいのは、役割の粒度をどう設定するかだ。

**粒度が細かすぎる問題**。エージェントが増えるほど、通信コスト・レイテンシ・コストが増大する。「調査エージェント」「URL抽出エージェント」「タイトル抽出エージェント」…と分割しすぎると、オーバーヘッドが支配的になる。

**粒度が粗すぎる問題**。逆に大きな役割を一つのエージェントに詰め込むと、マルチエージェント化の恩恵が薄れる。

実践的な指針として、**「ひとつの役割がひとつの明確な成果物を生む」** レベルで分割するのが良い。成果物が「調査レポート」「分析サマリー」「最終文書」のように明確なら、それが適切な粒度だ。

## エージェント間の情報共有をどう設計するか

マルチエージェントのもう一つの難所は、エージェント間でどう情報を渡すかだ。

### メッセージパッシング

エージェント間でメッセージを直接やり取りする方式。シンプルだが、エージェントが増えると「誰が何を知っているか」の管理が難しくなる。

### 共有メモリ（ブラックボード）

すべてのエージェントがアクセスできる共有の「黒板（ブラックボード）」に情報を書き込み・読み取る方式。

```
[Shared Memory / Blackboard]
  - task_status: "in_progress"
  - research_result: {...}
  - analysis_done: true
```

エージェントは自分が必要な情報だけをブラックボードから取得し、完了したらそこに書き込む。中央集権ではなく、情報だけが共有される構造だ。

大規模なシステムでは、Vector DBやKVストアがこのブラックボードの役割を担うことが多い。

## 失敗から学ぶ：よくある設計ミス

### 「エージェントに考えさせすぎる」

エージェントの役割に「何でもやる」と書いてしまうと、エージェントは何でもやろうとする。役割は**制約**でもある。「これはやらない」を明示的に書くことで、エージェントの動作が安定する。

### 「ループ終了条件がない」

エージェントが互いに「もっと情報が必要」と判断し続けると、無限ループに陥る。オーケストレーターには常に**最大反復回数**や**タイムアウト**を設けることが必須だ。

### 「エラーが伝播しない」

あるエージェントが失敗しても、次のエージェントに「失敗した」という情報が伝わらないと、誤った情報を基に処理が続く。エラー状態を明示的に型として扱い、次のエージェントが適切に判断できる設計にすべきだ。

## いつマルチエージェントを使うべきか

マルチエージェントは万能ではない。以下の基準で判断するのが現実的だ。

**使うべきとき**:
- タスクが自然に分解できる（研究→分析→執筆など）
- シングルエージェントではコンテキストが溢れる
- 品質向上のためにレビューループが欲しい
- 並列処理でスループットを上げたい

**使わなくていいとき**:
- シンプルな1ステップのタスク
- レイテンシが最優先（エージェント間の通信コストがある）
- デバッグのコストを下げたい

## まとめ

マルチエージェントシステムの本質は、「複雑な問題を、得意な存在たちで分担する」という人間の組織設計に似ている。役割を明確に、情報共有をシンプルに、終了条件を必ず定義する。

「なぜ複数エージェントが必要か」という問いを常に持ちながら設計することで、不必要な複雑さを避け、真に価値のある協調システムを構築できるはずだ。
