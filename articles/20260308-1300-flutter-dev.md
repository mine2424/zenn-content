## `async/await` だけでは解決できない問題がある

Flutterでの非同期処理というと、まず `async/await` と `Future` が思い浮かぶ。APIからのデータ取得やファイルの読み書きなど、I/O待ちが伴う処理はこれで十分に対処できる。

しかし、アプリが成長するにつれて直面する問題がある。JSONのパースや画像処理、暗号化、大量データのソートといった「計算量が多い処理」だ。

`async/await` は「待ち時間」を非同期にするのであって、CPUを使う計算処理は非同期にしても同じスレッドで実行される。Dartは基本的にシングルスレッドで動作するため、重い計算処理を実行している間、UIの描画（フレームのレンダリング）も止まってしまう。フレームが16ms以内に描画されなければ、ユーザーにはカクつきとして体感される。

これを解決するのがDartの **Isolate** だ。

## IsolateはOSスレッドに近いが、メモリを共有しない

Isolateは「隔離された実行環境」だ。それぞれが独立したメモリを持ち、メッセージパッシングによってのみ通信できる。同じプロセス内で複数のIsolateを動かすことで、マルチコアCPUを活用した真の並行処理が実現できる。

Dartの特徴的な設計は「メモリを共有しない」という点だ。JavaやKotlinのスレッドでは、共有メモリへの同時アクセスによる競合状態（Race Condition）がバグの温床になる。Isolateはメモリを完全に分離することでこの問題を構造的に回避している。代わりに、データはシリアライズしてメッセージとして送受信する必要があるが、コードの安全性は大幅に高まる。

## `compute` 関数：最もシンプルなIsolate利用

Flutterには `compute` という高レベルなAPIが用意されており、重い処理を別のIsolateで実行する最も手軽な方法だ。

```dart
// 重い処理を行う関数（トップレベル関数である必要がある）
List<User> parseUsers(String jsonString) {
  final List<dynamic> jsonList = jsonDecode(jsonString);
  return jsonList.map((e) => User.fromJson(e)).toList();
}

// UIスレッドからの呼び出し
final users = await compute(parseUsers, jsonString);
```

`compute` は引数としてトップレベル関数（またはstatic関数）とその引数を受け取り、新しいIsolateを生成してその中で関数を実行し、結果を返してくれる。内部的にはIsolateの生成・通信・終了を自動でハンドリングしてくれるため、シンプルな一度きりの重い処理には最適だ。

注意点は、渡す引数と戻り値は **シリアライズ可能な型** である必要があることだ。プリミティブ型、リスト、マップ、そして `@immutable` なデータクラスは基本的に問題ない。クロージャやネイティブリソースを持つオブジェクトは渡せない。

## `Isolate.spawn` と `ReceivePort`：継続的な通信が必要な場合

`compute` は「一度だけ重い処理をする」ケースに向いているが、Isolateと継続的にメッセージをやり取りする場合は `Isolate.spawn` と `ReceivePort`・`SendPort` を直接使う。

```dart
void heavyWorker(SendPort sendPort) {
  // このIsolateは生き続けてメッセージを受け取る
  final receivePort = ReceivePort();
  sendPort.send(receivePort.sendPort); // こちらのSendPortを返す

  receivePort.listen((message) {
    // メッセージを受け取って処理し、結果を送り返す
    final result = doHeavyWork(message);
    sendPort.send(result);
  });
}
```

このパターンは、バックグラウンドでデータを継続的に処理するワーカー（例えばリアルタイムデータの変換やセンサーデータの処理）に向いている。

## Flutter 3以降の `Isolate.run`

Flutter 3（Dart 2.19）以降では、`Isolate.run` というシンプルなAPIが追加された。

```dart
final result = await Isolate.run(() => parseUsers(jsonString));
```

`compute` と同様のシンプルさで、かつトップレベル関数の制約がない（クロージャも使える）。新しいコードでは基本的にこちらを使う方がシンプルに書ける。

## どんな処理をIsolateに出すべきか

すべての処理をIsolateに移すのは逆効果だ。Isolateの生成とメッセージのシリアライズにはオーバーヘッドがあるため、軽い処理ならメインIsolateで直接実行した方が速い。

Isolateが有効なケース：
- **大量JSONのパース**：数百KBを超えるJSONの `jsonDecode` + モデル変換
- **画像処理**：フィルタ適用、リサイズ、カラー変換など
- **暗号化・ハッシュ計算**：パスワードハッシュやデータ暗号化
- **大量データのソート・フィルタリング**：数万件以上のリストへの操作
- **PDFやファイルの生成**：計算量が多いドキュメント生成処理

Isolateが不要なケース：
- API通信の待機（I/O待ちは `async/await` で十分）
- 数十件程度のリスト操作
- 状態の管理・更新（UIと共有する状態はIsolateでは管理できない）

## Isolateとデバッグ

Isolate内の処理はFlutter DevToolsのProfillerには別のスレッドとして表示される。UIジャンクが発生しているときに、Profilerでメインスレッド（UIスレッド）が高負荷になっていれば、重い計算処理がメインIsolateで実行されているサインだ。その処理をIsolateに移すことで改善できる。

エラーハンドリングについては、`Isolate.run` や `compute` は内部で例外を捕捉してメインIsolateで再スローしてくれるため、通常の `try-catch` で対応できる。

## 「UIを止めない」を設計の原則にする

Isolateを使う上での最も重要な判断は、「この処理はUIスレッドで本当に実行する必要があるか？」という問いかけだ。

Flutterのパフォーマンスチューニングの多くは「メインIsolateで何をしないか」の選択に帰着する。Isolateへの処理の移譲は、ユーザーが感じる「サクサク感」を守るための設計判断であり、アプリの品質を左右する重要なアーキテクチャの決断だ。

`async/await` で非同期にしても解決しないカクつきに出会ったとき、それはIsolateを使うタイミングかもしれない。その一線を理解することが、Flutterのパフォーマンス設計の第一歩になる。
