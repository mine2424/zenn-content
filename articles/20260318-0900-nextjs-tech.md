## 「リアルタイム」とは何を意味するのか

Webアプリケーションにおける「リアルタイム通信」という言葉は、しばしば曖昧に使われる。チャット、ライブ更新、通知バナー、コラボレーション編集——これらはすべて「リアルタイム」と呼ばれるが、その要件は大きく異なる。

重要なのは、「何を・どの方向に・どれだけの頻度で」送りたいのかを先に明確にすることだ。その答えによって、選ぶべき技術は変わる。

Next.js App Router時代における主な選択肢は大きく3つある。

1. **Server-Sent Events（SSE）** — サーバー→クライアントの一方向ストリーム
2. **WebSocket** — 双方向の全二重通信
3. **ポーリング（Long Polling含む）** — 古くからある定番手法

この記事では、それぞれの仕組みと設計上の向き不向きを整理したうえで、Next.jsにおける実装の考え方を解説する。

---

## Server-Sent Events：HTTPの延長としてのストリーム

SSEはHTTPの仕組みをそのまま活かした技術だ。クライアントがGETリクエストを送ると、サーバーはレスポンスを「閉じない」まま、テキストイベントを逐次プッシュし続ける。

### Next.js Route HandlerでのSSE

App RouterのRoute Handlerは`ReadableStream`を返せるため、SSEとの相性が非常によい。

```typescript
// app/api/events/route.ts
export async function GET() {
  const encoder = new TextEncoder();

  const stream = new ReadableStream({
    start(controller) {
      const interval = setInterval(() => {
        const data = `data: ${JSON.stringify({ time: Date.now() })}\n\n`;
        controller.enqueue(encoder.encode(data));
      }, 1000);

      // クリーンアップ
      return () => clearInterval(interval);
    },
  });

  return new Response(stream, {
    headers: {
      "Content-Type": "text/event-stream",
      "Cache-Control": "no-cache",
      Connection: "keep-alive",
    },
  });
}
```

クライアント側は`EventSource` APIで接続する。再接続も自動で行われるため、ネットワークが一時的に切れても自動復帰する。これはSSEの大きな強みだ。

### SSEが向いているケース

- サーバーからクライアントへのプッシュ通知（新着メッセージ、ステータス更新）
- AI生成テキストのストリーミング表示（LLMの逐次出力）
- ダッシュボードのライブデータ更新
- CI/CDパイプラインのログストリーミング

クライアントからサーバーへのデータ送信は通常のfetchやAPIコールで賄えるため、**「サーバーが主体的にイベントを送る」設計**においてSSEは非常にシンプルな解だ。

## WebSocket：本当に双方向が必要な場面だけ使う

WebSocketはHTTPアップグレードを経由して確立される持続的な双方向チャネルだ。クライアントとサーバーがリアルタイムに互いへメッセージを送れる。

ただし、App RouterのRoute HandlerはWebSocketをネイティブにサポートしていない。これはEdge RuntimeがWebSocketのハンドシェイクを正しく処理できないためだ。Next.js単体でWebSocketを実装する場合、現時点では以下のアプローチが現実的だ。

**パターン1：カスタムサーバーを使う**
`next/server`ではなく`server.js`でNode.jsのhttpサーバーを起動し、そこに`ws`ライブラリを組み合わせる。ただし、Vercelへのデプロイ時はサーバーレス化されるためWebSocketは動作しない。

**パターン2：外部サービスに切り出す**
Pusher、Ably、Supabase Realtime、Partykit——これらの「マネージドWebSocketサービス」をNext.jsと組み合わせる。Next.js本体のデプロイはVercel等のサーバーレスに任せ、WebSocketサーバーだけを外部サービスに委ねる構成だ。

**パターン3：専用サーバーを立てる**
WebSocketを担うExpressサーバーやBunアプリをDockerで別途運用し、Next.jsはUIとREST APIのみを担当する。マイクロサービス的な分離だ。

### WebSocketが本当に必要な場面

- コラボレーションツール（複数ユーザーが同時編集）
- マルチプレイヤーゲーム
- リアルタイムカーソル共有、ホワイトボード
- クライアント→サーバーへも高頻度でデータを送る必要がある場合

逆に言えば、「通知を受け取るだけ」「AIの返答を表示するだけ」という用途ならWebSocketは過剰設計になる。

---

## ポーリング：シンプルさが武器になる場面

SSEもWebSocketも必要なく、数秒〜数十秒に1回更新されれば十分なら、ポーリングは依然として有効な選択肢だ。

Next.jsのRoute Handlerに対して`setInterval`でGETリクエストを送る実装は、複雑な設定なしに動き、Vercelへの通常デプロイも問題ない。Long Pollingであれば、サーバー側でデータが来るまでレスポンスを保留することで疑似的なプッシュも実現できる。

「新着注文の通知を30秒ごとに確認したい」レベルであれば、ポーリングの方がインフラへの負荷が予測しやすく、実装もデバッグも容易だ。

## 選択の判断軸を整理する

| 要件 | 推奨アプローチ |
|------|----------------|
| サーバー→クライアントのみ | SSE |
| LLMテキストストリーミング | SSE |
| チャット・コラボレーション | WebSocket（外部サービス推奨） |
| 低頻度な更新確認 | ポーリング |
| Vercelデプロイ前提 | SSE または ポーリング または外部WS |

Vercelを使っている場合、WebSocketのサポートはBeta段階（2024年時点）であり、本番での採用には慎重さが必要だ。Edge Runtimeの制約を把握したうえで選択する必要がある。

## 状態管理との統合を考える

リアルタイム通信はデータフローの問題でもある。SSEやWebSocketで受け取ったイベントをUIに反映するためには、状態管理との接続が必要だ。

一般的なパターンは、カスタムフックにストリームの購読ロジックをカプセル化し、`useState`や`useReducer`に流し込む形だ。Zustandのような外部ストアと組み合わせることで、複数のコンポーネントへの配信も容易になる。

Server Componentsはリアルタイム更新と直接相性が悪い（サーバーサイドでレンダリングされるため）。リアルタイム更新が必要なコンポーネントは`"use client"`でClient Componentとして定義し、そこでフックを呼ぶ設計が自然だ。

## まとめ：通信方向と頻度で技術を選ぶ

Next.jsでリアルタイム通信を設計するとき、最初に問うべきは「誰が誰に、何をどれくらいの頻度で送るか」だ。

- **一方向・高頻度プッシュ** → SSE（App Routerとの親和性が高い）
- **双方向・高頻度** → WebSocket（外部サービスか専用サーバーへの委譲を検討）
- **低頻度・シンプル** → ポーリング

「リアルタイム」という言葉に引っ張られてWebSocketを選びがちだが、多くのユースケースはSSEで十分に解決できる。インフラの制約（特にサーバーレス環境）を踏まえたうえで、最もシンプルな解から検討を始めることが、保守性の高い設計への近道だ。
