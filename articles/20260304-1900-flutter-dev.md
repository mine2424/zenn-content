## はじめに

Flutterは「一つのコードベースでiOS・Android両方に対応できる」ことを強みとする。しかし現実には、ネイティブプラットフォームの機能にアクセスしなければならない場面が必ず訪れる。

Bluetooth通信、生体認証、カメラのカスタム制御、プラットフォーム固有のUI——これらはFlutterのDartコードだけでは扱えない。Platform Channelはこの「DartとネイティブコードをつなぐAPI」だ。

本稿では、Platform Channelの設計思想と実装の考え方を整理する。

## Platform Channelが解決する問題

Flutterのアーキテクチャを理解するには、レイヤー構造を把握する必要がある。

Flutterエンジンはネイティブアプリの「ホスト」の中に組み込まれる。Dart/Flutterのレイヤーとネイティブ（iOS/Android）のレイヤーは分離した世界として動作している。

この2つの世界を繋ぐのがPlatform Channelだ。メッセージングシステムをベースにしており、DartからネイティブへのメッセージとネイティブからDartへの応答（またはイベント通知）を非同期で処理する。

重要なのは、Platform Channelは「メソッド呼び出しのラッパー」ではなく、**シリアライズされたメッセージのやりとり**だということだ。Dartのオブジェクトはそのままネイティブに渡せるわけではなく、MessageCodecによってエンコードされてから送信される。

## 3種類のChannelとその使い分け

Flutterには3つのChannel型が用意されている。

### MethodChannel

最も一般的なChannel。DartからネイティブのメソッドをCall-and-Responseの形で呼び出す。

```dart
final channel = MethodChannel('com.example.app/biometrics');

Future<bool> authenticate() async {
  final result = await channel.invokeMethod<bool>('authenticate');
  return result ?? false;
}
```

ネイティブ側（iOS/Swift）では`FlutterMethodChannel`を使い、`setMethodCallHandler`で呼び出しを受け取る。「Dartがリクエストを送り、ネイティブが結果を返す」という1回限りのやりとりに向いている。

### EventChannel

ネイティブからDartへの継続的なストリームデータの送信に使う。加速度センサー、位置情報の連続取得、Bluetoothデバイスのスキャン結果など、時系列データの流し込みに適している。

Dart側ではStreamとして受け取れるため、Flutterのリアクティブな処理と自然につながる。

### BasicMessageChannel

双方向のシンプルなメッセージ送受信。頻繁には使われないが、独自のコーデックを使いたい場合や、ネイティブ→Dartの単発通知に使うことがある。

## Channel設計で最初に考えること

Platform Channelを設計する際、最初に問うべきことは「どこまでをDart、どこからをネイティブにするか」だ。

この境界設計が後の保守性を大きく左右する。

**Dartレイヤーに置くべきもの：**
- ビジネスロジック全般
- UIとの統合処理
- エラーハンドリングとフォールバック

**ネイティブレイヤーに任せるべきもの：**
- プラットフォームAPIの呼び出しそのもの
- プラットフォーム固有のパーミッション処理
- ネイティブSDKとの統合

よくある失敗は、ネイティブ側にビジネスロジックを書きすぎることだ。Dart側では「認証が成功したか失敗したか」だけを受け取り、その後の振る舞いはDartで制御する——このように境界を「データの受け渡し」に絞ると、プラットフォーム間の差異をDart層で吸収できる。

## Channel名の設計

MethodChannelのコンストラクタに渡す名前（チャンネル識別子）は、アプリケーション内でユニークである必要がある。公式推奨は逆ドメイン記法だ。

```
com.yourcompany.yourapp/feature_name
```

複数のChannelを作る場合、機能単位でChannelを分けることが推奨される。カメラ用のChannel、Bluetooth用のChannel、センサー用のChannel——それぞれを独立させることで、ネイティブ実装の変更が他のChannelに影響しなくなる。

1つのChannelに全機能を詰め込む「God Channel」パターンは避けるべきだ。ネイティブ側のhandlerが巨大になり、Channelの責務が不明確になる。

## 型安全性とエラーハンドリング

Platform Channelのやりとりはシリアライズを挟むため、型の安全性が自動では保証されない。`invokeMethod`は`dynamic`を返し、型キャストは開発者の責任だ。

また、ネイティブ側で例外が発生した場合、Dart側では`PlatformException`として受け取る。このエラーハンドリングを丁寧に設計することが重要だ。

実践的なアプローチとして、MethodChannelを直接使う代わりに**Repositoryパターンでラップする**ことが多い。DartコードがMethodChannelに直接依存するのではなく、抽象化されたinterfaceに依存する設計にすることで、テスト時にモックと差し替えられる。

```dart
abstract class BiometricsRepository {
  Future<AuthResult> authenticate();
}

class PlatformBiometricsRepository implements BiometricsRepository {
  final _channel = MethodChannel('com.example.app/biometrics');
  
  @override
  Future<AuthResult> authenticate() async {
    try {
      final result = await _channel.invokeMethod<bool>('authenticate');
      return AuthResult(success: result ?? false);
    } on PlatformException catch (e) {
      return AuthResult(success: false, errorCode: e.code);
    }
  }
}
```

テストでは`MockBiometricsRepository`を注入することで、ネイティブコードを動かさずにビジネスロジックをテストできる。

## Pigeon：型安全なコード生成

Platform Channelの型安全性の問題に対するFlutterチームの回答が`Pigeon`だ。

Pigeonはスキーマ定義から、Dart側とネイティブ側（Swift/Kotlin）のボイラープレートコードを自動生成するコードジェネレーターだ。手動でMethodChannelを書く代わりに、Dartで定義したインターフェースからコードを生成する。

生成されたコードは型チェックをコンパイル時に行えるため、メソッド名のタイポや型ミスマッチをビルド前に検出できる。プロダクションでネイティブ連携が多い場合、Pigeonの導入はChannel設計の品質を大きく改善する。

## パフォーマンスの注意点

Platform Channelの呼び出しは、同一プロセス内の関数呼び出しと比べてコストが高い。メッセージのシリアライズ/デシリアライズ、スレッドをまたいだ通信が発生する。

頻繁なセンサーデータの取得（毎フレームなど）をMethodChannelで行うと、パフォーマンスボトルネックになりうる。高頻度のデータ更新にはEventChannelを使い、バッファリングをネイティブ側で行うなどの工夫が必要になる。

一方、起動時の一度きりの設定値取得や、ユーザーアクションに応じた認証など、低頻度の呼び出しであればMethodChannelで十分だ。

## おわりに

Platform Channelは「FlutterとネイティブをつなぐAPI」だが、その本質は**境界の設計**にある。何をDartで管理し、何をネイティブに委ねるかという設計判断が、コードの保守性・テスタビリティ・パフォーマンスに直結する。

「Dartで書けないからChannelを使う」という消極的な姿勢ではなく、「この機能はネイティブレイヤーで最も自然に実装できる」という設計的な判断としてChannelを使う視点を持つと、より良いFlutterアーキテクチャに近づける。
