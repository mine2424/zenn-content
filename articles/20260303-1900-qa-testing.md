## はじめに

マイクロサービスやフロントエンド・バックエンド分離のアーキテクチャで、こんな経験をしたことはないだろうか。

バックエンドチームがAPIのレスポンス形式を変更した。ちゃんとドキュメントを更新したし、Slackでも通知した。でも、フロントエンドチームが気づかず、本番デプロイ後にエラーが発生した——。

あるいはその逆。フロントエンドが「このフィールドはもう使っていない」と思っていたが、別のサービスがそのフィールドに依存していた。

こういった「サービス間の約束の破れ」を検知するのが**コントラクトテスト**であり、その代表的なツールが**Pact**だ。

## E2Eテストでは解決できない問題

サービス間の統合テストとして、多くのチームはE2Eテストを使う。実際に全サービスを起動して、シナリオを流す方法だ。

しかしE2Eテストには構造的な限界がある。

**フィードバックが遅い**。全サービスを起動する必要があるため、テスト実行に時間がかかる。CIで問題を検知するまでに数十分かかることも珍しくない。

**どのサービスが壊れたか分かりにくい**。E2Eテストが失敗したとき、原因がバックエンドなのかフロントエンドなのか、それともインフラなのかが分かりにくい。

**組み合わせが爆発する**。サービスが増えるにつれて、全ての組み合わせをE2Eでテストすることは現実的でなくなる。

コントラクトテストはこれらの問題に別のアプローチで立ち向かう。

## コントラクトテストの仕組み

コントラクトテストの核心にあるのは「**コンシューマー駆動**」という考え方だ。

APIを呼び出す側（コンシューマー）が「自分はこういうリクエストを送るし、こういうレスポンスを期待する」という**コントラクト（契約）**を定義する。APIを提供する側（プロバイダー）は、そのコントラクトを満たしているか検証する。

Pactの場合、このフローは次のようになる。

1. コンシューマー側のテストで、モックサーバーに対してAPIを呼び出す
2. その際の「リクエストとレスポンスの期待値」がPactファイルとして出力される
3. プロバイダー側のテストで、実際のサービスがPactファイルを満たせるかを検証する

```json
// 自動生成されるPactファイルのイメージ
{
  "consumer": { "name": "frontend-app" },
  "provider": { "name": "user-api" },
  "interactions": [
    {
      "description": "ユーザー情報の取得",
      "request": {
        "method": "GET",
        "path": "/users/123"
      },
      "response": {
        "status": 200,
        "body": {
          "id": 123,
          "name": "John Doe",
          "email": "john@example.com"
        }
      }
    }
  ]
}
```

このコントラクトファイルが「サービス間の約束書」になる。

## 何が嬉しいのか

### 独立したデプロイが可能になる

コントラクトが明示されていれば、「このコントラクトを満たしている限り、どちらのサービスが先にデプロイされても問題ない」と言える。

サービス間の結合度を下げ、チームが独立してリリースできる状態を作ることがコントラクトテストの最大の価値だ。

### 破壊的変更を早期に検知できる

プロバイダーがAPIの変更を加えたとき、既存のコントラクトを破る変更であれば即座に検知できる。全サービスを起動したE2Eテストを待つ必要はない。

「Can I Deploy」という概念——「このバージョンを本番にデプロイしても安全か？」——をPact Brokerという仕組みと組み合わせることで、デプロイの安全性を継続的に確認できる。

### ドキュメントとして機能する

Pactファイルは、コンシューマーが実際に使っているAPIのインターフェースを正確に記述している。これは、OpenAPIスペックのような「あるべき姿」ではなく、「実際に使われている姿」のドキュメントだ。

使われていないエンドポイントや使われていないフィールドを特定する手がかりにもなる。

## コントラクトテストの限界

コントラクトテストは万能ではない。いくつかの重要な限界を理解しておく必要がある。

**ビジネスロジックはテストしない**。コントラクトテストは「インターフェースの約束を守っているか」を確認するものだ。「正しいビジネスロジックで処理されているか」は対象外だ。

**複雑な状態依存のシナリオは苦手**。「ユーザーが〇〇した後に△△するとどうなるか」のような、複数ステップにまたがるシナリオはコントラクトテストで表現しにくい。

**チームの協力が必要**。コンシューマーとプロバイダーの両チームがPactを採用・維持する必要がある。片方だけが使っても効果は半減する。

テストピラミッドの中でコントラクトテストは、ユニットテストとE2Eテストの間に位置する。E2Eテストを完全に置き換えるものではなく、**重複を減らしながら検知速度を上げる**ための道具だ。

## 導入のタイミングを考える

コントラクトテストが特に効果を発揮するのは、次のような状況だ。

- フロントエンドとバックエンドが別チームで開発されている
- マイクロサービス間のAPI変更が頻繁に起きている
- E2Eテストのメンテナンスコストが高くなっている
- チームが独立してデプロイできる状態を目指している

逆に、小規模なモノリシックなアプリケーションや、フロントエンドとバックエンドを同じチームが開発している場合は、コストに見合わないことが多い。

## まとめ

コントラクトテストが解決するのは「**サービス間の期待のずれ**」という問題だ。

- コンシューマーが期待するインターフェースをコントラクトとして明示する
- プロバイダーはそのコントラクトを継続的に検証する
- 約束の破れをE2Eより早いフィードバックサイクルで検知できる

ツールより先に「何を約束として定義するか」という設計の議論をすることが、コントラクトテスト導入の第一歩だ。Pactはその議論を構造化する手助けをしてくれる道具に過ぎない。
