## 画像はWebパフォーマンスの最大の敵であり、最大の改善余地だ

Webページのバイト数の約半分を画像が占めると言われる。PageSpeed Insightsで赤判定を受けるページの多くで、画像の最適化が改善項目として挙がる。特に **LCP（Largest Contentful Paint）**——最大のコンテンツが描画されるまでの時間——は、多くのケースでヒーロー画像やOGP画像が原因で遅くなる。

Next.jsはこの問題を `next/image` コンポーネントによって解決しようとしている。ただ、コンポーネントを置き換えるだけで最適化が完了するわけではない。next/imageが何をしてくれて、何をしてくれないのかを理解することが、本当の最適化への第一歩だ。

## next/imageが自動でやってくれること

`next/image` を使うと、以下が自動で適用される。

**フォーマット変換**: ブラウザがサポートしている場合、WebPやAVIFに自動変換する。AVIFはJPEGの約半分のファイルサイズで同等の品質を実現できる。古いブラウザには適切なフォールバックフォーマットが提供される。

**サイズのリサイズ**: `width` と `height` プロパティを指定すると、実際に表示されるサイズに合わせてリサイズされた画像がサーバーサイドで生成される。4000×3000の高解像度写真を400px幅で表示するために元画像を配信する無駄がなくなる。

**遅延読み込み（Lazy Loading）**: デフォルトで `loading="lazy"` が設定される。ビューポート外の画像は、ユーザーがスクロールして近づくまで読み込まれない。

**サイズの予約（CLSの防止）**: `width` と `height` の指定により、ブラウザが画像読み込み前にスペースを確保する。これにより **CLS（Cumulative Layout Shift）**——コンテンツが読み込み中に動いてしまう現象——を防ぐ。

## `priority` プロパティがLCPを左右する

next/imageで最も見落とされがちで、最もインパクトの大きいプロパティが `priority` だ。

デフォルトの lazy loading はビューポート外の画像には効果的だが、**最初のビューポートに表示されるヒーロー画像に対しては逆効果**になる。なぜなら、lazy loadingが有効だと画像の読み込みが遅延され、LCPを悪化させるからだ。

```tsx
// ヒーロー画像には必ずpriority={true}を指定する
<Image
  src="/hero.jpg"
  alt="ヒーロービジュアル"
  width={1200}
  height={600}
  priority={true}
/>
```

`priority={true}` を指定すると、next/imageは `<link rel="preload">` タグをドキュメントの `<head>` に自動で追加し、ブラウザに「この画像を優先して読み込め」と指示する。

**ルール**: 最初のビューポートに表示される最大の画像（= LCPの原因となりうる画像）には `priority={true}` を設定する。それ以外には設定しない。不必要に `priority` を多用すると、他のリソースの読み込みを遅らせることになる。

## `sizes` プロパティで「どのサイズを配信するか」を正確に伝える

next/imageはデフォルトで複数の解像度の画像（srcset）を生成する。ブラウザは `sizes` プロパティを参照して、現在のビューポートで適切なサイズを選択する。

`sizes` を正確に指定することで、必要以上に大きな画像を配信するムダをなくせる。

```tsx
<Image
  src="/product.jpg"
  alt="商品画像"
  width={800}
  height={600}
  sizes="(max-width: 768px) 100vw, (max-width: 1200px) 50vw, 33vw"
/>
```

この例では「モバイル（768px以下）ではビューポート幅全体、タブレット（1200px以下）では50%、デスクトップでは33%」と指示している。ブラウザはこの情報をもとに最適なサイズの画像を選ぶ。

`sizes` を省略すると、ブラウザはデスクトップ向けの大きな画像をモバイルでも読み込む可能性がある。特にモバイルユーザーが多いサービスでは、`sizes` の設定が大幅なデータ削減につながる。

## `fill` レイアウトと `object-fit` の組み合わせ

コンテナのサイズに合わせて画像を表示したい場合（アスペクト比が不明な画像を一定の枠内に収めるケースなど）は、`fill` プロパティが便利だ。

```tsx
<div style={{ position: "relative", width: "100%", height: "300px" }}>
  <Image
    src="/thumbnail.jpg"
    alt="サムネイル"
    fill
    style={{ objectFit: "cover" }}
  />
</div>
```

`fill` を使う場合、親要素に `position: relative`（または absolute / fixed）が必要だ。`objectFit: "cover"` は画像をコンテナに収めながら比率を保つ（はみ出た部分はクロップ）。`objectFit: "contain"` はコンテナ内に全体が収まるよう縮小する。

## 外部ドメインの画像を使う場合の設定

CDNや外部サービスの画像を `next/image` で使う場合、`next.config.js` で許可するドメインを設定する必要がある。

```javascript
// next.config.js
module.exports = {
  images: {
    remotePatterns: [
      {
        protocol: "https",
        hostname: "images.example.com",
        pathname: "/products/**",
      },
    ],
  },
};
```

この設定は**セキュリティ上の理由から必須**だ。任意のURLの画像を最適化APIに渡せてしまうと、サーバーリソースを悪用される恐れがある。`remotePatterns` でパスも制限できるため、できる限り細かく設定するのが望ましい。

## 画像最適化の費用対効果を考える

Vercelのホスティングを使う場合、next/imageによる画像変換はリクエスト数にカウントされる。アクセス数の多いサービスでは、この費用が無視できなくなることがある。

その場合の選択肢として、**カスタムローダー**がある。Cloudinary、Imgix、CloudflareのImage Resizingなどのサービスを使い、変換処理を外部に委ねることができる。`next/image` のインターフェースはそのまま使いつつ、バックエンドの変換サービスを差し替えられる設計だ。

## 設計の原則：何を「最適化すべきか」を選ぶ

next/imageは強力なツールだが、「すべての `<img>` を `<Image>` に置き換えれば最適化できる」という発想は危険だ。

たとえばBase64埋め込みのアイコンやSVGは、next/imageの最適化対象にならない。サーバーサイドで動的に生成される画像は、別途キャッシュ戦略が必要になる。

本当に大事なのは「どの画像がユーザー体験に最も影響するか」を分析し、LCPを構成している画像を優先的に最適化することだ。PageSpeed InsightsやLighthouseの「LCP element」セクションが、優先対象を特定する出発点になる。

画像最適化は地味な作業だが、LCPの改善はSEOスコアと直接連動し、ユーザーの離脱率にも影響する。「画像を送るだけ」から「ユーザーに必要な分だけ、必要なときに」送る設計への移行が、next/imageの本質的な価値だ。
