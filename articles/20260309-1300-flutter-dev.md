## はじめに

Flutterアプリが成長するにつれ、「どこでどのクラスを生成するか」という問いが設計の核心になる。Repositoryを使うUsecase、UsecaseをViewModelに渡す、ViewModelをWidgetに注入する——こうした依存関係の連鎖を整理する仕組みが**依存性注入（Dependency Injection, DI）**だ。

FlutterのDIソリューションとして最も広く使われているのが**GetIt**（サービスロケーター）と、その上に構築されたコード生成ライブラリ**injectable**だ。本記事では、「なぜDIが必要か」という問いから始め、GetItとinjectableの設計思想を考えていく。

## 依存性注入が解決する問題

まず「DIなし」の世界を考えてみよう。

```dart
class UserViewModel {
  final UserRepository _repository = UserRepositoryImpl(
    api: ApiClient(baseUrl: 'https://api.example.com'),
    cache: LocalCache(),
  );
  // ...
}
```

このコードはViewModelが`UserRepositoryImpl`の具体的な実装と、その依存先（`ApiClient`や`LocalCache`）の生成責任を持っている。問題は2つある。

**テストが困難だ。** ViewModelのテストでは実際の`ApiClient`が呼ばれてしまい、ネットワークに依存したテストになる。モックに差し替えることができない。

**責任の分離ができていない。** ViewModelは本来「UIの状態を管理する」責任を持つべきだが、依存関係の構築という全く別の責任を担っている。

DIはこの問題を「依存関係の生成と利用を分離する」ことで解決する。クラスは「何に依存するか」を宣言するだけで、「どうやって生成するか」は外部に委ねる。

## GetItの設計思想：サービスロケーター

GetItは**サービスロケーター**パターンを採用している。アプリの起動時に依存関係を登録し、必要な場所から取り出すというアプローチだ。

```dart
// 登録
final getIt = GetIt.instance;
getIt.registerSingleton<UserRepository>(UserRepositoryImpl());

// 取り出し
final repository = getIt<UserRepository>();
```

シンプルだが、ここに重要な設計判断がある。GetItはコンテナへの**グローバルなアクセス**を提供する。`getIt<T>()`という呼び出しはコードのどこからでもできる。

これは利便性が高い反面、「どのクラスが何に依存しているか」がコードから読み取りにくくなるという批判もある。コンストラクタを見ただけでは依存関係がわからず、内部で`getIt<T>()`を呼んでいることを知らないと依存グラフが把握できない。

GetItの立場は「コンストラクタ注入より取り回しが良い」という実用主義だ。Flutterの場合、WidgetのコンストラクタはFlutterフレームワークに管理されているため、複雑なコンストラクタ注入を実現するのが難しい事情もある。

## GetItの登録タイプを理解する

GetItには3種類の登録タイプがある。これを理解することがDI設計の核心だ。

**Singleton**：アプリのライフサイクル全体で一度だけインスタンスが生成され、同じインスタンスが返される。DBのコネクションやAPIクライアントなど、状態を持ち共有されるべきオブジェクトに使う。

**Factory**：取り出すたびに新しいインスタンスが生成される。ViewModel（画面ごとに独立した状態を持つべきオブジェクト）に使うことが多い。

**LazySingleton**：Singletonと同じだが、最初にアクセスされた時点で初めてインスタンスが生成される。起動時間の最適化に有効だ。

この3つの使い分けが、アプリの状態管理の健全性を左右する。「これはSingletonか、Factoryか」を意識的に判断することがDI設計の出発点だ。

## injectableによるコード生成

手でGetItへの登録コードを書くのは、アプリが大きくなると管理が大変になる。**injectable**はアノテーションからGetItの登録コードを自動生成するライブラリだ。

```dart
// アノテーションで登録タイプを宣言
@lazySingleton
class UserRepositoryImpl implements UserRepository {
  final ApiClient api;
  
  UserRepositoryImpl(this.api);
}

@injectable
class UserViewModel {
  final UserRepository repository;
  
  UserViewModel(this.repository);
}
```

`flutter pub run build_runner build`を実行すると、以下のような登録コードが自動生成される。

```dart
// 自動生成されるコード（例）
extension GetItInjectableX on GetIt {
  GetIt init() {
    gh.lazySingleton<UserRepository>(() => UserRepositoryImpl(get<ApiClient>()));
    gh.factory<UserViewModel>(() => UserViewModel(get<UserRepository>()));
    return this;
  }
}
```

開発者はアノテーションを付けるだけで、依存グラフの解決はコード生成が担う。依存の順序（`UserRepositoryImpl`より先に`ApiClient`を登録する必要がある等）も自動的に解決される。

## 環境別の依存切り替え

injectableの強力な機能のひとつが**環境（Environment）**による実装の切り替えだ。

```dart
@LazySingleton(as: UserRepository, env: [Environment.prod])
class UserRepositoryImpl implements UserRepository { ... }

@LazySingleton(as: UserRepository, env: [Environment.test])
class MockUserRepository implements UserRepository { ... }
```

テスト時は`MockUserRepository`、本番時は`UserRepositoryImpl`が自動的に注入される。FlavorやFlagsと組み合わせることで、ステージング環境ではモックAPIを使う設計も実現できる。

## GetItとProvider/Riverpodの違い

FlutterにはProviderやRiverpodという別のDIアプローチもある。これらとGetItの違いを整理しておく。

ProviderやRiverpodは**Widgetツリーへの統合**を前提とした設計だ。依存関係がWidgetツリーを通じて流れ、BuildContextを使ってアクセスする。これはFlutterの「すべてはWidget」という哲学と調和している。リビルドの最適化もフレームワーク側が担ってくれる。

GetItはWidgetツリーに依存しないサービスロケーターだ。ビジネスロジック層（UseCase、Repository）の依存関係管理に向いており、ViewModelやUIの状態管理にはRiverpodを使い、サービス層の依存解決にGetItを使うという**ハイブリッド設計**を採用するチームも多い。

## 設計上のトレードオフ

DIを導入することのコストも正直に見ておく必要がある。

**ボイラープレートと複雑性**：injectableを使っても、アノテーションの追加やコード生成の実行ステップが増える。小規模なアプリでは過剰な複雑性になりうる。

**魔法感**：アノテーションとコード生成は「何がどう動いているか」を隠蔽する。生成されたコードを読めばわかるが、初学者には依存関係の解決が「見えにくい」と感じられることがある。

こうしたコストと、テスタビリティ・保守性の向上というメリットをバランスさせて、チームの規模と段階に応じて導入を判断することが大切だ。

## まとめ

GetItとinjectableは、Flutterアプリの「依存関係の管理」に実用的なソリューションを提供する。GetItのサービスロケーターパターンはシンプルで取り回しがよく、injectableのコード生成は大規模アプリでの管理コストを削減する。

しかし本質は技術選択ではなく、「依存関係の生成と利用を分離する」という設計思想にある。どのライブラリを使うかよりも、「このクラスは何に依存すべきか」「この依存はSingletonかFactoryか」という問いに答えることが、健全なアーキテクチャの基盤を作る。
