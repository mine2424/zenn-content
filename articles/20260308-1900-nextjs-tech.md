## APIルートなしにデータを変更できる

Next.js App Routerに導入された**Server Actions**は、フォームの送信やデータの変更を扱う方法を根本から変えた。

従来のパターンでは、フォームを送信するとき、クライアントサイドのJavaScriptが`fetch`を使って`/api/xxx`のAPIエンドポイントにリクエストを送り、サーバー側でデータを処理して返すという流れが一般的だった。

Server Actionsはこの中間層を取り除く。サーバー上で実行される関数を直接フォームの`action`として指定できる。クライアントとサーバーの境界を、コード上で明示することなく自然に越えられる。

---

## Server Actionsとは何か

Server Actionsは、サーバー上でのみ実行される非同期関数だ。`"use server"`ディレクティブを付けることで定義する。

```typescript
// app/actions.ts
'use server'

import { db } from '@/lib/db';
import { revalidatePath } from 'next/cache';

export async function createPost(formData: FormData) {
  const title = formData.get('title') as string;
  const content = formData.get('content') as string;

  if (!title || title.trim().length === 0) {
    return { error: 'タイトルは必須です' };
  }

  await db.post.create({
    data: { title, content }
  });

  revalidatePath('/posts');
}
```

このアクションをフォームの`action`に渡す。

```tsx
// app/posts/new/page.tsx
import { createPost } from '../actions';

export default function NewPostPage() {
  return (
    <form action={createPost}>
      <input name="title" placeholder="タイトル" />
      <textarea name="content" placeholder="本文" />
      <button type="submit">投稿する</button>
    </form>
  );
}
```

APIルートを書く必要がない。`createPost`はサーバーで実行され、DBを直接操作し、キャッシュを無効化する。クライアントサイドのコードにサーバーのロジックが漏れない。

## なぜServer Actionsが生まれたのか

この設計の背景には、「フォームはもともとサーバーと対話するためのものだった」という原点回帰がある。

Webの初期、`<form>`要素はそれ自体でHTTPリクエストを生成し、サーバーにデータを送るための仕組みだった。JavaScriptがなくても動作した。フロントエンドの複雑化とともに、この単純な仕組みはSPA + APIというアーキテクチャに置き換えられていった。

Server Actionsはその振り子を戻す。JavaScriptが無効な環境でも基本的なフォーム送信が動作する（Progressive Enhancement）。同時に、JavaScriptが有効な環境では`useActionState`などのReact Hooksと組み合わせてリッチなUXを実現できる。

## フォームの状態とフィードバックを扱う

`useActionState`（旧`useFormState`）を使うと、アクションの結果をUIに反映できる。

```typescript
// app/actions.ts
'use server'

export type FormState = {
  error?: string;
  success?: boolean;
};

export async function createPost(
  prevState: FormState,
  formData: FormData
): Promise<FormState> {
  const title = formData.get('title') as string;

  if (!title || title.trim().length === 0) {
    return { error: 'タイトルは必須です' };
  }

  // DBへの書き込み処理
  await db.post.create({ data: { title } });

  return { success: true };
}
```

```tsx
// app/posts/new/page.tsx
'use client'

import { useActionState } from 'react';
import { createPost, type FormState } from '../actions';

const initialState: FormState = {};

export default function NewPostForm() {
  const [state, action, isPending] = useActionState(createPost, initialState);

  return (
    <form action={action}>
      <input name="title" placeholder="タイトル" />
      {state.error && <p className="text-red-500">{state.error}</p>}
      {state.success && <p className="text-green-500">投稿しました</p>}
      <button type="submit" disabled={isPending}>
        {isPending ? '送信中...' : '投稿する'}
      </button>
    </form>
  );
}
```

`isPending`でローディング状態を扱え、`state.error`でバリデーションエラーをUIに表示できる。送信中はボタンをdisabledにする。これだけで基本的なUXが揃う。

## バリデーションをどこで行うか

Server Actionsにおけるバリデーションの設計には選択肢がある。

**サーバーサイドのみのバリデーション。** アクション内でZodなどを使ってバリデーションし、エラーをstateとして返す。シンプルで、クライアントサイドのコードが増えない。UXとしては、送信してから初めてエラーを知ることになる。

**クライアント + サーバーの二重バリデーション。** React Hook Formを使ってクライアントサイドでもリアルタイムにバリデーションし、送信時にもサーバーサイドで検証する。ユーザー体験は向上するが、スキーマを共有しなければバリデーションルールが二重管理になる。

実践的には、Zodスキーマを共有ファイルに定義して、クライアントとサーバーの両方で参照するアプローチが多い。クライアントでは`react-hook-form`の`zodResolver`を使い、サーバーアクションでも同じスキーマで`schema.safeParse()`する。

## 認証とアクセス制御

Server Actionsはサーバー上で実行されるが、**自動的に保護されるわけではない**。

クライアントからJavaScriptを通じて呼び出されるため、誰が呼び出しているのかをアクション内で検証する必要がある。

```typescript
'use server'

import { auth } from '@/lib/auth';

export async function deletePost(postId: string) {
  // セッションを検証
  const session = await auth();
  if (!session?.user) {
    throw new Error('認証が必要です');
  }

  // 所有者チェック
  const post = await db.post.findUnique({ where: { id: postId } });
  if (post?.authorId !== session.user.id) {
    throw new Error('この操作を行う権限がありません');
  }

  await db.post.delete({ where: { id: postId } });
}
```

「サーバーアクションだから安全」という思い込みは危険だ。入力の検証、認証の確認、権限のチェックは、すべてのデータ変更アクションで実施する。これはAPIルートのときと変わらない原則だ。

## APIルートとの使い分け

Server Actionsがすべての場合に最適なわけではない。APIルート（Route Handlers）と使い分ける判断軸がある。

**Server Actionsに向いているケース：**
- フォームの送信（ページ内のデータ変更）
- ボタンクリックによるデータ変更（いいね、削除など）
- Next.jsのApp Router内で完結する操作

**APIルートに向いているケース：**
- モバイルアプリや外部サービスからのリクエストを受け付ける
- WebhookのエンドポイントとしてGET/POSTを明示的に定義したい
- REST APIとして他チームと仕様を共有する
- ファイルアップロードや特殊なHTTPメソッドを扱う

Server ActionsはNext.jsのエコシステム内で完結するフロントエンドとバックエンドの統合に強い。外部に公開するAPIが必要な場合は、引き続きRoute Handlersを使う。

## revalidateとリダイレクト

アクション実行後、Next.jsのキャッシュをどう扱うかも設計のポイントだ。

`revalidatePath('/posts')`で特定のパスのキャッシュを無効化し、次回のアクセス時に最新データを取得させる。`revalidateTag('posts')`でfetchにタグを付けてタグ単位で無効化することもできる。

作成・更新後にリダイレクトしたい場合は`redirect()`を使う。ただし`redirect()`は内部的に例外を投げるため、try-catchで包むと意図せず捕捉してしまう点に注意が必要だ。

---

## まとめ

Server Actionsは「フォームはサーバーと対話するためのもの」というWebの原点を、現代的な開発体験で再解釈した機能だ。

APIルートを書かずにサーバーサイドのロジックを直接呼び出せる。Progressive Enhancementによりアクセシビリティが向上し、`useActionState`でリッチなフィードバックUIを実現できる。しかしその利便性に甘えず、認証・バリデーション・権限チェックをきちんと実装することが前提だ。

フォームとデータ変更というWebの基本操作を、Server Actionsはより自然で安全な形で設計できるようにしてくれる。
