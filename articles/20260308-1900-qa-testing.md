## テストコードはなぜ壊れやすいのか

E2Eテストを書いていると、同じ問題に繰り返しぶつかる。UIが少し変わるたびにテストが壊れる。セレクター（`div.button-primary > span`）が変更されると、そのセレクターを使っている全テストケースを修正して回らなければならない。

テストが壊れるのはUIが変わったからだけではない。テストコード自体の設計が「変化に弱い」構造になっているからだ。

Playwrightで用意されている**ページオブジェクトパターン（Page Object Model、POM）**は、この問題に対するひとつの答えだ。

---

## ページオブジェクトパターンの本質

ページオブジェクトパターンの考え方はシンプルだ。

**「UIの実装詳細（セレクター）をテストケースから分離する」**

テストケースは「何をテストしたいか（意図）」を記述する。UIのどのボタンをクリックするか、どのフォームに入力するか——そういった「どうやって操作するか」の詳細は、ページオブジェクトに委ねる。

```typescript
// ページオブジェクト：LoginPageクラス
export class LoginPage {
  constructor(private page: Page) {}

  async login(email: string, password: string) {
    await this.page.getByLabel('メールアドレス').fill(email);
    await this.page.getByLabel('パスワード').fill(password);
    await this.page.getByRole('button', { name: 'ログイン' }).click();
  }

  async getErrorMessage() {
    return this.page.getByRole('alert').textContent();
  }
}

// テストケース：意図が読める
test('無効な認証情報でログインに失敗する', async ({ page }) => {
  const loginPage = new LoginPage(page);
  await loginPage.login('invalid@example.com', 'wrongpassword');
  expect(await loginPage.getErrorMessage()).toBe('メールアドレスまたはパスワードが正しくありません');
});
```

テストケースを読めば「何をテストしているか」が一目でわかる。セレクターの詳細は見えない。`LoginPage`クラスの内部にあるからだ。

## セレクターの設計：ロールと意味を使う

ページオブジェクトを作る際、セレクターをどう書くかは重要な設計判断だ。

Playwrightが推奨するのは、**ユーザーが実際に知覚する属性でセレクターを書く**というアプローチだ。

```typescript
// 避けるべき：実装依存のセレクター
await page.locator('.btn-primary.login-button');  // CSSクラス
await page.locator('#submit-button');               // id
await page.locator('div > form > button:last-child'); // DOM構造

// 推奨：役割と名前に基づくセレクター
await page.getByRole('button', { name: 'ログイン' });  // WAI-ARIAロール
await page.getByLabel('メールアドレス');               // フォームラベル
await page.getByText('利用規約に同意する');            // テキスト
await page.getByTestId('submit-button');              // テスト用data属性
```

`getByRole`や`getByLabel`は、アクセシビリティツリーを通じてUIを操作する。これにはふたつの利点がある。ひとつは、CSSクラスやDOM構造が変わってもテストが壊れにくくなること。もうひとつは、アクセシビリティ的に正しいマークアップを自然に促進すること——ロールやラベルが正しく設定されていないとテストが書けないからだ。

## フィクスチャで初期化を整理する

ページオブジェクトをテストファイルで毎回インスタンス化するのは煩雑だ。Playwrightの**テストフィクスチャ**を使うと、ページオブジェクトの提供を共通化できる。

```typescript
// fixtures.ts
import { test as base } from '@playwright/test';
import { LoginPage } from './pages/LoginPage';
import { DashboardPage } from './pages/DashboardPage';

type Fixtures = {
  loginPage: LoginPage;
  dashboardPage: DashboardPage;
};

export const test = base.extend<Fixtures>({
  loginPage: async ({ page }, use) => {
    await use(new LoginPage(page));
  },
  dashboardPage: async ({ page }, use) => {
    await use(new DashboardPage(page));
  },
});

export { expect } from '@playwright/test';
```

```typescript
// テストファイルはシンプルになる
import { test, expect } from './fixtures';

test('ログイン後にダッシュボードが表示される', async ({ loginPage, dashboardPage }) => {
  await loginPage.login('user@example.com', 'password');
  expect(await dashboardPage.getTitle()).toBe('ダッシュボード');
});
```

フィクスチャによって、「どのページオブジェクトを使うか」というセットアップの詳細がテストケースから消える。

## ページオブジェクトの粒度をどう決めるか

「1ページに1クラス」という原則は出発点としてよいが、必ずしも正解ではない。

**コンポーネント単位のオブジェクト。** ナビゲーションバーやフッター、モーダルダイアログのように複数ページに横断的に登場するUIは、それぞれ独立したオブジェクトとして定義するのが合理的だ。

```typescript
export class NavigationBar {
  constructor(private page: Page) {}

  async navigateTo(section: '商品' | '注文' | '設定') {
    await this.page.getByRole('navigation').getByRole('link', { name: section }).click();
  }

  async getUserName() {
    return this.page.getByTestId('user-name').textContent();
  }
}
```

**フロー単位のオブジェクト。** 複数ページにまたがるユーザーフロー（購入フローなど）は、ページ単位ではなくフロー全体を表現するオブジェクトで包む考え方もある。

適切な粒度の判断軸は「変更の局所性」だ。UIのある部分が変わったとき、修正が1クラスに収まるか。変更が複数クラスに散らばるようであれば、設計を見直す余地がある。

## テストの「意図」を守るということ

ページオブジェクトパターンの本当の価値は、テストコードを「読みやすくする」だけではない。

テストケースが「何をテストしたいか」を明示的に示すことで、**テストの意図がチームに伝わる**という効果がある。

セレクターまみれのテストコードは、「ボタンをクリックして入力して送信する」という手順は読めても、「これは認証失敗のシナリオをテストしている」という意図が見えにくい。新しいチームメンバーがそのテストを読んでも、何を守るために書かれたテストなのかわからない。

ページオブジェクトを使うことで、テストケースはビジネスロジックの言葉で書ける。「ログインする」「エラーメッセージを確認する」「ダッシュボードへ遷移したことを確認する」——これは仕様書の言葉でもある。テストコードが仕様の表現になる。

---

## 過剰なページオブジェクトに注意する

ページオブジェクトパターンを使いすぎると、かえって複雑になる場合もある。

すべての操作をメソッドで包み込み、ページオブジェクトが巨大化すると、「どのメソッドで何ができるのか」を把握するのが難しくなる。

適切な判断は「繰り返し使われる操作か」というシンプルな基準だ。1テストケースにしか登場しない一度きりの操作は、ページオブジェクトに切り出す必要はない。複数のテストケースで共通する操作こそ、抽象化の候補だ。

---

## まとめ

Playwrightのページオブジェクトパターンが解決する問題は、「テストの意図とUIの実装を分離する」という一点に集約される。

セレクターをテストケースから追い出し、ページオブジェクトに閉じ込める。テストケースはビジネスロジックの言語で意図を表現する。UIが変わったときの修正はページオブジェクトの中で完結する。

テストコードも「設計する」ものだ。プロダクションコードと同じように、変化に強く、読みやすく、意図が伝わる構造を意識することが、長く維持できるE2Eテストスイートへの道になる。
