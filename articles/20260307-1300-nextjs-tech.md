## はじめに

Next.jsとPrismaの組み合わせはここ数年で標準的な構成になりつつある。しかし「とりあえずPrismaを使っている」という状態のまま、パフォーマンスの問題やスキーマ設計の混乱に陥っているプロジェクトも少なくない。

Prismaは高機能なORMだが、その設計思想を理解せずに使うと、本来の価値を引き出せないどころか、意図せず問題を埋め込むことになる。この記事では「Prismaとは何を解決するツールか」という設計思想の整理から始め、Next.jsとの組み合わせで特に意識すべき点を掘り下げる。

---

## PrismaはコードファーストORMではなく「スキーマファーストORM」だ

まず最初に押さえるべき特徴として、Prismaはスキーマファーストのアプローチを取る点がある。

TypeORMやSequelizeはコードにアノテーションを書いてスキーマを定義する。対してPrismaは`schema.prisma`という専用の定義ファイルを中心に設計が展開する。

```prisma
model User {
  id        String   @id @default(cuid())
  email     String   @unique
  name      String?
  posts     Post[]
  createdAt DateTime @default(now())
}
```

この`schema.prisma`から、TypeScriptの型定義・DBマイグレーション・Prisma Clientの3つが生成される。これが「Single Source of Truth（唯一の正解）」として機能するのがPrismaの設計思想の核心だ。

スキーマを変更したら`prisma generate`でクライアントを再生成し、`prisma migrate dev`でマイグレーションファイルを作る——このサイクルが開発フローに組み込まれることで、「DBのスキーマ」と「アプリのコード」が常に同期する。

## N+1問題はPrismaでも自然に発生する

ORMを使う最大のリスクの一つがN+1問題だ。Prismaも例外ではなく、関係性（リレーション）を扱う際に意識が必要になる。

例えば「全投稿と、各投稿のAuthorを取得する」ケースで、以下のように書くと問題が起きる。

```typescript
const posts = await prisma.post.findMany();
// 各postに対してユーザーを別途取得すると N+1 になる
for (const post of posts) {
  const author = await prisma.user.findUnique({ where: { id: post.authorId } });
}
```

Prismaの解決策は`include`（JOIN相当）と`select`（必要なフィールドのみ取得）だ。

```typescript
const posts = await prisma.post.findMany({
  include: {
    author: {
      select: { id: true, name: true }
    }
  }
});
```

ここで`select`の活用が重要だ。`include`だけだとauthorオブジェクト全体が取得されるが、`select`で必要なフィールドのみを指定することで転送データ量を削減できる。

設計の基本として「APIレスポンスとして返すフィールドのみをselectで指定する」という習慣を持つだけで、クエリ効率は大幅に改善する。

## Next.jsのサーバーサイドとの統合設計

Next.js App RouterとPrismaを組み合わせるとき、最も気をつけるべきはPrismaClientのインスタンス管理だ。

開発環境ではHot Reloadのたびにモジュールが再評価される。その度に新しいPrismaClientのインスタンスが生成されると、DB接続数が上限を超えて接続エラーが発生する。

これを防ぐための標準パターンがグローバルシングルトンだ。

```typescript
// lib/prisma.ts
import { PrismaClient } from "@prisma/client";

const globalForPrisma = globalThis as unknown as {
  prisma: PrismaClient | undefined;
};

export const prisma =
  globalForPrisma.prisma ?? new PrismaClient();

if (process.env.NODE_ENV !== "production") {
  globalForPrisma.prisma = prisma;
}
```

本番環境ではモジュールは一度しか評価されないため問題は起きないが、開発環境でこのパターンを使わないと再現性の低い接続エラーに悩むことになる。Prismaの公式ドキュメントでも推奨されているが、Next.jsプロジェクトで忘れられがちな重要な設定だ。

## スキーマ設計：マイグレーションを意識した設計

Prismaのスキーマ設計で見落とされがちなのが「後からの変更コスト」への意識だ。

例えば`User`モデルに`role`フィールドを追加する際、`String`型で自由に値を入れるより`enum`を定義するほうが型安全だ。しかしenumに値を追加するマイグレーションは、DBによっては（PostgreSQLでは）一度enumの再作成が必要になる。本番DBでのダウンタイムを避けるために、enumの設計変更は慎重に行う必要がある。

一方で「後から追加するフィールドは基本的にOptional（nullable）にする」という原則は、デプロイのリスクを下げる。新しいフィールドを`NOT NULL`かつデフォルト値なしで追加するマイグレーションは、既存データがある場合にエラーになる。段階的な移行（まずnullableで追加→データを埋める→NOT NULLに変更）という設計が安全だ。

## トランザクションと整合性

Prismaでは`prisma.$transaction`を使ってトランザクションを明示的に扱える。例えば「注文を作成して在庫を減らす」という処理は、どちらかが失敗した場合に両方をロールバックする必要がある。

```typescript
await prisma.$transaction(async (tx) => {
  await tx.order.create({ data: orderData });
  await tx.inventory.update({
    where: { productId: orderData.productId },
    data: { quantity: { decrement: orderData.quantity } }
  });
});
```

`$transaction`にはインタラクティブトランザクション（上記のようなasync関数を渡す形式）とバッチトランザクション（配列で複数クエリをまとめる形式）の2種類がある。インタラクティブトランザクションはDB接続を保持したままになるため、長時間の処理（外部API呼び出しなど）を中に含めるのは避けるべきだ。

## Prisma Accelerateとコネクションプーリング

サーバーレス環境（Vercel EdgeやAWS Lambda）でPrismaを使う場合、コネクションプーリングの設計が必須になる。サーバーレス関数はリクエストごとに独立して起動するため、従来のコネクションプーリングが機能しない。

Prismaが提供する**Prisma Accelerate**はこの問題を解決するグローバルなキャッシュ・コネクションプーリングサービスだ。Accelerateを経由することで、サーバーレス環境でもDBへの接続数を制御でき、クエリキャッシュによるレスポンス高速化も実現できる。

セルフホストの場合は**PgBouncer**（PostgreSQL向け）などのコネクションプーラーをPrismaの前段に置く設計が一般的だ。

---

## まとめ

Prismaの価値は「型安全にDBを操作できること」にあるが、それはスキーマファーストの思想を守り、N+1を意識したクエリ設計をして、接続管理を正しく行うことで初めて発揮される。

「便利だから使う」ではなく「どういう制約と思想を持ったツールか」を理解することが、Prismaを長く安心して使い続けるための前提だ。スキーマはコードとDBの橋渡しであり、その設計の質がアプリケーション全体の堅牢性を左右する。
