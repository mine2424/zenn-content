## はじめに

Webフォントはサイトのビジュアル品質を高める重要な要素だ。しかし実装を誤ると、パフォーマンスとユーザー体験を著しく損なう。フォントの読み込みが遅れてテキストが後から描画されるFOUT（Flash of Unstyled Text）や、フォントの読み込み完了時にレイアウトがずれるCLS（Cumulative Layout Shift）は、Core Web Vitalsのスコアを下げる主要因のひとつだ。

Next.js 13で導入された`next/font`は、これらの問題を「デフォルトで解決する」という設計思想で作られている。本記事では、`next/font`が何を解決し、どのような設計判断の上に成り立っているかを考えていく。

## Webフォントの問題を正確に理解する

`next/font`の設計思想を理解するには、まずWebフォントがなぜ問題を引き起こすかを理解する必要がある。

**パフォーマンス上の問題**として、Googleフォントなどの外部フォントサービスを利用する場合、ブラウザはそのドメインへのDNS解決、TCP接続、HTTPS認証を行う追加のネットワークラウンドトリップが必要になる。これはレンダリングのブロッキング要因になりうる。

**CLS（レイアウトシフト）の問題**は、フォントが読み込まれる前はシステムフォント（フォールバックフォント）でテキストが描画され、フォントが読み込まれると表示が切り替わるときに発生する。問題の本質は、ウェブフォントとシステムフォントは通常、字形のサイズや文字間隔が異なるため、フォントの切り替わりでレイアウトが変化してしまうことだ。

**プライバシー上の問題**もある。外部フォントサービスを利用すると、ユーザーのIPアドレスなどの情報が外部サービスに送信される。GDPRなどの観点からこれを問題視する見方もある。

## next/fontが採用した設計思想

`next/font`はこれらの問題を、次の2つの核心的な設計で解決している。

**1. フォントのセルフホスティング**

`next/font`はGoogleフォントを使う際も、ビルド時にフォントファイルをダウンロードし、自分のドメインから配信する形に変換する。これにより、外部ドメインへの接続が不要になり、パフォーマンスとプライバシーの両問題を解決する。

開発者がGoogleフォントのCDN URLを`<link>`タグで直接参照するアプローチとの決定的な違いは、「実行時にユーザーのブラウザがフォントを取得する場所」が変わることだ。

**2. 自動的なsize-adjustによるCLS防止**

`next/font`の最も巧妙な設計は、フォールバックフォントに`size-adjust`CSSプロパティを自動適用して、Webフォントとシステムフォントの見た目を可能な限り一致させる点だ。

`size-adjust`は比較的新しいCSSプロパティで、フォントのグリフのサイズを調整するために使われる。`next/font`はビルド時にWebフォントとシステムフォントのメトリクスを比較し、`size-adjust`やascent-override、descent-override、line-gap-overrideといったCSSフォントメトリクスオーバーライドを計算して、フォールバックフォントに適用する。

これにより、フォントが読み込まれる前後でテキストの占める空間がほぼ同じになり、CLSがゼロ（あるいは限りなくゼロに近い）状態を実現できる。

## 実際の使い方と設計パターン

```typescript
// app/fonts.ts
import { Inter, Noto_Sans_JP } from 'next/font/google'

export const inter = Inter({
  subsets: ['latin'],
  display: 'swap',
  variable: '--font-inter',
})

export const notoSansJP = Noto_Sans_JP({
  subsets: ['latin'],
  weight: ['400', '700'],
  display: 'swap',
  variable: '--font-noto-sans-jp',
})
```

フォントをCSS変数として定義し、Tailwind CSSや`font-family`でアクセスするパターンが現代的な使い方だ。`variable`オプションでCSS変数名を指定することで、グローバルなフォント設計をCSS変数を通じて管理できる。

`display: 'swap'`は`font-display`プロパティの設定だ。フォントの読み込みが完了する前はフォールバックフォントを使い、読み込みが完了したらWebフォントに切り替えるという挙動を指定する。`next/font`の`size-adjust`計算はこの切り替え時のシフトを最小化するためにある。

## preload設計——何を事前にロードするか

`next/font`はデフォルトで使用されているフォントの最初のページで`preload: true`が適用される。これにより、`<link rel="preload">`タグが自動的に挿入され、フォントファイルの取得を優先させる。

しかし全ページでフォントをpreloadするとリソース消費が増える。`preload: false`を明示的に設定して、重要なページのみpreloadする設計も選択肢だ。「どのフォントをいつpreloadするか」という判断は、そのフォントがファーストビューで使われるかどうかに基づくべきだ。

## ローカルフォントの扱い

カスタムフォントファイル（`.woff2`など）を使う場合は`next/font/local`を使う。Googleフォント向けの`next/font/google`と同様のCLS防止機構が適用される。

デザインシステムでブランドフォントを持っているプロダクトでは、ライセンスを持つカスタムフォントをセルフホスティングすることになる。`next/font/local`を使うことで、CDNキャッシュ戦略を自分でコントロールしつつ、自動的なパフォーマンス最適化の恩恵を受けられる。

## 日本語フォントの設計上の注意

日本語フォントはラテン文字と比べてファイルサイズが格段に大きい。日本語には数千の漢字が含まれるため、フォントファイルが数MBになることも珍しくない。

この問題への対処として**サブセット化**が重要になる。`subsets`オプションで使用する文字セットを絞り込むことが第一歩だ。ただし日本語の場合、`subsets`に`'japanese'`を指定しても完全な日本語文字セットを含むため依然として大きい。

実運用では、Google Fontsが提供するUnicode-rangeによる**フォントのスライス（範囲分割）**が有効だ。ブラウザはページに実際に使われているグリフのみを含むスライスだけをダウンロードする仕組みで、ほとんどのページでは全文字の一部しか使わないため、実効的なダウンロードサイズを大幅に削減できる。

## まとめ

`next/font`は「Webフォントの最適化を手動でやる必要がない」という強いオピニオンを持った設計だ。フォントのセルフホスティングと、自動的なCSSメトリクス調整によるCLS防止が、デフォルトで適用される。

フォントはパフォーマンスのコストが見えにくいが、CLSやLCP（Largest Contentful Paint）を通じてユーザー体験に直接影響する。`next/font`の設計思想を理解した上で適切に設定することで、見た目の品質とパフォーマンスの両立が実現できる。「見た目のための装飾」ではなく「体験のための設計」として、フォント最適化に向き合いたい。
