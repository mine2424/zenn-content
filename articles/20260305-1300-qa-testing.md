## 「Googleのスコアを上げる」ではない

Core Web Vitalsという言葉を聞いたとき、「SEOのためにスコアを上げる作業」として捉えてしまうことがある。しかしそれは本質からずれている。Core Web VitalsはGoogleが提唱した指標体系だが、その背後にある問いは「ページがユーザーにとって実際にどれだけ使いやすいか」だ。

指標を追いかけることと、ユーザー体験を改善することは本来同じ方向を向いている。ただし、指標の意味を正確に理解せずにスコアだけを改善しようとすると、数字は動いても体験は変わらないという状況が生まれる。本記事では、Core Web Vitalsの各指標が「何を測っているのか」を起点に、QAプロセスへの組み込み方と改善サイクルの設計を考える。

## 3つの指標が測っているもの

Core Web Vitalsは現在3つの主要指標で構成される（INPは2024年にCLSと並ぶ主要指標として昇格）。

### LCP（Largest Contentful Paint）——ページの主要コンテンツはいつ表示されるか

LCPは、ビューポート内で最も大きな要素（画像、テキストブロック、動画サムネイルなど）が描画されるまでの時間だ。「ページがロードされた」という感覚と最も相関が高い指標とされる。

良好の基準は2.5秒以内。ただし「2.5秒以内に収める」という目標設定よりも、「ユーザーが最初に目にしたい情報が遅延なく表示されているか」という問いで考えるべきだ。LCPの要素は自動で決まるが、意図的に「この要素がLCPであるべき」と設計できる。ヒーロー画像や主要な見出しを優先して読み込む設計がそれにあたる。

### INP（Interaction to Next Paint）——操作への応答はどれだけ速いか

INPは2024年3月にFID（First Input Delay）に代わって導入された指標だ。ページ上のすべてのインタラクション（クリック、タップ、キーボード入力）の応答時間を計測し、最悪値に近い値を代表スコアとして使う。

FIDが「最初のインタラクション」しか測らなかったのに対し、INPは「ページを使い続ける中でのすべてのインタラクション」を測る。これはSPA（シングルページアプリケーション）での体験をより正確に反映する。フォーム入力、アコーディオンの開閉、フィルタリング操作——こうした繰り返しの操作がもたつくとINPが悪化する。

良好の基準は200ms以内。メインスレッドをブロックする長いJavaScript処理がINP悪化の主な原因だ。

### CLS（Cumulative Layout Shift）——コンテンツが読み込み中にずれないか

CLSは、ページロード中に発生するレイアウトのずれを累積スコアで表す。画像や広告が遅れて読み込まれ、それによってコンテンツが押し下げられる現象がCLSを悪化させる。

「読もうとしていたテキストが突然ずれてタップミスした」という体験はCLSが高い状態だ。良好の基準は0.1以下。原因の多くは明示的なサイズ指定のない画像・動画・広告枠だ。

## LighthouseとField Dataの違いを理解する

Core Web Vitalsの計測には、大きく「ラボデータ」と「フィールドデータ」の2種類がある。

**ラボデータ（Lighthouse、WebPageTest）**: 制御された環境でページを読み込んだときの計測値。再現性が高く、開発中のデバッグに向く。ただし「実際のユーザーの計測値」ではない。

**フィールドデータ（CrUX、RUM）**: 実際のユーザーのブラウザで計測されたデータ。Google Search ConsoleのCore Web Vitalsレポートや、Web Vitals JSライブラリを使ったRUM（Real User Monitoring）がこれにあたる。

QAプロセスでよくある失敗は、「Lighthouseでは良好なスコアが出ているのに、Search Consoleでは要改善と表示される」という乖離だ。Lighthouseはネットワーク帯域を絞った模擬環境で動くが、実際のユーザーはさまざまな端末とネットワーク環境を使う。フィールドデータの改善こそが本来の目標だと押さえておく必要がある。

## QAプロセスへの組み込み方

Core Web Vitalsを「リリース前に一度確認する」だけの作業にしてしまうと、改善が属人的になりやすい。継続的なQAプロセスとして組み込むための考え方を整理する。

### CI/CDでのLighthouse計測

Lighthouseは`lighthouse-ci`（LHCI）を使ってCI/CDパイプラインに組み込める。プルリクエスト単位でスコアを計測し、閾値を下回った場合にCIを失敗させる設定が可能だ。

```bash
# .lighthouserc.js での閾値設定イメージ
assert: {
  assertions: {
    'categories:performance': ['error', { minScore: 0.8 }],
    'first-contentful-paint': ['warn', { maxNumericValue: 2000 }],
    'largest-contentful-paint': ['error', { maxNumericValue: 2500 }],
  }
}
```

ただしCIでの計測はラボデータだという点は意識する。「回帰を検知する」用途として使い、絶対値の評価はフィールドデータで行うという役割分担が適切だ。

### Performance Budgetの設定

「前回リリースより5%以上スコアが下がったらアラート」というPerformance Budgetの概念が、継続的改善に役立つ。絶対的な目標値よりも「悪化を早期に検知する」仕組みが長期的には重要だ。

### RUMによるフィールドデータ収集

`web-vitals` ライブラリを使えば、実ユーザーのCore Web Vitals計測値をアナリティクスに送れる。デバイス種別（モバイル/デスクトップ）やページ種別ごとに集計することで、どのページのどのユーザーセグメントで問題が起きているかが見える。

## 改善の優先順位をどう決めるか

Core Web Vitalsを改善しようとすると、施策の候補が多くて優先順位に迷いやすい。以下の観点で絞り込むのが実務的だ。

**トラフィックの多いページから始める**: Core Web VitalsはURL単位で評価される。まずトラフィックが集中するページを改善する方がインパクトが大きい。

**原因ごとに施策を対応させる**: LCPが遅い場合、原因はサーバーレスポンスの遅延、レンダーブロッキングリソース、画像の最適化不足など複数あり得る。Lighthouseの「Opportunities」タブが原因を整理するヒントを与えてくれる。

**INP改善はプロファイリングが必要**: INPの改善にはChrome DevToolsのPerformanceパネルでインタラクション中の処理を録画し、どの関数がメインスレッドを長時間占有しているかを特定するプロセスが必要だ。「何かJSを減らす」ではなく「どの処理が原因か」を特定してから動く。

## Next.jsでの特有の注意点

Next.jsを使っている場合、いくつかの最適化は標準機能として提供されている。`next/image`はWebP変換とサイズの自動最適化を行い、CLSの原因になりやすい画像サイズ未指定を防ぐ。`next/font`はFONT-displayとpreloadを最適化してCLSを抑制する。

一方でApp Routerに固有の落とし穴もある。Client ComponentにデフォルトでバンドルされるJavaScriptが増えやすく、INPに影響することがある。「どのコンポーネントをServer Componentに保てるか」というアーキテクチャの判断がINPのパフォーマンスに直結する。

## まとめ

Core Web Vitalsは、スコアを上げるための指標ではなく、ユーザーがページをどう体験しているかを数値で見るための窓だ。LCPは主要コンテンツの表示速度、INPはインタラクションの応答性、CLSはレイアウトの安定性——それぞれが異なる体験の側面を測っている。ラボデータで回帰を検知し、フィールドデータで実際の改善を確認するサイクルをQAプロセスに組み込むことで、指標は継続的な改善のフィードバックループとして機能するようになる。
