## はじめに

Next.js 13以降、開発環境で `--turbo` フラグを付けると Turbopack が使えるようになった。VercelはWebpackの10倍、Viteよりも高速とアナウンスし、大きな注目を集めた。しかし、「速い」という事実の背景にある設計思想はあまり語られない。

Turbopackはなぜ速いのか。何を変えたのか。そして開発者はTurbopackとどう付き合えばよいのか——この記事ではその問いを整理する。

## Webpackの限界はどこにあったか

まず、なぜTurbopackが生まれたかを理解するために、Webpackの限界を振り返る。

Webpackはモジュールバンドラーとして長年Webフロントエンドを支えてきた。しかし、アプリケーションが大規模化するにつれて、開発サーバーの起動時間やHMR（Hot Module Replacement）の応答速度が問題になってきた。

Webpackの中核的な課題は「スコープの広さ」にある。変更を検知すると、依存グラフを辿って影響範囲を計算し、必要なモジュールを再ビルドする。このプロセスはアプリが大きくなるほど遅くなる。

JavaScriptで書かれていることも影響している。Node.jsはシングルスレッドで動作するため、並列処理に限界がある。ファイルのパース・トランスフォーム・バンドルという処理を、CPUをフル活用して並列化することが難しい。

## Turbopackが取った設計の選択

Turbopackは同じ問題を「Rustで書く」「インクリメンタル計算エンジンを使う」という二つのアプローチで解決しようとした。

**Rustによる実装**は処理速度の改善に直結する。RustはJavaScriptと比べてパースや変換処理が高速で、マルチスレッド処理も自然に行える。ファイルのトランスフォームをCPUのコアを使って並列に行えるため、大規模プロジェクトでの恩恵が特に大きい。

**インクリメンタル計算エンジン**はTurbopackの核心だ。これは「Turbo Engine」と呼ばれており、Vercelが以前に開発したTurborepoのアイデアを発展させたものだ。

インクリメンタル計算の本質は、「計算結果をキャッシュし、入力が変わったときだけ再計算する」という原則だ。ファイルが変わったとき、そのファイルに依存する処理だけを再実行する。依存していない処理の結果はキャッシュから取り出す。

Webpackも依存グラフを持っているが、Turbopackのそれは関数レベルで細かくキャッシュされる。つまり、ファイルのパース結果、トランスフォーム結果、バンドル結果がそれぞれ独立してキャッシュされ、変更の影響が最小の範囲に限定される。

```
Webpack: ファイル変更 → 影響モジュール全体を再ビルド
Turbopack: ファイル変更 → 変更されたキャッシュノードだけ再計算
```

## 「オンデマンドバンドル」という発想

Turbopackが開発環境で特に速い理由に、もう一つのアプローチがある——オンデマンドバンドルだ。

Webpackの開発サーバーは起動時にアプリ全体をバンドルしようとする。一方、Turbopackはブラウザからリクエストされたファイルだけをバンドルする。

ブラウザが `http://localhost:3000/` を開くとき、その画面を描画するのに必要なモジュールだけを処理する。画面外のルートや、まだ表示されていないコンポーネントはビルドしない。プロジェクトが大きくても、実際に触っている部分だけが速く応答する。

この発想はViteも同様のアプローチ（ESModulesを使ったオンデマンド処理）を取っているが、Turbopackはそれをネイティブコード＋インクリメンタルキャッシュと組み合わせている点が異なる。

## Next.jsにおける現在地

重要な注意点として、2025年時点では TurbopackはNext.jsの**開発環境**向けとして安定化が進んでいる段階だ。本番ビルド（`next build`）でのTurbopack対応も進んでいるが、Webpackとの完全な機能同等性がすべての環境で保証されているわけではない。

`next dev --turbo` を使うと開発サーバーにTurbopackが使われる。多くの一般的なNext.jsの設定ではそのまま動作するが、カスタムWebpack設定を多用しているプロジェクトでは注意が必要だ。

Turbopackは独自のプラグインAPIを持っておらず、Webpackプラグインをそのまま使うことはできない。Webpackの `config.plugins` に手を加えているプロジェクトでは、Turbopack対応の設定に書き換えるか、対応を待つ必要がある。

## 開発者として何が変わるか

Turbopackが普及すると、開発体験で最も体感できる変化は「HMRの応答速度」だ。ファイルを変更してブラウザに反映されるまでの時間が、プロジェクト規模によらず数十〜数百ミリ秒に収まる。

大規模なモノレポやページ数が多いNext.jsプロジェクトでは、起動時間の差が顕著になる。Webpackで30秒かかっていた起動が数秒になる、というケースも報告されている。

一方、本番ビルドの最終成果物（バンドルサイズ・コード分割）はWebpackとほぼ同等になることを目指している。「速く動くが最適化が弱い」というトレードオフではなく、「速くかつ同等の最適化」を目指しているのがTurbopackの設計方針だ。

## ビルドツールの変化が意味すること

TurbopackのようなRust製ツールの台頭（BunやRspackも同様）は、フロントエンドビルドツールの世代交代を示している。

JavaScript製ツールの限界はパフォーマンスだ。エコシステムの豊かさや設定の柔軟性ではJavaScriptが有利でも、純粋な処理速度ではRustやGoに軍配が上がる。

この流れの中でWebpackが過去のものになるかというと、そう単純ではない。Webpackには膨大なプラグインエコシステムと、実績から来る信頼がある。しかし新規プロジェクトや開発体験を重視するチームが、TurbopackやRspackを選ぶケースは着実に増えている。

## まとめ

- TurbopackはRust実装＋インクリメンタル計算エンジンで速さを実現している
- 「変わったものだけ再計算する」キャッシュ戦略が大規模プロジェクトで特に効く
- オンデマンドバンドルにより、開発時に必要なものだけを処理する
- 2025年現在は開発サーバー（`next dev --turbo`）での利用が主流
- カスタムWebpack設定を持つプロジェクトは移行に注意が必要
- ビルドツールのRust化という大きな流れの中にTurbopackは位置している

Turbopackを「速いWebpack」として捉えるより、「ビルドの概念を再設計したツール」として理解することで、その価値と将来性がより明確に見えてくる。
