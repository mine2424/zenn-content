## テストデータ問題は「テスト自体」の問題より深刻になる

自動テストを運用していると、ある時期から「テストは書けているのに、なぜか不安定で信頼できない」という状況に陥ることがある。その原因の多くは、テストコードそのものではなく、**テストデータの管理**にある。

テストが依存するデータが曖昧だと、以下のような問題が起きる。

- ローカルでは通るがCIで落ちる（環境ごとにデータが違う）
- 先週は通っていたが今週は落ちた（誰かがデータを変えた）
- テストを特定の順番で実行しないといけない（テスト間でデータが干渉している）
- 本番に近いデータでテストしたいが、個人情報が含まれていて使えない

これらはすべて「テストデータ管理の設計」の問題だ。本稿ではその設計思想を整理する。

## テストデータの3つの課題

テストデータには本質的に3つの課題がある。

### 1. 再現性

同じテストを何度実行しても、同じ結果が得られなければ意味がない。しかし共有データベースを複数のテストが使い回すと、あるテストが別のテストのために作ったデータに依存し、実行順序によって結果が変わる。

再現性のないテストは「成功したらラッキー」なギャンブルになる。

### 2. 独立性

テストは互いに干渉してはいけない。テストAがユーザーを作成し、テストBがそのユーザーを削除するような状況は、並列実行を不可能にし、CI/CDのスケールを妨げる。

理想は「各テストが自分のデータだけを持ち、他のテストに影響を与えない」状態だ。

### 3. 現実性

テストデータがあまりにも単純すぎると、本番環境でしか発生しないバグを見逃す。「全フィールドが埋まった顧客データ」と「生年月日が空の顧客データ」では、アプリケーションの挙動が違うことがある。本番データに近い多様性を持つテストデータが必要だが、個人情報はそのまま使えない。

## アプローチ1: テストごとのデータ生成とクリーンアップ

最もシンプルで信頼性の高いアプローチは、**テストごとにデータを生成し、テスト後に削除する**方法だ。

テスト開始時に必要なデータをプログラム的に作成し（フィクスチャやファクトリパターンを使う）、テスト終了後にトランザクションをロールバックするか明示的に削除する。

```typescript
// ファクトリパターンの例
const user = await UserFactory.create({
  email: `test-${Date.now()}@example.com`,
  role: "admin",
});

// テスト実行

await user.delete(); // クリーンアップ
```

この方法のメリットは、テストが自己完結することだ。どのテストも他のテストの状態に依存しない。デメリットは、データ生成・削除のコストがかかることと、ファクトリの管理が必要になることだ。

トランザクションベースのクリーンアップ（テスト全体を1トランザクション内で実行してロールバック）は、削除処理を書かなくて済む分シンプルで、特にデータベースを使った統合テストに有効だ。

## アプローチ2: シード（Seed）データの標準化

全テストが共通して使う「基盤データ」（マスターデータやデフォルトの設定値など）は、シードデータとして管理するのが現実的だ。

重要なのは、シードデータは**読み取り専用として扱う**ことだ。テストはシードデータを参照するが変更しない。変更が必要なテストは、シードデータをコピーして自前のデータを作る。

シードデータのバージョン管理も重要だ。マイグレーションファイルと同様に、シードデータの変更もgitで管理し、変更履歴をレビューできるようにする。「誰かが気づかずシードデータを変更してテストが壊れた」という事故を防げる。

## アプローチ3: テストデータのマスキングと生成

本番に近いデータでテストしたい場合、本番データをそのまま使うのはコンプライアンス上問題がある。そこで使うのが**データマスキング**と**合成データ生成**だ。

**マスキング**は、本番データの構造やバリエーションを保ちながら、個人を特定できる情報を偽のデータに置き換える手法だ。メールアドレスを `user@company.com` → `masked_1234@example.com` に変換するなど。

**合成データ生成**は、本番データの統計的な特性（データの分布、エッジケースの比率など）を模倣した偽データをプログラム的に生成する手法だ。FakerやFactoryBotのようなライブラリがこれを支援する。

現実的なテストデータには、**エッジケース**を意図的に含めることが重要だ。空の配列、最大長の文字列、タイムゾーンをまたぐ日時、特殊文字を含む名前——こうしたケースを手で書くのは漏れやすいため、データ生成戦略に組み込む。

## アプローチ4: データベーススキーマ変更との同期

テストデータが壊れる原因の一つが、スキーマ変更だ。新しいカラムに `NOT NULL` 制約が追加されたとき、既存のテストデータが無効になる。

これを防ぐには、**マイグレーションとフィクスチャを同期させる仕組み**が必要だ。マイグレーション実行後にフィクスチャのバリデーションを自動で走らせるCIステップを設けるか、型付きのファクトリを使ってスキーマ変更がコンパイルエラーとして検出されるようにする。

TypeScriptを使った開発であれば、Prismaのようなスキーマファーストなツールと、型付きのファクトリを組み合わせることで、スキーマ変更を「型エラー」として捕捉できる。

## テスト環境の分離設計

テストデータ管理は、テスト環境の設計とセットで考える必要がある。

- **ユニットテスト**: データベースを使わない。モック・スタブで代替。
- **統合テスト**: テスト用DBを使用。テストごとにトランザクションをロールバック。
- **E2Eテスト**: 本番に近いシードデータを持つ専用環境。テスト後にリセット。

環境ごとに「どのデータ戦略を取るか」を明示することで、チーム全体でテストデータへの共通認識ができる。

## テストデータ管理は品質への投資だ

テストデータの設計は地味で後回しにされがちだが、自動テストが増えれば増えるほどその重要性が増す。データが不安定だと、テストを増やすほど「テストへの信頼」が低下するという逆説が起きる。

「このテスト、本当に信頼できるの？」という疑問が生まれた時点で、QAとしての価値は半減する。再現性と独立性を確保したテストデータ管理は、テストスイート全体の信頼性を土台から支える投資だ。テストコードの品質と同じくらい、データ設計に時間を割く価値がある。
