# WebLLMとWebGPUで「ブラウザの中にAI」を動かす——Edge AIの設計思想

LLMはクラウドで動くもの、という前提が崩れつつある。

WebGPUの登場によって、ブラウザはGPUを直接扱えるようになった。そしてWebLLMのようなプロジェクトが、クラウドAPIを使わずにブラウザ単体でLLMを動かす可能性を現実のものにした。

この記事では、ブラウザ内AI（Edge AI）の仕組みと設計思想を整理する。「なぜそれが可能になったのか」「何に使えて何に向かないのか」——その本質を理解することが目的だ。

---

## WebGPUとは何か

WebGPUは2023年にChrome 113で安定版として登場したブラウザAPIだ。従来のWebGLが主にグラフィックス描画を目的としていたのに対し、WebGPUはGPUを汎用計算（GPGPU）に使うことを正式にサポートしている。

LLMの推論処理は、大量の行列演算から成り立っている。これはGPUが最も得意とする処理形態だ。WebGPUによってブラウザがこの計算を直接扱えるようになったことで、クラウドに頼らないLLM推論が現実的になった。

WebGLとの違いを一言で表すなら、「描画のついでに計算するか、計算のために設計されているか」だ。WebGPUはCompute Shaderというネイティブな汎用計算の仕組みを持っており、LLMの行列積演算に直接マッピングできる。

---

## WebLLMの仕組み

WebLLMはMLC（Machine Learning Compilation）プロジェクトの成果物で、LLMのモデルウェイトをWebGPUで実行できる形式にコンパイルして配布するフレームワークだ。

処理の流れはおおよそ次のようになる：

1. **モデルのダウンロード**: 量子化されたモデルウェイト（数GBから数十GB）をCDNから取得し、IndexedDBにキャッシュする
2. **WebAssemblyランタイム**: MLCのランタイムがWasmとして動き、モデル実行のオーケストレーションを担う
3. **WebGPUで推論**: Compute Shaderを通じてGPUでトークン生成を行う

```javascript
import * as webllm from "@mlc-ai/web-llm";

const engine = await webllm.CreateMLCEngine("Llama-3.1-8B-Instruct-q4f32_1-MLC", {
  initProgressCallback: (progress) => {
    console.log(`ロード中: ${Math.round(progress.progress * 100)}%`);
  }
});

const reply = await engine.chat.completions.create({
  messages: [{ role: "user", content: "こんにちは" }],
});
```

OpenAI互換のAPIを提供しているため、既存のコードをほぼそのまま流用できる点が実用上の強みだ。

## なぜ「ブラウザ内AI」に価値があるのか

クラウドAPIで十分ではないか、という疑問は当然だ。しかしブラウザ内で動かすことには、いくつか本質的なメリットがある。

**プライバシーの構造的な保証**

データがサーバーに送信されない。医療・法律・個人的な相談など、「そもそも外部サーバーに送りたくない」ユースケースに対して、ブラウザ内推論はアーキテクチャレベルでの解決策を提供する。プライバシーポリシーや利用規約の問題ではなく、物理的にデータが外に出ない。

**オフライン動作**

ネットワーク接続が不安定または存在しない環境でも動作する。一度モデルをダウンロードしてキャッシュすれば、以降はオフラインで推論できる。

**レイテンシとコスト**

APIコールが不要なためネットワーク往復時間がゼロになる。また、トークン単価という形での従量課金も発生しない。大量の推論を行うアプリケーションでは、この差は無視できない。

---

## 何に向かないか

現実的な制約も正直に理解しておく必要がある。

**モデルサイズの制限**

現実的に動かせるのは量子化された小〜中規模のモデルだ。7Bや8Bクラスのモデルでも、4bit量子化で数GBになる。GPT-4クラスの大規模モデルをブラウザで動かすことは、現時点では現実的ではない。

**デバイス依存性**

WebGPUに対応しているのはモダンブラウザ（Chrome/Edge 113以降、Firefox nightly）のみで、iOSはWebGPU対応が限定的だ。また、内蔵GPUしか持たないローエンドデバイスでは推論速度が極端に落ちる。

**初回ロード時間**

数GBのモデルをダウンロードする初回は、ユーザーを待たせることになる。IndexedDBへのキャッシュで2回目以降は改善されるが、初回体験の設計は慎重に考える必要がある。

## 設計上の判断軸

Edge AIを採用するかどうかは、以下の軸で考えると整理しやすい。

| 軸 | Edge AI向き | クラウドAPI向き |
|---|---|---|
| プライバシー要件 | 高い（医療・個人情報） | 低い |
| ネットワーク環境 | オフライン・不安定 | 常時接続 |
| モデル規模 | 小〜中（〜10B程度） | 大規模 |
| 推論量 | 大量・頻繁 | 少量・散発的 |
| デバイス対象 | ハイエンド中心 | 全デバイス |

この表は単純化だが、「プライバシーが最優先で、ユーザー層がモダンなデバイスを持っている」場合にEdge AIは強い選択肢になる。

## Transformerウェブ実行の別アプローチ：Transformers.js

WebLLMとは別に、Hugging FaceのTransformers.jsも重要なエコシステムだ。こちらはONNX Runtimeをバックエンドとして使い、テキスト分類・感情分析・Named Entity Recognitionなど、LLM以外のモデルも幅広くカバーする。

推論専用モデル（embeddings、classifier）であれば、Transformers.jsの方が軽量で扱いやすい。WebLLMはチャット向けのテキスト生成に特化している、という使い分けが実用的だ。

---

## ブラウザ内AIが示す未来

Webアプリケーションとデバイスネイティブアプリケーションの境界は、WebGPUによって改めて引き直されている。

「クラウドのAPIを呼ぶだけの薄いクライアント」ではなく、「デバイスのリソースを使って推論する厚いクライアント」という新しいアーキテクチャが成立しつつある。

これはエンジニアとしての設計思考に問い直しを迫る。データはどこに置くべきか、処理はどこで行うべきか——その答えは「常にクラウド」ではなくなった。

WebLLMやWebGPUはまだ発展途上の技術だが、その方向性が示す「エッジでの推論」という思想は、次のWebアーキテクチャの中心テーマになっていくだろう。

---

技術の本質を理解することは、トレンドに流されないための地力になる。ブラウザ内AIが「何を可能にするか」だけでなく「なぜそれが可能になったか」を理解した上で、適切な場面で適切な選択ができるエンジニアを目指したい。
