## はじめに

プロダクトが複数言語を対象にするとき、どの時点で国際化（i18n）を設計するかが問われる。「日本語で作ってから英語対応する」という後追い方式を取ると、コンポーネント中にハードコードされた文字列を全て洗い出す作業が待っている。

一方、最初からi18nを見越した設計をしていれば、言語追加はほぼ翻訳ファイルを用意するだけで済む。

Next.js App Routerと**next-intl**の組み合わせは、現時点でこの問題に対するもっともバランスの取れたアプローチの一つだ。この記事ではnext-intlの設計思想と、App Routerとの統合の考え方を整理する。

---

## App Router時代のi18n設計の変化

Pages Router時代は、Next.jsにi18nのルーティングサポートが組み込まれており、`next.config.js` に `i18n` の設定を書くだけで `/ja` `/en` といったロケールプレフィックスが自動で付与された。

App Routerでは、このビルトインのi18nルーティングサポートが廃止された。代わりに、ディレクトリ構造で言語ルーティングを表現する方式が推奨されている。

```
app/
├── [locale]/
│   ├── layout.tsx
│   ├── page.tsx
│   └── about/
│       └── page.tsx
```

`[locale]` という動的セグメントを最上位に置くことで、`/ja/about` `/en/about` というルーティングを実現する。ロケールを「パスの一部」として明示することで、SEOにも有利だ（言語ごとに異なるURLを持てる）。

## next-intlの選択理由

App Router向けのi18nライブラリはいくつかあるが、next-intlが優れているのは**Server Componentsとの統合**が設計の中心に置かれている点だ。

多くのi18nライブラリはReact Contextに依存しているため、Server Componentsでは使えない。next-intlはServer Components内で直接翻訳関数を呼べる設計になっており、App Routerの思想と整合している。

```typescript
// Server Component内で使える
import { getTranslations } from 'next-intl/server';

export default async function Page() {
  const t = await getTranslations('HomePage');
  return <h1>{t('title')}</h1>;
}
```

Client Componentsでは通常のフック形式で利用する。

```typescript
// Client Component
'use client';
import { useTranslations } from 'next-intl';

export function Button() {
  const t = useTranslations('Common');
  return <button>{t('submit')}</button>;
}
```

この使い分けがシームレスにできるのが、next-intlの大きな強みだ。

## セットアップの全体像

next-intlのセットアップは3つの要素で構成される。

### 1. Middlewareでロケールを判定

```typescript
// middleware.ts
import createMiddleware from 'next-intl/middleware';

export default createMiddleware({
  locales: ['ja', 'en'],
  defaultLocale: 'ja',
});

export const config = {
  matcher: ['/((?!api|_next|.*\\..*).*)'],
};
```

Middlewareがリクエストのヘッダー（`Accept-Language`）やCookieを見てロケールを判定し、適切なパスにリダイレクトする。

### 2. 翻訳ファイルの構造

```json
// messages/ja.json
{
  "HomePage": {
    "title": "ようこそ",
    "description": "このアプリについて"
  },
  "Common": {
    "submit": "送信",
    "cancel": "キャンセル"
  }
}
```

翻訳ファイルはJSON形式で、名前空間（`HomePage`、`Common`など）によって整理する。名前空間はページやコンポーネント単位で切ると管理しやすい。

### 3. i18n設定ファイル

```typescript
// i18n.ts
import { getRequestConfig } from 'next-intl/server';

export default getRequestConfig(async ({ locale }) => ({
  messages: (await import(`./messages/${locale}.json`)).default,
}));
```

## 設計で意識すべきポイント

### 翻訳キーの命名規則

翻訳キーは「どの画面の何か」が分かる名前にする。フラットに `submit` と書くよりも、名前空間を活用して `Common.submit` や `OrderForm.submitButton` とした方が管理が楽になる。

特に複数人で翻訳ファイルを管理する場合、命名規則が曖昧だと似たようなキーが乱立する。「ページ名.コンポーネント名.要素名」という階層を決めておくと衝突を防ぎやすい。

### 動的な値の埋め込み

next-intlは変数埋め込みと複数形をネイティブにサポートする。

```json
{
  "welcome": "{name}さん、ようこそ",
  "itemCount": "{count, plural, =0 {アイテムなし} other {{count}件のアイテム}}"
}
```

```typescript
t('welcome', { name: 'Ryota' }) // → "Ryotaさん、ようこそ"
t('itemCount', { count: 3 })     // → "3件のアイテム"
```

複数形の扱いは言語によって大きく異なる（英語は単複2形、ロシア語は6形など）。next-intlはICUメッセージフォーマットに対応しているため、言語ごとの複数形規則を考慮した翻訳が書ける。

### SEOのための言語タグ

国際化対応において`hreflang`の設定は必須だ。検索エンジンが「この日本語ページの英語版はどこか」を理解するために使われる。

Next.jsのMetadataを使って各ページに設定する。

```typescript
export async function generateMetadata({ params: { locale } }) {
  return {
    alternates: {
      languages: {
        'ja': '/ja',
        'en': '/en',
      },
    },
  };
}
```

## RTL（右から左）言語への備え

アラビア語・ヘブライ語などの言語はテキストが右から左に流れる。将来的にこれらの言語対応が必要になる可能性があるなら、CSSのlogical properties（`margin-inline-start`、`padding-inline-end`など）を最初から使う習慣をつけておくと、RTL対応時のCSSの書き直しが大幅に減る。

```css
/* 物理的な方向 → RTL対応が必要 */
margin-left: 16px;

/* 論理的な方向 → RTLでも自動で反転 */
margin-inline-start: 16px;
```

## まとめ

next-intlをApp Routerと組み合わせることで、Server Components・Client Components両方でシームレスに翻訳が使える構成が実現できる。設計のポイントは3つだ。

1. `[locale]` ディレクトリで言語ルーティングを明示する
2. 翻訳キーを名前空間で整理し、命名規則を統一する
3. 変数・複数形・SEOメタデータを最初から考慮する

国際化は「最後に追加する機能」ではなく、ルーティングとコンポーネント設計の根幹に関わる。早いフェーズでi18nを設計に組み込むことで、言語追加の際のコストを劇的に下げられる。
