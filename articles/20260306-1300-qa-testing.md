## はじめに

モバイルアプリのE2Eテストは、Webアプリのそれより一段と難しい。デバイスの多様性、ネットワーク状態の不安定さ、OS固有の挙動、そしてアニメーションの存在が、テストを「不安定なもの」にしやすい。

Detoxは、React NativeアプリのE2Eテストフレームワークとして広く使われているが、その設計思想はFlutterを含むモバイルE2Eテスト全般に通じる考え方を含んでいる。本記事では、Detoxの仕組みを入口にして、モバイルE2Eテストに向き合うための考え方を整理する。

## モバイルE2Eテストが難しい理由

Webアプリのテストでは、Playwrightや Cypressが「ページの要素を待って、クリックして、結果を確認する」という流れを自然に扱える。しかしモバイルアプリでは、この「待つ」という部分が格段に複雑になる。

まず、アニメーションの問題がある。ボタンをタップしても、アニメーション完了前に次の操作を行うと意図しない挙動が起きる。Webではほとんど問題にならないが、ネイティブアプリでは遷移アニメーション・ローディングアニメーションが随所に存在する。

次に、デバイスとエミュレーターの差異がある。エミュレーター上では再現しない実機特有の問題は珍しくない。逆に実機でのテストは環境構築が複雑で、CIへの組み込みコストが高い。

そして、非同期処理の可視化が難しい。ネットワークリクエスト、データベースの書き込み、バックグラウンドタスクの完了を「待機」するロジックを正確に書かないと、タイミングに依存したフレイキーテストが生まれる。

## Detoxの設計思想：グレーボックステスト

Detoxが他のモバイルテストツールと一線を画す点は、「グレーボックステスト」という考え方を採用していることだ。

ブラックボックステストはアプリを外側から操作する。内部状態を知らずにUIを叩くため、「アニメーションが終わったか」「非同期処理が完了したか」を知る術がない。これがフレイキーテストの主な原因になる。

Detoxはアプリの内部に軽量なサーバーを埋め込み、テストランナーとリアルタイムに通信する。これによって、テストランナーはアプリの「アイドル状態」を検知できる。アニメーションが走っていない、保留中のネットワークリクエストがない、Javascriptの処理キューが空——こうした条件が揃ったときに「準備完了」と判断してから次の操作を行う。

この仕組みにより、開発者は `await waitFor(...).toBeVisible()` などの明示的な待機を書く必要が大幅に減る。テストコードがよりシンプルになり、タイミング依存のバグが発生しにくくなる。

## Flutter × モバイルE2Eテストの現実

FlutterではDetoxをそのまま使うことはできない（Detoxは主にReact Native向け）。Flutter向けのE2Eテストには以下の選択肢がある：

- **Flutter Integration Tests**（公式）: Dartで書けるが、Firebase Test Labなどとの連携が必要
- **Appium**（クロスプラットフォーム）: 汎用性は高いが設定が複雑で速度が遅い
- **Maestro**（後発）: YAMLベースで書けるシンプルさが特徴

Detoxの設計思想——「アイドル状態を検知する」——はFlutterのIntegration Testsにも共通する。FlutterのテストフレームワークはWidgetツリーの再レンダリング完了を待機する仕組みを内蔵しており、`pumpAndSettle()` がその役割を担う。

```dart
// アニメーション・非同期処理が完了するまで待機
await tester.pumpAndSettle();
await tester.tap(find.byKey(Key('submit-button')));
await tester.pumpAndSettle();
expect(find.text('送信完了'), findsOneWidget);
```

この「settle（安定する）まで待つ」という概念は、Detoxの「アイドル状態を待つ」と本質的に同じだ。

## テスト設計で最初に決めるべきこと

Detoxを使う・使わないに関わらず、モバイルE2Eテストの設計前に決めるべきことがある。

**何を守るためのテストか**を明確にすることだ。モバイルE2Eは高コストなテストだ。Unit TestやWidget Testで確認できることは、そちらに任せる。E2Eが担うべきは「ユーザーが実際に経験する重要な操作フロー」の保証だ。

例えば、ログイン→商品検索→カート追加→決済というフローがアプリの核心であれば、そこに絞ってE2Eを書く。ナビゲーションのすべてのパターンをE2Eで網羅しようとするのは間違いだ。

**テスト対象環境を決める**ことも重要だ。エミュレーターのみか、実機か、複数OSバージョンか。コストとカバレッジのバランスをチームで合意しておく。Firebase Test Lab、BrowserStack、AWS Device Farmなどのクラウドサービスを使えば、多様なデバイスでのテストが現実的になる。

**CI/CDへの組み込みタイミング**も設計に関わる。全PRで実行するのか、マージ後のみか、リリース前のみか。モバイルE2Eは実行時間が長いため、開発フローへの影響を考えて組み込み方を決める。

## フレイキーテストと向き合う構え

モバイルE2Eを書いた開発者の多くが最初に直面するのは「テストが不規則に失敗する」問題だ。

原因のほとんどは、実はアニメーション待機の漏れ、非同期処理の完了チェック不足、テスト間の状態汚染のいずれかだ。

状態汚染はモバイルテストで特に注意が必要な問題で、テストが実行されるたびにアプリの状態がきれいにリセットされているかを保証しなければならない。Detoxでは `device.reloadReactNative()` や `device.launchApp({ delete: true })` でアプリをリセットする。Flutterでは `setUp` / `tearDown` で共有状態を明示的にクリアする。

「フレイキーなテストは直さなければならない」という原則は、モバイルE2Eでも変わらない。不安定なテストは信頼を失わせる。失敗を見て「またかな」と思われるようになったテストは、もはや価値を失っている。

## まとめ

- Detoxは「グレーボックステスト」の発想でモバイルE2Eの根本課題（フレイキー）を解決しようとした
- 「アイドル状態を待つ」という設計思想はFlutter Integration Testの `pumpAndSettle()` と共通
- モバイルE2Eは高コストなため「何を守るか」を明確にしてから設計する
- フレイキーテストの主原因は待機漏れと状態汚染——根本的に直す姿勢が重要
- CI/CDへの組み込みタイミングは実行コストを考慮してチームで合意しておく

モバイルE2Eテストは「魔法の安全網」ではない。どの部分を守り、どの部分はより軽いテストに委ねるかを意識した設計が、持続可能なQAプロセスをつくる。
