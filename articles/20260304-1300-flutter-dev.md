# freezedで「変わらないデータ」を設計する——Flutterにおける不変性の価値

Flutterアプリの状態管理を設計するとき、データモデルの作り方は意外と見落とされがちだ。RiverpodやBlocでどう状態を管理するかを考える前に、その「状態を表すデータ」をどう定義するかが土台になる。

freezedは、Dartにおける不変データクラス（Immutable Data Class）を実現するコード生成パッケージだ。本稿では、freezedが解決する問題とその設計思想、そしてなぜ「変わらないデータ」がFluttterアプリの品質を高めるのかを考える。

## Dartのデータクラスに潜む問題

Dartでシンプルなデータクラスを書くと、こんな形になる。

```dart
class User {
  final String name;
  final int age;

  User({required this.name, required this.age});
}
```

一見問題なさそうだが、実際の開発でよく遭遇する問題がいくつかある。

**コピーが難しい**: 一部のフィールドだけ変えた新しいオブジェクトを作りたいとき、手書きの`copyWith`メソッドが必要になる。フィールドが増えるたびに維持コストがかかる。

**等値比較が直感に反する**: `User(name: 'Alice', age: 30) == User(name: 'Alice', age: 30)`は`false`を返す。デフォルトでは参照比較になるためだ。

**JSONシリアライズが面倒**: APIレスポンスとの変換コードを手書きすると、フィールドが増えるたびに更新が必要だ。

**null安全性との格闘**: オプショナルなフィールドを持つデータの表現が冗長になりやすい。

freezedはコード生成によってこれらを解決する。

## freezedが生成するもの

freezedを使うと、アノテーションを付けたクラスからボイラープレートを自動生成できる。

```dart
import 'package:freezed_annotation/freezed_annotation.dart';

part 'user.freezed.dart';
part 'user.g.dart'; // json_serializableと組み合わせる場合

@freezed
class User with _$User {
  const factory User({
    required String name,
    required int age,
    String? bio,
  }) = _User;

  factory User.fromJson(Map<String, dynamic> json) => _$UserFromJson(json);
}
```

`build_runner`でコード生成を実行すると、以下が自動的に得られる。

- **`copyWith`**: `user.copyWith(age: 31)`で一部フィールドだけ変えた新インスタンスを生成
- **`==`と`hashCode`**: 値が同じなら等しいと判定される値比較
- **`toString`**: デバッグしやすい文字列表現
- **`fromJson`/`toJson`**: json_serializableと連携したJSONシリアライズ

これだけでもボイラープレートの大幅削減になるが、freezedの本当の価値は別のところにある。

## 不変性（Immutability）がなぜ重要か

freezedが提供するのはコードの削減だけではない。データを「変わらないもの」として扱う設計思想を強制することで、バグの発生源を根本的に減らす。

Flutterの状態管理でよくあるバグのパターンを想像してほしい。状態オブジェクトを参照で保持し、その内部フィールドを直接変更してしまう。`widget.rebuild()`や`notifyListeners()`を呼ぶのを忘れて、UIが更新されない。複数の箇所から同じオブジェクトを参照していて、一方の変更が意図せず別の場所に影響する。

これらはすべて「ミュータブル（可変）なデータ」に起因するバグだ。

不変データを使うと、状態を変更するときは必ず新しいオブジェクトを生成する。元のオブジェクトは変わらない。これにより「誰かが裏でデータを変えた」という問題が起きなくなる。状態の変化は常に明示的なオブジェクトの置き換えとして現れ、追跡しやすくなる。

Reactの世界でReduxが「状態は不変に扱え」を原則としたのと同じ思想だ。freezedはDart・Flutterにこの原則を持ち込む。

## sealed classとパターンマッチング

freezedの機能の中で、特に強力なのがsealed classのサポートだ。

```dart
@freezed
class AuthState with _$AuthState {
  const factory AuthState.loading() = AuthStateLoading;
  const factory AuthState.authenticated(User user) = AuthStateAuthenticated;
  const factory AuthState.unauthenticated() = AuthStateUnauthenticated;
  const factory AuthState.error(String message) = AuthStateError;
}
```

このように定義すると、`AuthState`は4つの状態のどれかを表すsealed classになる。これをUIで使うとき、`when`メソッドで全ケースを網羅的に処理できる。

```dart
authState.when(
  loading: () => CircularProgressIndicator(),
  authenticated: (user) => HomeScreen(user: user),
  unauthenticated: () => LoginScreen(),
  error: (message) => ErrorScreen(message: message),
);
```

`when`はコンパイル時に全ケースの処理を要求する。新しい状態（例：`suspended`）を追加すると、すべての`when`呼び出し箇所でコンパイルエラーが出る。状態の追加忘れがランタイムではなくコンパイル時に検出される。

これはタグ付きユニオン型（Tagged Union / Algebraic Data Type）と呼ばれる設計パターンで、Rustや関数型言語に親しみがある人には馴染み深い概念だ。

## Riverpodとの組み合わせ

freezedはRiverpodと自然に組み合わさる。Providerが管理する状態をfreezedで定義することで、状態管理の設計が整理される。

状態クラスをfreezedで定義し、Notifierがそれを`copyWith`で更新していく。新しい状態は常に新しいオブジェクトとして生成されるため、Riverpodのリビルドシステムと相性が良い。RiverpodはProviderの値が変わったかどうかを`==`で判定するが、freezedが値比較の`==`を実装してくれるため、正確なリビルド制御が実現する。

不変性と等値比較の保証は、パフォーマンス最適化にも直結する。値が変わっていなければリビルドしない——この判定がfreezedによって確実に機能する。

## コード生成のコスト

freezedのデメリットとして語られるのが、コード生成のコストだ。

`build_runner`の実行が必要で、ファイルを変更するたびに生成を走らせる必要がある。開発中のフィードバックループが遅くなることへの不満もある。生成されたコードがリポジトリに含まれるため、PRのdiffが大きくなることもある。

これらは現実のトレードオフだ。しかしほとんどのプロジェクトでは、freezedが提供する型安全性・ボイラープレート削減・バグ防止の恩恵がコストを上回る。Dart 3でのパターンマッチング強化により、freezedとの連携もよりスムーズになってきた。

## 「変わらないこと」の設計的価値

freezedが体現するのは、「データは変えるな、作り直せ」という設計原則だ。

UIを関数として捉えるFlutterのモデル（`UI = f(state)`）において、stateが不変であることは自然な要求だ。同じstateが渡れば同じUIが出力される——この予測可能性がバグを減らし、テストを書きやすくし、デバッグを容易にする。

freezedはDartの制約の中でこの原則を実現するための道具だ。コードが少し増えても、その分だけ「予測可能で変化を追いやすいシステム」に近づける。

## まとめ

- freezedはDartにおける不変データクラスをコード生成で実現するパッケージ
- `copyWith`・値比較の`==`・`toString`・JSONシリアライズを自動生成
- sealed classとパターンマッチングで状態の網羅的な処理を実現
- 不変性により「裏でデータが変わっている」バグを根本から排除
- Riverpodのリビルドシステムとfreezedのequalityは自然に協調する
- `UI = f(state)`という設計モデルにおいて、不変なstateは本質的な要求
