## はじめに

FlutterアプリでAPIを叩く——これはほぼすべてのアプリで必要になる基本的な処理だ。しかし実際のプロジェクトを見ると、http通信の設計が場当たり的になっているケースは多い。

エラーハンドリングが各画面に散らばっている。認証トークンの更新処理が重複している。タイムアウトが設定されていない。ログが出力されず、問題発生時に原因が追えない——こうした問題は、最初の設計の時点で構造を整えることで大部分が防げる。

この記事では「どのパッケージを使うか」ではなく「http通信をどう設計するか」という視点から、Flutterにおけるネットワーク層の設計思想を整理する。

---

## httpパッケージ vs Dio：選択基準の整理

DartにはPub.devで提供されている代表的なhttpパッケージが2つある。

**httpパッケージ（dart-lang公式）**は、Dart公式が提供するシンプルなクライアントだ。依存が少なく、軽量で、シンプルなAPIを持つ。ただしインターセプター・タイムアウト・キャンセルといった機能は標準では持たず、自前での実装が必要になる。

**Dio**は多機能なサードパーティパッケージで、インターセプター機能・リトライ処理・タイムアウト・フォームデータのアップロード・ダウンロード進捗コールバックなどをデフォルトで備える。

一般的な判断基準として、シンプルなデータ取得のみで良い小規模アプリならhttpパッケージで十分だ。認証トークンのリフレッシュ・共通エラーハンドリング・ログ出力が必要なプロダクションアプリにはDioが適している。

ただし本記事の核心は「どちらを使うか」ではなく、「どちらを選んでも、インフラ層を適切にカプセル化できているか」だ。

---

## 抽象化層を設けることの意味

http通信のパッケージを直接Widgetや状態管理クラスから呼ぶ設計は、後から大きな問題を引き起こす。

パッケージのバージョンアップでAPIが変わった場合、呼び出しが散らばっていると全て修正が必要になる。テスト時にhttp通信をモックしようとすると、依存が深くて差し替えられない。

そのため多くの本番Flutterアプリでは以下のような抽象化層を設ける。

```
Widget / ViewModel
    ↓
Repository（ビジネスロジックのI/F）
    ↓
ApiClient（http通信の抽象化層）
    ↓
Dio / http（実装の詳細）
```

`ApiClient`クラスがDioをラップし、エラーハンドリング・ヘッダー設定・レスポンスのデシリアライズを担当する。`Repository`は`ApiClient`を通してデータを取得し、ドメインモデルに変換して返す。

このレイヤーがあることで、テスト時は`ApiClient`を差し替えてモックデータを返すことができ、パッケージ変更の影響が`ApiClient`の実装に閉じる。

## インターセプターで「横断的関心事」を処理する

Dioのインターセプターは、全てのリクエスト・レスポンス・エラーに対して共通処理を挟む仕組みだ。横断的関心事（Cross-cutting Concerns）を一元管理できる点が最大の強みだ。

**認証インターセプター**はアクセストークンのリフレッシュを自動化する典型的なユースケースだ。401エラーが返ってきたとき、リフレッシュトークンでアクセストークンを再取得し、元のリクエストをリトライする——この処理を各APIコールに書くのではなく、インターセプターに集約する。

```dart
class AuthInterceptor extends Interceptor {
  @override
  void onError(DioException err, ErrorInterceptorHandler handler) async {
    if (err.response?.statusCode == 401) {
      // トークンリフレッシュを試みる
      final refreshed = await _authRepository.refreshToken();
      if (refreshed) {
        // 元のリクエストをリトライ
        final opts = err.requestOptions;
        final response = await _dio.request(
          opts.path,
          options: Options(method: opts.method, headers: opts.headers),
        );
        return handler.resolve(response);
      }
    }
    return handler.next(err);
  }
}
```

**ロギングインターセプター**はデバッグに不可欠だ。Dioには`LogInterceptor`が標準で含まれており、リクエスト・レスポンスの詳細をコンソールに出力できる。本番ビルドでは無効にする条件分岐を忘れずに設定する。

## エラーハンドリングの設計

http通信のエラーは大きく2種類に分かれる。

**ネットワークエラー**：接続タイムアウト・オフライン状態・DNSエラーなど。これらはDioの`DioExceptionType`で分類できる（`connectionTimeout`・`receiveTimeout`・`connectionError`など）。

**APIエラー**：サーバーが返すHTTPエラーレスポンス（4xx・5xx）。ステータスコードとレスポンスボディを組み合わせて意味を解釈する必要がある。

これらを統一して扱うために、アプリ独自のエラー型を定義することが重要だ。

```dart
sealed class AppError {
  const AppError();
}

class NetworkError extends AppError {
  final String message;
  const NetworkError(this.message);
}

class ApiError extends AppError {
  final int statusCode;
  final String message;
  const ApiError({required this.statusCode, required this.message});
}

class UnexpectedError extends AppError {
  final Object error;
  const UnexpectedError(this.error);
}
```

`sealed class`（Dart 3以降）を使うことで、エラー型のパターンマッチングが網羅性チェックと共に機能する。UIレイヤーで「全てのエラーケースを処理したか」をコンパイル時に確認できる。

## タイムアウト設計

見落とされがちなのがタイムアウト設定だ。タイムアウトを設定しないと、サーバーが応答しない場合にリクエストが永遠にペンディング状態になり、ユーザーは操作不能なUIを前に待ち続けることになる。

```dart
final dio = Dio(BaseOptions(
  baseUrl: 'https://api.example.com',
  connectTimeout: const Duration(seconds: 10),
  receiveTimeout: const Duration(seconds: 30),
  sendTimeout: const Duration(seconds: 15),
));
```

`connectTimeout`はサーバーへの接続確立のタイムアウト。`receiveTimeout`はサーバーからのレスポンス受信のタイムアウト。`sendTimeout`はリクエストボディ送信のタイムアウトだ。

エンドポイントの特性に応じて値を変えることも重要だ。画像アップロードのような大きなデータを扱うエンドポイントは`sendTimeout`を長めに設定し、単純なGETリクエストは短く設定する。

## キャンセルとリトライ

Flutterアプリでよくある問題が「画面遷移後にリクエストが残り続ける」ことだ。画面が破棄された後にAPIレスポンスが返ってきて、存在しないWidgetを更新しようとしてエラーになるケースだ。

Dioでは`CancelToken`を使ってリクエストをキャンセルできる。状態管理クラス（RiverpodのNotifierやProviderなど）のdispose時に`cancelToken.cancel()`を呼ぶ設計が推奨される。

リトライについては、Dioのインターセプターに`dio_retry`パッケージを組み合わせるか、独自に実装する。ネットワークエラーは1〜3回リトライし、指数バックオフ（1秒→2秒→4秒）で間隔を広げる設計が一般的だ。認証エラー（401）はリトライではなくトークンリフレッシュで対応する。

---

## まとめ

Flutterのhttp通信設計において「どのパッケージを使うか」は表面的な問いだ。本質的な問いは「どこにエラーハンドリングを集約するか」「認証の横断的処理をどう管理するか」「テストで差し替え可能なレイヤーを作れているか」だ。

インターセプターで横断的関心事を集約し、抽象化層でパッケージへの依存を封じ込め、独自エラー型でUIとの契約を明確にする——この3点を設計の出発点にすることで、後から変更しやすく、テストしやすいネットワーク層が作れる。
