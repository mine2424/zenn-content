## はじめに

アクセシビリティ（a11y）は「障がいのある人のための配慮」と捉えられがちだ。しかし実際には、アクセシビリティの高いUIはすべてのユーザーにとって使いやすい。高コントラストは暗い環境で誰もが恩恵を受け、キーボード操作は電力節約モードのスマートフォンユーザーにも助かる。

にもかかわらず、アクセシビリティテストはQAプロセスの中で後回しにされやすい。「最後に確認すればいい」という扱いを受けるうちに、修正コストが膨れ上がる。

この記事では、アクセシビリティテストをQAプロセスに組み込むための考え方と、自動化と手動テストをどう組み合わせるかを整理する。

---

## WCAGとは何か

**WCAG（Web Content Accessibility Guidelines）**は、W3Cが策定するWebアクセシビリティの国際標準だ。現在の主流はWCAG 2.1で、日本の公共機関や多くの企業ではこれへの準拠が求められている。

WCAGは4つの原則で構成される。

1. **知覚可能（Perceivable）**: 情報や操作が、すべての感覚で知覚できること（例：画像にaltテキスト）
2. **操作可能（Operable）**: UIの操作がキーボードなど様々な手段で行えること
3. **理解可能（Understandable）**: 情報や操作の意味が理解できること
4. **堅牢（Robust）**: 様々な支援技術と互換性があること

各原則には達成基準が設けられており、A（最低限）・AA（標準）・AAA（最高）の3段階でレベルが分類されている。多くのプロジェクトではLevel AAを目標とする。

## 自動化でカバーできる範囲と限界

アクセシビリティの問題は**自動化で約30〜40%しか検出できない**とよく言われる。これはアクセシビリティの本質的な問題だ——「操作できるか」は機械的に検査できるが、「意味が伝わるか」「使い心地が良いか」は人間にしか判断できない。

自動化が得意なのは以下のような機械的な確認だ。

- altテキストの有無（存在チェックのみ）
- フォームラベルとinputの対応関係
- カラーコントラスト比の計算
- ARIAロールの文法的な整合性
- フォーカス可能要素の存在確認

代表的なツールとして**axe-core**がある。axe-coreはCI/CDパイプラインに組み込みやすいライブラリで、Playwright・Cypress・Storybook など主要なテストフレームワークと統合できる。

```javascript
import { chromium } from 'playwright';
import AxeBuilder from '@axe-core/playwright';

const browser = await chromium.launch();
const page = await browser.newPage();
await page.goto('https://example.com');

const results = await new AxeBuilder({ page })
  .withTags(['wcag2a', 'wcag2aa'])
  .analyze();

// violations が空配列であればパス
expect(results.violations).toEqual([]);
```

自動化で検出された違反は「確実に問題がある箇所」だ。まずここを0にすることを目指す。

## 手動テストで補うべきポイント

自動化が届かない領域を手動テストで補う。優先度の高い手動テスト項目を整理する。

### キーボード操作の確認

マウスを使わず、Tabキーだけで全機能が操作できるかを確認する。確認ポイントは3つだ。

- フォーカスの移動順序が論理的か（左から右、上から下）
- フォーカスが視覚的に明確に表示されているか（フォーカスリング）
- モーダルを開いたとき、フォーカスがモーダル内に閉じ込められているか（フォーカストラップ）

フォーカストラップは自動化で検出が難しく、モーダルやドロワーを実装するたびに手動確認が必要になる。

### スクリーンリーダーによる確認

macOSのVoiceOver・WindowsのNarrator・NVDA（無料）などのスクリーンリーダーで操作し、音声読み上げが意味を伝えているかを確認する。

特に注意が必要なのは：

- グラフや図表の代替テキストが内容を説明しているか（単に「グラフ」では不十分）
- ボタンのラベルが行動を表しているか（「こちら」「詳細」ではなく「◯◯の詳細を見る」）
- エラーメッセージがフォームフィールドと正しく紐づいているか

### 色だけに依存していないか

情報を色だけで伝えていないかを確認する。「赤いテキストはエラー」という表現は、色覚特性を持つユーザーには伝わらない。アイコンやテキストラベルを併用する。

## QAプロセスへの組み込み方

アクセシビリティテストは「リリース前の最終チェック」ではなく、開発の各フェーズに組み込む。

### 設計フェーズ

カラーパレット選定時にコントラスト比をチェックする（4.5:1以上がWCAG AA要件）。UIコンポーネントの設計時に「キーボード操作でどう動くか」を仕様に含める。

### 開発フェーズ

StorybookにAxeのアドオンを追加し、コンポーネント単位でアクセシビリティをチェックする習慣をつける。PRレビューの観点にARIAロールの使い方を加える。

### CI/CDパイプライン

mainブランチへのマージ時に、axe-coreを用いた自動テストを必ず実行する。違反があればマージをブロックする設定が理想だ。ただし最初から全ルールを有効にすると既存の問題で詰まるため、まずwcag2a・wcag2aaタグに絞って段階的に厳格化する。

### リリース前

重要フローをキーボード操作・スクリーンリーダーで確認するチェックリストを用意し、リリースチェックリストに組み込む。

## よくある誤解：ARIAを使えばいい？

「ARIA属性を追加すればアクセシビリティが上がる」という誤解がある。ARIAはHTMLのセマンティクスを補強するものであって、セマンティクスを置き換えるものではない。

ファーストルールとして、**「ネイティブHTMLで表現できるなら、ARIAを使わない」**を徹底する。`<div onclick>` に `role="button"` を付けるよりも、素直に `<button>` を使う方が正しい。ネイティブ要素にはキーボード操作・フォーカス管理・スクリーンリーダーへの通知が標準で備わっている。

---

## まとめ

アクセシビリティテストの本質は「誰も取り残さない設計」だ。自動化ツール（axe-core）で機械的な問題を素早く検出し、手動テスト（キーボード操作・スクリーンリーダー）で人間的な体験を確認する。

この2つを組み合わせて開発の各フェーズに組み込むことで、リリース後の大規模修正を防ぎ、すべてのユーザーにとって使いやすいプロダクトを継続的に届けられるようになる。

アクセシビリティは後付けのチェックではなく、設計の最初から考えるべき品質の一部だ。
