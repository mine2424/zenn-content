## はじめに

Next.js App Routerが本格的に普及した今、認証の実装方法を巡る議論が活発になっている。NextAuth.js（Auth.js）、Clerk、Supabase Auth、自前JWT実装——選択肢は豊富だが、それぞれの設計思想と向き不向きを理解せずに選ぶと、後から痛い目を見ることがある。

本記事では、App Routerの文脈で認証をどう設計するかという思想と、主要な選択肢の本質的な違いを整理する。

---

## App Routerが認証設計を変えた理由

Pages Routerの時代、認証はシンプルだった。`getServerSideProps` でセッションを確認し、未ログインなら `redirect` する。クライアントサイドでは `useSession` で状態を参照する。

App Routerはこの構造を根本的に変えた。

- **Server Components がデフォルト**: クライアントサイドのコンテキストが使えない
- **ネストされたルーティング**: レイアウト・ページ・コンポーネントが入れ子で認証境界をどこに引くかが複雑
- **Server Actions**: フォーム送信やデータ変更が直接サーバーで処理される
- **Middleware**: Edge Runtimeでリクエストをインターセプトできる

これらの変化により、「認証情報をどこで確認し、どこまで伝搬させるか」という設計が以前より重要になった。

---

## 認証設計の2つの軸

認証ライブラリを選ぶ前に、自分のシステムがどこに位置するか整理する必要がある。

### 軸1: セッション管理の場所

**Cookie + Server Session方式**: セッションIDをCookieに格納し、サーバー側でセッションストア（Redis等）と照合する。古典的だが確実。

**JWT（Stateless）方式**: 全ての情報をJWTに含め、サーバーはDBを参照せず署名検証だけで認証する。スケーラブルだが、トークン無効化が難しい。

**外部IDプロバイダー方式**: ClerkやAuth0のようなSaaS、またはSupabaseのようなBaaSに認証状態を委任する。自前管理コストがゼロに近い一方、ベンダーロックインがある。

### 軸2: 認証の境界

App Routerでは認証チェックをどこに置くかで使い勝手が変わる。

**Middlewareで全件チェック**: EdgeのMiddlewareでリクエストを捕捉し、認証されていないリクエストをloginページにリダイレクトする。パフォーマンスが良く、漏れが起きにくい。

**Route Handler / Server Actionで都度チェック**: APIエンドポイントや個々のServer Actionで認証を確認する。細粒度の制御ができるが、漏れやすい。

**Server Componentで確認**: ページコンポーネントで `auth()` を呼んでリダイレクト。シンプルだが、コンポーネントごとに書く必要がある。

実際には、Middlewareで大域的な保護をかけつつ、重要なServer Actionでも個別に確認するという多層防衛が現実的だ。

## NextAuth.js（Auth.js）の設計思想

Auth.jsはNext.jsの事実上の標準認証ライブラリとして長く使われてきた。v5（Auth.js）でApp Router対応が大幅に改善された。

**強み**:
- OAuthプロバイダー（Google、GitHub等）の設定が数行で完了
- Prismaや各種DBアダプターとの連携が充実
- 自前ホスティングなのでデータが外に出ない

**設計の特徴**:

```typescript
// auth.ts
import NextAuth from "next-auth";
import GitHub from "next-auth/providers/github";

export const { handlers, auth, signIn, signOut } = NextAuth({
  providers: [GitHub],
  callbacks: {
    authorized({ auth, request: { nextUrl } }) {
      const isLoggedIn = !!auth?.user;
      const isOnDashboard = nextUrl.pathname.startsWith("/dashboard");
      if (isOnDashboard) {
        return isLoggedIn;
      }
      return true;
    },
  },
});

// middleware.ts
export { auth as middleware } from "@/auth";
```

Middlewareでの認証チェックがシンプルに書ける。`auth()` をServer Componentで呼べば現在のセッションを取得できる。

**向いているケース**: 社内ツール、個人プロジェクト、OAuthのみの認証、Prismaを既に使っているプロジェクト。

**注意点**: メール/パスワード認証（Credentials Provider）はセキュリティ上の考慮が増える。自前でパスワードハッシュ、レート制限、ブルートフォース対策を実装する必要がある。

## Clerkの設計思想

Clerkは「認証をSaaSとして提供する」というアプローチを取る。ユーザー管理、MFA、Passkeys、組織管理、webhookまでをホスト型サービスとして提供する。

**強み**:
- 実装が圧倒的に速い（初期設定15分以内）
- UIコンポーネント（SignIn、SignUp等）が付属
- Passkeys、MFA、SSO、組織管理が追加コストなし
- App Router対応が公式にしっかりサポートされている

**設計の特徴**:

```typescript
// middleware.ts
import { clerkMiddleware, createRouteMatcher } from "@clerk/nextjs/server";

const isProtectedRoute = createRouteMatcher(["/dashboard(.*)", "/api/(.*))"]);

export default clerkMiddleware((auth, req) => {
  if (isProtectedRoute(req)) {
    auth().protect();
  }
});
```

Middlewareで保護するルートのパターンを宣言的に書ける。Server Componentでは `auth()` からユーザー情報とセッションを取得できる。

**向いているケース**: SaaS製品、組織・チーム機能が必要なアプリ、Passkeysや高度なMFAを使いたいケース、認証に時間をかけたくないスタートアップ。

**注意点**: ユーザーデータがClerkのインフラに保存される。GDPRやデータレジデンシーの要件が厳しい場合は確認が必要。また、月間アクティブユーザーが増えると費用が発生する。

## Supabase Authという選択肢

Supabase自体をバックエンドとして使っている場合、Supabase Authは自然な選択になる。PostgresベースのRow Level Security（RLS）と組み合わせることで、APIを介さずにDBレイヤーでアクセス制御できる。

```typescript
// Server Component
import { createServerClient } from "@supabase/ssr";
import { cookies } from "next/headers";

export async function getUser() {
  const cookieStore = cookies();
  const supabase = createServerClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
    { cookies: { get: (name) => cookieStore.get(name)?.value } }
  );
  const { data: { user } } = await supabase.auth.getUser();
  return user;
}
```

**向いているケース**: Supabaseを既に使っている、RLSによるDB層でのアクセス制御が必要、OSSなのでセルフホストしたい。

## 選択基準のまとめ

どれを選ぶかは「何を自前管理したいか」で決まる。

| 基準 | Auth.js | Clerk | Supabase Auth |
|------|------|------|------|
| 初期実装コスト | 中 | 低 | 低〜中 |
| ユーザーデータ管理 | 自前 | Clerk管理 | Supabase管理 |
| 高度な認証機能 | 自前実装 | 付属 | 中程度 |
| ベンダーロックイン | なし | あり | あり |
| スケール時コスト | サーバー費のみ | MAU課金 | プラン依存 |

「速く作りたい」→ Clerk  
「データを外に出せない」→ Auth.js  
「Supabaseと統合したい」→ Supabase Auth

## 認証以上に大切なこと

ライブラリ選択よりも重要なのは「認証チェックの漏れをいかになくすか」という設計だ。

App Routerでは、Middlewareをゲートキーパーとして使いつつ、重要なServer Actionでも二重チェックする設計が鉄則だ。Server ActionはHTTPエンドポイントとして外部からアクセスできるため、「画面でボタンが表示されないから安全」という発想は危険だ。

また、認証（Authentication）と認可（Authorization）は別物だ。「ログインしているか」の確認が認証で、「このリソースにアクセスしていいか」が認可だ。認証ライブラリはあくまで認証部分を担当し、認可ロジックはアプリケーション層で設計する必要がある。

---

## まとめ

Next.js App Routerにおける認証設計は、Server ComponentsとMiddlewareの登場によって新しいパターンが確立されつつある。ライブラリの選択は、データ管理方針・チームの速度・要求される認証機能によって変わるが、どれを選んでもMiddlewareでの大域的な保護とServer Actionでの個別確認という多層防衛は共通の原則だ。

認証は「動けばいい」で済ませると後から大きな問題になる領域だ。設計の思想を理解した上で、自分のプロジェクトに合った選択をしてほしい。
