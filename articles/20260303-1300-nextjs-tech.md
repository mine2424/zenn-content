## はじめに

Next.js App RouterはReact Server Componentsを中心に据えた、新世代のフルスタックフレームワークだ。Pages Routerとの最も大きな違いの一つが、データフェッチングとキャッシュの仕組みだ。

`getServerSideProps`や`getStaticProps`というAPIは姿を消し、代わりにコンポーネントの中で直接`fetch`を呼ぶスタイルになった。一見するとシンプルだが、背後にはNext.jsが管理する高度なキャッシュレイヤーが存在する。

この記事では、App Routerのデータフェッチングとキャッシュの設計思想を深く理解し、パフォーマンスとユーザー体験を両立するアプリケーション設計について考える。

---

## 4つのキャッシュレイヤーを理解する

Next.js App Routerには4つのキャッシュメカニズムが存在する。これを理解することが、パフォーマンス設計の出発点だ。

### 1. Request Memoization

同一のリクエストサイクル内で、同じURLへの`fetch`呼び出しを自動的にメモ化する。

たとえば、複数のServer Componentが独立してユーザー情報を取得しても、実際のHTTPリクエストは1回しか発生しない。コンポーネントを小さく分割しても、パフォーマンスペナルティがない設計だ。

これは「React Server Componentレンダリングの一回限り」のメモ化であり、ページリクエストをまたいでは持続しない。

### 2. Data Cache

`fetch`によるデータをサーバーサイドで永続化するキャッシュだ。デフォルトでは`force-cache`（無期限キャッシュ）になっており、明示的に設定を変えない限りキャッシュし続ける。

```typescript
// デフォルト: 無期限キャッシュ
const data = await fetch('https://api.example.com/data');

// キャッシュなし（毎回フェッチ）
const data = await fetch('https://api.example.com/data', {
  cache: 'no-store'
});

// 時間ベースの再検証（60秒ごと）
const data = await fetch('https://api.example.com/data', {
  next: { revalidate: 60 }
});
```

### 3. Full Route Cache

ビルド時またはリクエスト時にルート全体のHTML・RSCペイロードをキャッシュする。静的なルートは一度ビルドされたらCDNで配信され、ユーザーへの応答が非常に高速になる。

動的なルート（`cookies()`や`headers()`を使う、`no-store`でフェッチする）は自動的にこのキャッシュから除外される。

### 4. Router Cache

クライアントサイドでRSCペイロードをキャッシュする仕組みだ。一度訪問したルートへの再訪問が高速になる。ブラウザの戻る/進む操作でのナビゲーションにも効く。

デフォルトの有効期間は、静的ルートで5分、動的ルートで30秒だ。

## 「デフォルトキャッシュ」への誤解

App Routerを使い始めた開発者がよく混乱するのが、「なぜデータが更新されないのか」という問題だ。

これはData Cacheのデフォルトが`force-cache`であることから来る。データを更新したのにUIに反映されない、というバグの多くはキャッシュの設定忘れだ。

**重要な認識転換**: App Routerでは「キャッシュするものを明示する」ではなく、「キャッシュしないものを明示する」設計思想になっている。デフォルトは常にキャッシュされる。

動的なデータを扱う場合は、明示的にキャッシュを外す必要がある：

```typescript
// ルートセグメント全体を動的にする
export const dynamic = 'force-dynamic';

// または個別のfetchを動的にする
const data = await fetch(url, { cache: 'no-store' });
```

## Revalidationの2つの戦略

データを「適度に新鮮に保つ」ための戦略として、2種類のRevalidationがある。

### 時間ベースのRevalidation

`revalidate`オプションで指定した秒数が経過したら、バックグラウンドで再フェッチする。いわゆる「Stale-While-Revalidate」パターンだ。

ユーザーが見るのは常にキャッシュ済みのデータ（高速）で、裏では新しいデータを取得している。次回アクセス時には新しいデータが表示される。

ニュースサイトのトップページや、更新頻度が低い商品カタログなどに適している。

### オンデマンドのRevalidation

`revalidatePath`や`revalidateTag`を使って、特定のキャッシュを即座に無効化できる。データが更新されたタイミングでServer Actionsから呼び出すパターンが典型的だ。

```typescript
// Server Action
async function updateProduct(id: string, data: ProductData) {
  await db.products.update(id, data);
  revalidateTag(`product-${id}`);
}
```

タグによる管理は特に強力だ。「このタグが付いたすべてのfetchキャッシュを無効化する」という粒度でコントロールできる。

## Server Componentsのデータ設計パターン

Server Componentsでのデータフェッチングには、いくつかの良いパターンと避けるべきアンチパターンがある。

### Parallel Fetching（並列フェッチ）

依存関係のないデータは並列で取得する。これは特に意識しないと、ウォーターフォールが生じてパフォーマンスが劣化する。

```typescript
// ❌ シリアル（合計2秒かかる）
const user = await fetchUser(userId);
const products = await fetchProducts();

// ✅ 並列（最大1秒で終わる）
const [user, products] = await Promise.all([
  fetchUser(userId),
  fetchProducts()
]);
```

### データのコロケーション

データをコンポーネントの近くに置くのがApp Routerの哲学だ。「どのコンポーネントがどのデータを必要とするか」が明確になる。

Request Memoizationがあるため、同じデータを複数のコンポーネントが取得しても問題ない。Props drilling でデータを渡し続けるより、各コンポーネントが自分の必要なデータを取得する方が設計がシンプルになる。

## Suspenseとの協調

データフェッチングとSuspenseは切り離せないペアだ。

Server ComponentでのデータフェッチングはSuspenseと組み合わせることで、ストリーミングレンダリングが実現できる。重いデータフェッチが完了するのを待つ間、軽い部分のHTMLから順次クライアントに送り始める。

ページ全体が白いまま待たせるより、ナビゲーションバーや静的コンテンツをまず表示してから、データが必要な部分をSuspenseで囲んでローディング状態を見せる方が、体感パフォーマンスが大幅に改善する。

## キャッシュ設計の実践的指針

実際のアプリケーション設計でキャッシュをどう考えるか、判断のフレームワークを整理する。

**データの性質で決める:**

| データの性質 | キャッシュ戦略 |
|------------|-------------|
| ほぼ変化しない（規約・FAQなど） | 無期限 or 長時間revalidate |
| 1時間〜1日の鮮度で十分 | 時間ベースのrevalidate |
| 更新されたらすぐ反映したい | オンデマンドrevalidate |
| ユーザー固有・リアルタイム | no-store（キャッシュなし） |

**ユーザー認証も重要な軸**:
認証済みユーザーのパーソナルデータは、そのユーザーにしか見せてはいけない。Data CacheはサーバーサイドのキャッシュでCookieを含まないため、基本的に認証済みデータには使えない（`no-store`にする）。

## おわりに

Next.js App RouterのキャッシュとデータフェッチングはPagesRouterより確かに複雑に見える。しかし、その複雑さは解決しようとしている問題の複雑さに見合っている。

「デフォルトでキャッシュされる」という設計思想は、意識せずとも高速なアプリケーションが出来上がることを目指している。そこに動的な要件を「例外として明示する」スタイルは、パフォーマンスと動的性のバランスを取る現実的なアプローチだ。

キャッシュの仕組みを理解することは、App Routerを使いこなす上で避けられない投資だ。しかしその投資に見合う、従来の手動最適化では実現しにくいパフォーマンス特性をNext.jsは提供してくれる。
