## はじめに

「意味で検索する」という言葉は、いまや技術的な文脈でも日常的に使われるようになった。Semantic Search、ベクトル検索、エンベディング——これらの言葉は耳にするが、実際のところ「意味を捉える」とはどういうことなのか、腑に落ちていないエンジニアも多いのではないだろうか。

本記事では、エンベディングの仕組みと、それがなぜ「意味の近さ」を表現できるのか、そしてSemantic Searchをシステムとして設計する際に考えるべきことを整理する。

---

## エンベディングとは何か

エンベディング（Embedding）とは、テキスト・画像・音声などのデータを、高次元の数値ベクトルに変換したものだ。例えば「猫」という単語は、`[0.23, -0.71, 0.55, ...]` のような数百〜数千次元のベクトルとして表現される。

重要なのは、この変換が「意味的に似た概念を空間上で近くに配置する」ように学習されている点だ。

- 「犬」と「猫」は似たベクトル空間にある
- 「走る」と「歩く」も近い
- 「銀行（金融）」と「銀行（川岸）」は文脈によって異なるベクトルを持つ（文脈埋め込みの場合）

この「空間上の距離＝意味の近さ」という性質が、Semantic Searchを可能にする。

---

## なぜベクトルで意味を捉えられるのか

モダンなエンベディングモデル（BERT、OpenAI text-embedding シリーズ、GTE、E5など）は、大量のテキストデータから学習されたTransformerベースのモデルだ。

これらのモデルは、自己教師あり学習によって「同じ文章の中で一緒に出現する単語」「似た意味で置き換えられる表現」「文脈上近い概念」などを内部表現として学習している。

結果として、モデルは表層的な単語の一致ではなく、意味的な近接性を数値として埋め込むことを覚える。

```python
from sentence_transformers import SentenceTransformer
import numpy as np

model = SentenceTransformer("intfloat/multilingual-e5-large")

sentences = [
    "今日は天気がいい",
    "晴れていて気持ちがいい日だ",
    "データベースの設計をしている",
]

embeddings = model.encode(sentences)

# コサイン類似度を計算
def cosine_sim(a, b):
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

print(cosine_sim(embeddings[0], embeddings[1]))  # 高い（0.9前後）
print(cosine_sim(embeddings[0], embeddings[2]))  # 低い（0.3前後）
```

「天気がいい」と「晴れていて気持ちがいい日だ」は、単語は一致していないが、意味的に近いためコサイン類似度が高くなる。

## Semantic Searchのアーキテクチャ

Semantic Searchをシステムとして構築する場合、大きく3つのフェーズがある。

### 1. インデックス構築フェーズ

検索対象のドキュメントをあらかじめエンベディングモデルで変換し、ベクトルデータベース（Pinecone、Weaviate、Qdrant、pgvectorなど）に格納する。

この段階で重要なのが**チャンキング戦略**だ。ドキュメントをどの粒度で分割してエンベディングするかによって、検索精度が大きく変わる。

- 粒度が大きすぎると → 意味が希薄になる
- 粒度が小さすぎると → 文脈が失われる

一般的には段落単位・セクション単位が起点になるが、オーバーラップを設けることで境界部分の意味損失を防ぐ手法もよく使われる。

### 2. クエリ処理フェーズ

ユーザーのクエリも同じモデルでエンベディングし、ベクトルDB上でANN（Approximate Nearest Neighbor）検索を実行する。コサイン類似度やドット積でスコアリングされた上位N件が返される。

### 3. リランキングフェーズ（オプション）

ANN検索は速度を重視した近似探索なので、上位候補を取得した後にクロスエンコーダーモデルで再スコアリング（リランキング）することで精度を高めるアプローチが主流になってきた。

## Hybrid Searchという現実解

純粋なSemantic Searchには弱点もある。

- **固有名詞・略語に弱い**: 「GPT-4o」「Supabase」のような固有表現は、意味的な近さよりも文字列の一致で探した方が正確なことが多い
- **最新情報への対応**: 学習データに含まれていない新しい概念はうまく埋め込めない

この問題に対処するのが**Hybrid Search**だ。

Hybrid Searchは、BM25などのキーワードベース検索（スパース検索）とSemantic Search（デンス検索）を組み合わせる手法だ。両者のスコアをRRF（Reciprocal Rank Fusion）などで統合することで、固有名詞にも意味にも対応できるバランスの良い検索が実現する。

| 観点 | キーワード検索 | Semantic Search | Hybrid |
|------|------|------|------|
| 固有名詞 | ◎ | △ | ◎ |
| 意味的類義語 | △ | ◎ | ◎ |
| 速度 | 速い | やや遅い | 中間 |
| 実装コスト | 低 | 中 | 高め |

## エンベディングモデルの選択

どのエンベディングモデルを使うかは、検索精度に直結する重要な選択だ。

**多言語対応が必要な場合**: `multilingual-e5-large`（Microsoft）や`paraphrase-multilingual-mpnet-base-v2`（SBERT）が候補になる。日本語を含むシステムでは多言語モデルを選ぶことが基本だ。

**英語特化の場合**: `text-embedding-3-large`（OpenAI）や`e5-mistral-7b-instruct`などが高精度を誇る。

**ローカル運用したい場合**: `intfloat/e5-large-v2`や`BAAI/bge-large-en-v1.5`がOSSで利用でき、Ollamaでホスティングすることも可能だ。

注意点として、インデックス構築時とクエリ処理時は**必ず同じモデル**を使わなければならない。モデルが変わればベクトル空間の構造が変わるため、全インデックスの再構築が必要になる。

---

## 「意味の近さ」の限界を知る

エンベディングは万能ではない。以下のような場合に精度が落ちることを意識しておく必要がある。

**ドメイン特化の専門用語**: 医療・法律・金融などの高度に専門化されたドメインでは、汎用モデルでは意味の近さを正確に捉えられないことがある。この場合、ドメインデータでファインチューニングしたモデルが有効だ。

**否定表現**: 「Aではない」と「Aである」が意味的に遠いベクトルになるとは限らない。否定の意味を空間上で正確に表現することはまだ難しい。

**多義語と文脈**: 文脈埋め込みモデル（BERT系）は文脈をある程度捉えるが、完全な文脈理解にはLLMレベルの推論が必要な場合もある。

## まとめ

エンベディングは「意味を数値ベクトルに変換する」技術だが、その本質は「類似した意味を空間上の近さとして表現する」ことだ。この性質を利用したSemantic Searchは、単なるキーワードマッチングを超えた検索体験を提供する。

一方で、固有名詞や専門用語への対応、モデル選択の重要性、チャンキング戦略など、設計上考慮すべきことも多い。Hybrid Searchを視野に入れながら、システムの要件に合わせた設計を選ぶことが実践的なアプローチになる。

「意味で検索する」という概念は、RAGやAIエージェントの基盤でもある。エンベディングの設計思想を理解することで、より信頼性の高いAIシステムを構築できるようになるはずだ。
