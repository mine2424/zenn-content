# ISR（Incremental Static Regeneration）がNext.jsの「静的と動的」を融合させる設計思想

静的サイトは速い。しかし内容が古くなる。動的サイトは常に新鮮だ。しかしリクエストのたびにサーバーで計算する。この二択に悩んできたWebエンジニアに対して、Next.jsは「第三の道」を提示した。それが**ISR（Incremental Static Regeneration）**だ。

ISRは技術的なトリックではなく、「何を事前計算して何をリアルタイムにするか」という設計判断を精緻化した思想の具現化だ。

---

## 静的生成と動的レンダリングの間にある問題

Webアプリケーションのレンダリング戦略を考えるとき、歴史的な選択肢は大まかに三つあった。

**SSG（Static Site Generation）**：ビルド時にすべてのHTMLを生成する。CDNから即座に返せるため高速だが、コンテンツを更新するには再ビルドが必要になる。

**SSR（Server-Side Rendering）**：リクエストのたびにサーバーでHTMLを生成する。常に最新のデータを返せるが、毎回の計算コストがかかりスケールが難しい。

**CSR（Client-Side Rendering）**：HTMLは最小限でクライアントがJSでDOMを構築する。動的だが初期表示が遅く、SEOにも不利になりやすい。

ECサイトを考えると、このジレンマが明確になる。商品ページは何百万もあり、毎回サーバー生成するとコストが高い。かといって全ページをビルド時に生成するとビルド時間が膨大になる。価格や在庫は刻々と変わるため、古い静的HTMLを長期間配信するのも問題だ。

ISRはこの問題を「事前生成 + バックグラウンド再生成」という構造で解く。

---

## ISRの動作原理：stale-while-revalidate の思想

ISRの核心はHTTPキャッシュの世界でおなじみの**stale-while-revalidate**という考え方だ。

「古いコンテンツをすぐ返しながら、バックグラウンドで新しいコンテンツを取得する」——この戦略をページ生成に応用したものがISRだ。

動作の流れを整理すると次のようになる。

1. 最初のリクエスト：ページがまだ生成されていなければ、SSR的にオンデマンドで生成しキャッシュする
2. 2回目以降（revalidate時間内）：キャッシュされた静的HTMLをすぐに返す
3. revalidate時間が過ぎた後のリクエスト：古いHTMLをすぐ返しながら（ユーザーを待たせない）、バックグラウンドでページを再生成してキャッシュを更新する
4. 次のリクエスト：再生成された新しいHTMLが返る

App RouterではServer Componentの `revalidate` オプションで、この挙動を宣言的に制御できる。

```javascript
// fetch のキャッシュオプションで revalidate を指定
const data = await fetch('https://api.example.com/products', {
  next: { revalidate: 3600 } // 1時間ごとに再検証
})
```

## Pages Routerの `getStaticProps` から App Routerへの設計の変化

Pages Routerでは `getStaticProps` に `revalidate` を指定するのがISRの書き方だった。

App Routerになって、ISRの設定はより細粒度になった。ページ単位ではなく、**データフェッチ単位**でrevalidateを設定できる。同じページ内でも「商品情報は1時間ごと」「在庫数は5分ごと」という異なる鮮度要件を、それぞれの `fetch` に設定できる。

この変化は本質的で、「ページの鮮度」ではなく「データの鮮度」を管理するという思想の転換だ。ページはデータの容れ物に過ぎず、データの要件に応じてキャッシュ戦略が決まる。

---

## On-Demand Revalidation：時間ではなくイベントで再生成する

時間ベースのrevalidationは、「いつ更新されるかわからないが、定期的に確認する」アプローチだ。これはCMSのブログ記事のような用途には適している。

しかしECサイトの商品情報なら、管理画面で更新が行われた瞬間に再生成したい。このためのAPIが**On-Demand Revalidation**だ。

```javascript
// app/api/revalidate/route.ts
import { revalidatePath } from 'next/cache'

export async function POST(request: Request) {
  const { path } = await request.json()
  revalidatePath(path)
  return Response.json({ revalidated: true })
}
```

CMSのWebhookがこのAPIを叩くことで、コンテンツが更新された瞬間にNext.jsが対象ページのキャッシュを無効化し、次のリクエスト時に再生成が走る。

`revalidatePath` はパスを指定して特定ページを無効化するが、`revalidateTag` を使うとより細かい制御ができる。データに「タグ」をつけておき、そのタグに対してrevalidateを呼ぶことで、複数ページにまたがるデータの無効化が一括でできる。

## ISRを選ぶべき場面とそうでない場面

ISRが最も効果を発揮するのは、「コンテンツは更新されるが、リアルタイム性が絶対ではない」ユースケースだ。

**ISRに向いている**：  
- ブログ・ドキュメントサイト（数時間に一度更新）  
- ECサイトの商品詳細（CMSからのWebhookでOn-Demand revalidation）  
- ニュースメディアのアーカイブページ  
- ランディングページ（マーケティングチームが随時更新）

**ISRが向いていない**：  
- ログイン後のパーソナライズされたコンテンツ（ユーザーごとに異なるため静的生成できない）  
- リアルタイムの価格・在庫（秒単位で変わるため静的化の意味がない）  
- セッション依存のページ（静的キャッシュとの相性が悪い）

後者はSSRかCSRで対応するのが適切だ。ISRとSSRを同一アプリ内で混在させることもNext.jsは許容しており、「このページはISR、このページはSSR」という設計が現実的に取れる。

---

## ビルド戦略との組み合わせ：全部事前生成しない勇気

App Routerでは `generateStaticParams` で静的生成するパラメータを指定できる。しかし、すべての商品IDを事前生成する必要はない。

`dynamicParams: true`（デフォルト）に設定しておけば、 `generateStaticParams` に含まれないIDへのリクエストは「オンデマンドに初回生成→以降はキャッシュ」という動きをする。これはPages Routerの `fallback: 'blocking'` に相当する。

「人気上位100商品だけ事前生成し、それ以外はアクセスされた時点で初回生成してキャッシュ」——この戦略はビルド時間を抑えながら、人気ページのパフォーマンスを最大化する実用的なアプローチだ。

---

## まとめ：鮮度とパフォーマンスのトレードオフを設計する

ISRが提示する設計思想は「鮮度とパフォーマンスはトレードオフではなく、適切なキャッシュ戦略によって共存できる」というものだ。

重要なのは、revalidate時間をどう設定するかという技術的な問いではない。「このデータはどの程度の鮮度が必要か」「ユーザーにとって許容できる古さはどこまでか」という要件を明確にし、それをコードで表現する設計判断が本質だ。

静的の速さと動的の新鮮さを、コンテンツの特性に応じて使い分ける——ISRはその判断軸を提供するための仕組みだ。
