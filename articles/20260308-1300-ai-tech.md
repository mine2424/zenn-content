## Hallucinationとどう付き合うか

LLM（大規模言語モデル）を実際のシステムに組み込んでいると、避けては通れない問題がある。それが「Hallucination（幻覚）」と呼ばれる現象だ。モデルが存在しない事実を自信満々に語り、ユーザーを誤った方向へ導く。

「精度の高いモデルを使えばいい」「プロンプトを工夫すれば減らせる」——そういったアプローチを試みた経験のある開発者は多いだろう。しかし実際には、どれだけ優れたモデルを使っても、Hallucinationを完全に排除することはできない。

本稿では、Hallucinationを「ゼロにしようとする」のではなく、「発生前提で設計する」という発想の転換と、その具体的なアーキテクチャパターンを考えていく。

## Hallucinationが生まれる構造的な理由

問題の本質を理解するには、LLMがどのように動いているかを知る必要がある。

LLMは大量のテキストデータから「次のトークンを予測する」という学習を積み重ねている。この学習の結果、モデルは「それらしい続き」を生成することが非常に得意になる。しかし「それらしい」と「正しい」は別物だ。

特にモデルが以下のような状況に置かれると、Hallucinationが起きやすくなる。

**知識のカットオフ**：学習データには時間的な限界がある。最新情報や学習後の出来事については、モデルは何も知らない。にもかかわらず、質問されると「それらしい答え」を返してしまう。

**ロングテールな知識**：有名な事実や一般的な知識は大量の学習データに含まれるため精度が高い。しかしニッチな専門知識や特定組織内の情報については、学習データが少ないためモデルの推測に頼る部分が大きくなる。

**プロンプトの誘導**：「〜について詳しく説明してください」という形式は、モデルに「詳しく語る」という振る舞いを促す。たとえ根拠がなくても、それらしく語ろうとするバイアスが働く。

これらは設計上の欠陥ではなく、言語モデルという仕組みの構造的な特性だ。だからこそ「防ぐ」ではなく「前提にする」という設計が重要になる。

## 検出アプローチの分類

Hallucinationへの対処法は大きく「生成前（事前）」「生成後（事後）」「ループ型」の3つに分類できる。

### 生成前の対策：情報を与えてから生成させる

最も効果的な事前対策はRAG（Retrieval-Augmented Generation）だ。モデルに「知っていること」を答えさせるのではなく、「与えられた文脈から答えを生成させる」形式にする。

```python
# RAGの基本的な考え方
context = retriever.search(user_query)
prompt = f"""
以下の情報のみを使って質問に答えてください。
情報に含まれていない場合は「わかりません」と答えてください。

【情報】
{context}

【質問】
{user_query}
"""
response = llm.generate(prompt)
```

重要なのは、モデルに「知らないと答えるオプション」を与えることだ。多くのHallucinationは、モデルが「答えなければいけない」と感じることで生まれる。「情報がない場合はその旨を伝える」という明示的な許可を与えることで、出力の信頼性は大きく向上する。

### 生成後の検証：出力を別のプロセスで確認する

生成されたテキストを後から検証するアプローチもある。いくつかのパターンを見ていこう。

**引用元の照合**：RAGシステムでは、モデルが回答に使ったはずの引用元を特定できる。回答文と引用元ドキュメントを比較して、回答の内容が実際に引用元に含まれているかを検証する。これは専用の検証モデルを使うこともあれば、同じLLMに「この回答は与えられた文脈から導けますか？」と確認させることもある。

**一貫性チェック（Self-Consistency）**：同じ質問を複数回（または異なるサンプリングパラメータで）生成し、回答の一貫性を見る。複数の生成結果が同じ方向を向いているなら、そのコンテンツは比較的信頼できる可能性が高い。反対に、毎回異なる答えが出てくる場合は、モデルが確信を持っていない可能性がある。

**信頼スコアの付与**：モデルに「この回答に何パーセントの確信がありますか？」と問いかける手法も使われる。ただしモデル自身の自己評価は必ずしも正確ではない。より精度の高い信頼スコアを得るためには、トークンの確率分布（log-probability）を直接参照する方法もある。

### ループ型：生成と検証を繰り返す

より高度なアーキテクチャとして、生成と検証をループさせる設計がある。モデルが回答を生成し、その回答が不十分または誤りと判断された場合に再生成を試みる。

このパターンはAIエージェントのアーキテクチャと親和性が高い。「計画→実行→検証→修正」というサイクルの中に、Hallucination検出を組み込む形だ。ただし、ループ回数が増えるほどレイテンシとコストが上昇するため、終了条件の設計が重要になる。

## ユーザーへの伝え方：確信度の透明性

技術的な対策と同様に重要なのが、ユーザーへの情報提示の設計だ。

「答えを返す」だけでなく、「この答えがどれだけ信頼できるか」を伝えることがUXとしても誠実なアプローチとなる。

実装上のパターンとしては：

- **情報源の提示**：回答の根拠となったドキュメントや URL を一緒に表示する
- **不確実性の明示**：「確認が取れていますが…」「不確かですが…」といった前置きをモデルに生成させる
- **フォールバック設計**：信頼スコアが一定以下の場合は、「この質問は専門家に確認することをお勧めします」といったメッセージを出す

## ドメインに応じたリスク評価

Hallucinationのリスクは、システムが扱うドメインによって大きく異なる。

医療や法律のような分野では、誤った情報が直接的な被害につながる可能性がある。これらのドメインでは、より厳しい検証ロジックと、人間によるレビューループを組み合わせる必要がある。

一方、クリエイティブな用途（文章のアイデア出しや下書き生成など）では、ある程度のHallucinationは許容できるどころか、むしろ新しいアイデアとして機能することもある。

「どの程度のHallucinationを許容するか」という意思決定は、技術的な問題である前にプロダクト設計の問題だ。ユースケースのリスク評価を行い、それに見合った検証の厳密さを設計する。

## 設計の指針：完璧を目指さず、透明性を高める

LLMのHallucinationを完全になくすことは、現時点では不可能だ。しかし、それは「使えない」という意味ではない。

設計の指針として：

1. **Hallucinationを前提にアーキテクチャを組む**：事後検証とフォールバックを最初から設計に含める
2. **知識を外部から注入する**：RAGやツール呼び出しで、モデルの記憶に頼らない設計を優先する
3. **不確実性を可視化する**：確信度の低い回答を「信頼できるもの」として提示しない
4. **ドメインリスクに応じた設計を**：リスクが高い領域では人間のレビューを組み込む

「信頼できるAIシステム」とは、間違えないシステムではなく、間違いを認識して適切に扱えるシステムだ。その設計思想が、実用的なLLMアプリケーションの土台となる。
