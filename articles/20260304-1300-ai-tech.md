# LLMをどう評価するか——AI評価指標の設計思想と実践

LLMを本番環境に投入する際、最も難しい問いのひとつが「このモデルは十分に良いか」をどう判断するか、だ。精度が高ければ良い？ユーザー満足度？コストが低ければ？何を指標にするかによって、開発の方向性はまったく変わってくる。

本稿では、LLMアプリケーションにおける評価指標の設計思想を整理し、「何を・なぜ・どう測るか」を深く考える。

## なぜLLM評価は難しいのか

従来の機械学習では、正解ラベルとの比較が評価の基本だった。分類モデルならF1スコア、回帰モデルならRMSEが使える。しかしLLMの出力は自由なテキストであり、「正解」が一意に定まらないことが多い。

たとえば「Pythonのリスト内包表記を説明してください」という問いへの回答は、何通りでも正しい答えがあり得る。短くシンプルな説明が良い場合もあれば、例示が豊富な詳細な説明が良い場合もある。文脈によって「正解」が変わるのだ。

この「正解の多様性」が、LLM評価を本質的に難しくしている。そして多くのチームが陥る罠は、評価の難しさから逃げて「主観的に良さそう」という感覚だけで本番投入を判断してしまうことだ。

## 評価の3層構造

LLM評価を整理するために、3つの層で考えると分かりやすい。

**第1層：タスク正確性（Task Accuracy）**

最も基本的な評価層。特定のタスクに対して、出力が期待される基準を満たしているかを測る。

- 分類タスク: 正解率、F1スコア
- 要約タスク: ROUGE、BERTScore
- コード生成: pass@k（k回中何回正しいコードが生成されるか）
- QA: Exact Match、F1

ただしこれらの指標には限界がある。たとえばROUGEは単語の重複率を測るが、意味の正確さは測れない。「犬が猫を追った」と「猫が犬に追われた」は意味が同じだがROUGEスコアは高く出る。

**第2層：品質評価（Quality Assessment）**

出力の質的な側面を評価する層。ここでは人間評価とLLM-as-a-Judgeの2つのアプローチがある。

人間評価は信頼性が高いが、コストと時間がかかる。LLM-as-a-Judgeは、GPT-4などの高性能モデルに評価者役を担わせるアプローチだ。「以下の回答を1〜5で評価し、理由を述べてください」という形式で、評価の自動化が実現できる。

```python
def evaluate_with_llm(question: str, answer: str, criteria: str) -> dict:
    prompt = f"""
    以下の質問と回答を評価してください。

    質問: {question}
    回答: {answer}

    評価基準:
    {criteria}

    評価（1-5）と理由を JSON 形式で返してください。
    """
    # LLM API呼び出し...
```

LLM-as-a-Judgeは便利だが、バイアスの問題がある。長い回答を好む傾向、自分自身（同じモデル）の出力を過大評価する傾向などが報告されている。

**第3層：ビジネス指標（Business Metrics）**

最終的に重要なのは、プロダクトの価値を高めているかどうかだ。

- ユーザー満足度（CSAT）
- タスク完了率
- セッションあたりの質問数（少ない方が解決が早い）
- エスカレーション率（人間対応に切り替わる頻度）

技術的評価と乖離しやすいのがこの層だ。モデルの精度が上がっても、ユーザー体験が改善しないケースはよくある。

## Hallucination評価

LLMの深刻な問題のひとつがHallucination（幻覚）——存在しない事実を自信を持って述べてしまう現象だ。

Hallucination評価は、大きく2種類に分けられる。

**忠実性（Faithfulness）**: RAGなどで参照文書がある場合、回答が文書の内容から外れていないかを測る。参照文書から根拠を引用できるか、回答内容が文書に含まれる情報のみで構成されているか、といった観点だ。

**事実性（Factuality）**: 参照文書なしで、回答が客観的な事実と一致しているかを評価する。これはより難しく、別の知識ベースや検索システムを用いた確認が必要になる。

Hallucinationを完全になくすことは現状困難だが、測定することで改善の方向性が見えてくる。どのカテゴリの質問でHallucinationが多いか、どのプロンプト設計が効果的かを定量的に追跡できるようになる。

## RAGシステムの評価フレームワーク

RAGを採用したシステムでは、評価がさらに複雑になる。生成（Generation）だけでなく、検索（Retrieval）の品質も評価対象だからだ。

RAGASというフレームワークが参考になる。以下の4指標を組み合わせる。

- **Context Precision**: 検索された文書が質問に関連しているか
- **Context Recall**: 回答に必要な情報が検索で取得できているか
- **Faithfulness**: 回答が検索文書の内容に忠実か
- **Answer Relevancy**: 回答が質問に対して適切か

これらを組み合わせることで、「検索は良いが生成が悪い」「生成は良いが検索が足りない」といった問題の切り分けができる。

## 評価データセットの設計

どんなに良い指標も、評価データセットが貧弱では意味がない。

評価セットの設計で重要なのは、**現実の分布を反映すること**だ。本番環境でどんな質問が来るかを観察し、それを評価セットに組み込む。開発者が「こんな質問が来るだろう」と想定したセットは、しばしば現実と乖離する。

また、**エッジケースを意図的に含める**ことも重要だ。曖昧な質問、複数の解釈ができる質問、モデルが苦手とする領域の質問を意識的に収集する。

評価セットは生き物だ。プロダクトが進化し、ユーザー層が変わり、新しい機能が追加されるたびに更新が必要になる。

## 継続的評価の仕組みづくり

一度評価して満足するのではなく、継続的に評価を回す仕組みが重要だ。

モデルバージョンのA/Bテスト、プロンプト変更前後の比較、本番ログからの自動評価サンプリング——これらを開発フローに組み込むことで、「勘」ではなく「データ」で判断できるようになる。

MLflowやW&Bのようなツールで評価結果を追跡し、モデルやプロンプトのバージョンと紐付けて管理することで、「なぜ性能が変わったのか」を遡って調査できる。

## 評価は設計から始まる

LLMの評価で最も大切なのは、**評価指標を後付けしないこと**だ。

プロダクトの目的を定義する段階で「何を成功とするか」を言語化し、それに対応する評価指標を設計する。そしてそのための評価セットを収集・設計する。

評価は面倒な作業ではなく、プロダクトの品質を支える設計行為だ。LLMアプリケーションの開発において、評価設計への投資は必ず報われる。

## まとめ

- LLM評価は「タスク正確性・品質評価・ビジネス指標」の3層で考える
- Hallucinationは忠実性と事実性の両面から評価する
- RAGシステムはRetrieval・Generationそれぞれを評価する
- 評価データセットは現実の分布を反映させ、継続的に更新する
- 評価指標はプロダクト設計の段階から定義する

「測れないものは改善できない」——LLMの世界でも、この原則は変わらない。
