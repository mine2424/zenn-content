## はじめに

Flutterのパフォーマンス問題の多くは、「不要なWidgetの再ビルド」に起因している。

`setState`を呼ぶたびにツリーが再構築される——これはFlutterの基本的な仕組みだが、設計を誤るとアプリ全体が毎フレーム再描画される地獄に陥る。

本記事では、FlutterのWidgetツリーがどのように再ビルドされるかのメカニズムを理解した上で、パフォーマンスを意識した設計パターンを考える。状態管理ライブラリを使っていても、この考え方は変わらない。

---

## Flutterのレンダリングの3ツリーを知る

Flutterには実は3種類のツリーが存在する。

1. **Widgetツリー**: 設定の軽量な記述（Immutable）
2. **Elementツリー**: WidgetとRenderObjectをつなぐ仲介（Mutable）
3. **RenderObjectツリー**: 実際のレイアウト・描画を担う

`setState`が呼ばれると、Widgetの`build`メソッドが再実行される。ただし、これはWidgetツリーの再構築であって、RenderObjectの再生成ではない。

Flutterのフレームワークは「差分検出（Reconciliation）」を行い、変化のあったElementのみを更新する。つまり、`build`が呼ばれても常に全画面が再描画されるわけではない。

しかし問題は、`build`メソッドの実行コスト自体と、不要なサブツリーへの伝播だ。

---

## 再ビルドが伝播する仕組み

`StatefulWidget`で`setState`を呼ぶと、そのWidgetの`build`が再実行される。そしてそのbuildが返したWidgetツリー全体が再ビルドの対象になる。

これが問題になるのは、「ルートに近いStatefulWidgetが状態を持ち、大きなツリーを返す」構造だ。

たとえば、カウンターの値をアプリルートの`State`で管理し、そこから深いネストのWidgetまで`count`を渡すとする。カウンターが変わるたびに、そのルートの`build`が再実行され、全子孫が再ビルドされる。

```dart
// ❌ よくないパターン
class HomeScreen extends StatefulWidget {
  @override
  State<HomeScreen> createState() => _HomeScreenState();
}

class _HomeScreenState extends State<HomeScreen> {
  int count = 0;

  @override
  Widget build(BuildContext context) {
    return Column(
      children: [
        HeavyListWidget(), // countと無関係なのに毎回再ビルドされる
        CounterWidget(count: count),
      ],
    );
  }
}
```

## 再ビルドを最小化する設計原則

### 原則1: 状態は「使う場所の最も近く」に置く

「状態はできるだけ上位に持て」という考えは誤解を生みやすい。正しくは「状態を共有する最小のスコープの最も近くに置け」だ。

ページ全体で共有する必要のない局所的な状態は、そのWidgetの中に閉じ込める。そうすれば再ビルドの範囲が自然と狭まる。

### 原則2: Widgetを細かく分割する

大きな`build`メソッドは、再ビルドの範囲が広い。これを細かいWidgetに分割すると、Flutterの差分検出がより精確に機能する。

`const`コンストラクタを使えるWidgetは積極的に`const`にする。`const`ウィジェットは再ビルド時に同一インスタンスとして扱われ、再評価されない。

```dart
// ✅ 変化しない部分はconstにする
const Header(title: 'マイアプリ'),
```

### 原則3: RepaintBoundaryで描画の境界を設ける

`RepaintBoundary`でWidgetを包むと、そのサブツリーが独立したレイヤーで描画される。アニメーションや頻繁に変化するWidgetに使うと、他の部分の再描画を防げる。

ただし、`RepaintBoundary`は多用するとメモリ使用量が増えるトレードオフがある。本当に重い再描画が起きている箇所にのみ適用するのが良い。

## 状態管理ライブラリとの関係

RiverpodやBlocなどの状態管理ライブラリを使えば自動的に再ビルドが最適化される——と思っているとつまずく。

ライブラリは「必要なProviderやStreamの変化のみをリッスンするWidgetだけを再ビルドする」仕組みを提供するが、その設計は開発者次第だ。

**Riverpodでの例**: 大きなStateクラスを一つのProviderで管理し、その全体をwatchすれば、任意のプロパティが変化するたびに全watcherが再ビルドされる。

```dart
// ❌ 全体をwatchしていると細かい変化でも全Consumerが再ビルドされる
final appStateProvider = StateProvider<AppState>((ref) => AppState());

// ✅ 細かくProviderを分割する
final userNameProvider = Provider<String>((ref) => ...);
final cartItemsProvider = Provider<List<Item>>((ref) => ...);
```

Providerの粒度を細かくし、Consumerが本当に必要な情報だけをwatchする設計が、Riverpodを使う上での肝だ。

## Flutterのパフォーマンスデバッグツール

設計の良し悪しは実際に計測して確認することが大切だ。Flutterには強力なデバッグツールが揃っている。

### Widget Inspector の Rebuild Counts

Flutter DevToolsのWidget InspectorにはRebuild Countsを表示する機能がある。どのWidgetが何回再ビルドされているかが一目瞭然だ。意図せず高頻度で再ビルドされているWidgetをすぐに発見できる。

### Performance Overlay

`MaterialApp`に`showPerformanceOverlay: true`を設定すると、UI threadとGPU threadの負荷をリアルタイムで表示できる。ジャンクフレームの発生源を特定するのに役立つ。

### Timeline Events

`debugPrintRebuildDirtyWidgets = true`をメインのbuildの前に設定すると、コンソールにどのWidgetが再ビルドされたかを出力できる。開発中の調査に便利だ。

## StatelessとStatefulの使い分けの本質

`StatefulWidget`は「ライフサイクルを持ち、状態を変更できる」Widgetだ。しかし多くの場面で、本当にStatefulである必要はない。

外部から受け取るデータを表示するだけのWidgetは`StatelessWidget`で十分だ。StatelessはStatefulより軽量で、Flutterのフレームワークも最適化しやすい。

「なんとなくStatefulにしてしまった」Widgetが積み重なると、アプリ全体の再ビルドコストが増大する。Widgetを作るたびに「これは本当に状態を持つ必要があるか？」と問いかける習慣が重要だ。

---

## カスタムウィジェットの設計指針

大規模なFlutterアプリを維持するために、カスタムWidgetの設計で守りたいことがある。

**小さく、単一責任に保つ**
一つのWidgetが複数の責任を持つと、再ビルドの影響範囲が広がり、テストも難しくなる。コンポーネントの分割はUIだけでなくパフォーマンスとテスタビリティに直結する。

**ビジネスロジックをWidgetから分離する**
`build`メソッドの中でデータ加工や判定を行うと、毎再ビルドのたびにそのロジックが実行される。ロジックはViewModelやNotifier、PresenterなどのUIと分離したレイヤーに置く。

**Keyを適切に使う**
リスト内のWidgetには適切な`Key`を設定することで、Flutterが差分検出を正確に行えるようになる。特に並び替えや削除が発生するリストでは、`Key`がないとアニメーションのバグや不要な再ビルドが起きやすい。

## おわりに

Flutterの再ビルドを完全に排除することはできないし、する必要もない。目標は「不要な再ビルドをなくすこと」だ。

そのためには、Flutterの3ツリーとReactconciliation（差分検出）の仕組みを理解し、状態の配置場所を設計段階から考えることが大切だ。

`const`を活用し、Widgetを小さく分割し、状態は使う場所に近く置く——これらはコードを書く上での基本習慣として身につけると、パフォーマンス問題のほとんどは予防できる。

「遅くなったら最適化する」より「最初から再ビルドの影響範囲を意識する」方が、長期的にはよりスムーズな開発体験につながる。
