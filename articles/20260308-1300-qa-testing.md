## テストケースは「あなたが想像できる範囲」でしか書けない

ユニットテストを書くとき、私たちは「どんな入力でどんな出力を期待するか」という形でテストケースを積み重ねていく。これは「例ベーステスト（Example-based Testing）」と呼ばれるアプローチだ。

このアプローチには根本的な限界がある。テストケースは書いた人間が「想像できた範囲」でしか網羅できない。バグが潜むのはしばしば、開発者が「こんな入力は来ないだろう」と想定しなかった場所だ。

プロパティベーステスト（Property-based Testing）は、この限界を突き破るための発想だ。「具体的な入力と期待値を人間が指定する」のではなく、「満たすべき性質（プロパティ）を定義し、入力はコンピュータが自動生成する」というアプローチを取る。

## 「プロパティ」とは何か

プロパティとは、「どんな入力に対しても成り立つはずの性質」のことだ。

例えば配列のソート関数を考えてみよう。「ある特定の配列をソートすると期待する並び順になる」という例ベーステストに対し、プロパティベーステストでは次のような性質を定義する。

- ソート後の配列の要素数は元の配列と同じ
- ソート後の配列は昇順（または降順）になっている
- ソート後の配列に含まれる要素の種類と数は元の配列と同じ（集合として等価）
- すでにソートされた配列に対してソートを実行しても結果は変わらない（冪等性）

これらの性質は、「[3, 1, 2]を並べ替えると[1, 2, 3]になる」という1つの例よりも、はるかに広い範囲の入力に対する正しさを記述している。

```typescript
import fc from "fast-check";

// プロパティ: ソート後は要素数が変わらない
test("ソート後も要素数は変わらない", () => {
  fc.assert(
    fc.property(fc.array(fc.integer()), (arr) => {
      const sorted = [...arr].sort((a, b) => a - b);
      return sorted.length === arr.length;
    })
  );
});
```

`fast-check`（JavaScriptのPBTライブラリ）の `fc.assert` と `fc.property` を組み合わせると、`fc.array(fc.integer())` が自動的に何百もの異なる整数配列を生成し、定義したプロパティを検証してくれる。

## なぜ「想定外のバグ」を見つけられるのか

ランダムな入力生成とセットで語られる重要な概念が「シュリンキング（Shrinking）」だ。

テストがある入力で失敗したとき、そのままでは失敗した入力が「なぜ失敗したのか」を理解しにくい大量のデータである場合が多い。シュリンキングは、失敗を引き起こした最小の入力例を自動的に探し出す仕組みだ。

例えば、1000要素の配列でバグが見つかったとき、シュリンキングは「この配列の最小のサブセットで同じ失敗が再現できるか？」を繰り返し試みる。その結果、「空の配列」「要素が1つの配列」「特定の値の組み合わせ」といった最小再現ケースを提示してくれる。

これが開発者にとって非常に有益だ。「なぜ失敗したか」が明確な、最小の反例を自動的に手に入れられる。

## 例ベーステストとの役割分担

プロパティベーステストは、例ベーステストを「置き換える」ものではない。両者は補完的な関係にある。

**例ベーステストが得意なこと：**
- 明確なビジネスルールや仕様を直接テストする
- バグの修正後に「この特定のケースが直ったか」を確認する回帰テスト
- ドキュメンテーションとしての役割（どんな入力でどんな出力かが一目でわかる）

**プロパティベーステストが得意なこと：**
- 「自分が思いつかなかった入力」でのバグ発見
- データ変換・アルゴリズム・パース処理などの正しさの検証
- 不変条件（invariant）の検証

実用的な設計としては、まず例ベーステストで主要なビジネスロジックをカバーし、データ変換や複雑なアルゴリズムにはプロパティベーステストを追加する形が多い。

## プロパティの見つけ方：典型的なパターン

プロパティを定義するのが難しいと感じる場合、以下のパターンを参考にすると発見しやすい。

**往復変換（Round-trip）**：`encode → decode` や `serialize → deserialize` のような変換では、「元のデータに戻るか」というプロパティが使える。

**冪等性（Idempotency）**：「同じ操作を2回適用しても1回と結果が同じ」という性質。ソートや正規化処理などに有効。

**交換法則・結合法則**：数学的な性質を持つ処理では、順序を変えても結果が同じかどうかを検証できる。

**オラクルとの比較**：最適化実装と単純な参照実装（oracle）を並置し、「同じ入力に対して同じ結果を返すか」を比較する。

**不変条件**：「残高が負にならない」「ユーザーIDが重複しない」といったドメイン上の不変条件を検証する。

## 実際に使えるライブラリ

JavaScriptとTypeScriptのエコシステムでは `fast-check` が最もよく使われる。Haskellの `QuickCheck`（PBTを広めた元祖）の思想を忠実に継承しており、シュリンキングや豊富なジェネレータが揃っている。

Pythonには `Hypothesis`、Java/Kotlinには `jqwik`、Goには `gopter` といったライブラリがある。どの言語でも基本的な考え方は共通しているため、一つを覚えると他の言語でも応用しやすい。

## QAプロセスへの組み込み方

プロパティベーステストをCIパイプラインに組み込む際には、1点注意が必要だ。デフォルト設定では何百回ものランダム入力を試すため、実行時間がかかる。

対策として：

- **CI環境では試行回数を制限する**：開発中は100回、CIでは50回といった形で環境に応じて調整する
- **失敗した例をシードとして保存する**：一度失敗した入力例を固定して回帰テストに加える（多くのライブラリがこの機能をサポートしている）
- **速いテストスイートとは分離する**：PBTは別のテストスイートとして定期実行（夜間バッチなど）に回す選択肢もある

## テストへの「不信任決議」として使う

プロパティベーステストの本質は、「開発者自身の思い込みに対する挑戦」だ。

「このコードはどんな入力でも正しく動く」という確信があるとき、その確信を自動化されたランダム入力でテストする。失敗したときに初めて「自分が見逃していた前提条件」に気づける。

例ベーステストは「仕様を文書化する」ツールとして機能する。プロパティベーステストは「自分の理解を検証する」ツールとして機能する。両者を組み合わせたとき、テストスイートは単なるドキュメント以上の価値を持ち始める。

「バグは想定外の場所に潜む」という現実を受け入れ、その探索をコンピュータに委ねるという発想——プロパティベーステストの真の価値はここにある。
