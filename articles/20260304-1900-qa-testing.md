## はじめに

「本番環境で意図的に障害を起こす」——そう聞くと無謀に聞こえるかもしれない。しかしカオスエンジニアリングは、まさにその逆説的な行為によってシステムの信頼性を高める工学的アプローチだ。

NetflixがChaos Monkeyを公開して以来、この思想はSREやプラットフォームエンジニアリングの文脈で広まっている。しかし「障害を入れる手法」として捉えるだけでは、その本質を見誤る。本稿ではカオスエンジニアリングの設計哲学を整理する。

## 従来のテストとの根本的な違い

従来のテストは「期待した動作をすること」を検証する。ユニットテストは関数の正しさを、E2Eテストはユーザーフローの成立を確かめる。これらは**既知の仮定を検証する**活動だ。

カオスエンジニアリングは異なる問いを立てる：**「想定外のことが起きたとき、システムはどう振る舞うか？」**

これは「バグを見つけること」ではなく、「システムの弱点と回復力を発見すること」だ。正しく動くかどうかではなく、壊れ方と回復のしかたを知ることが目的になる。

分散システムにおいては「完全に正常な状態」というものは幻想に近い。ネットワークは断絶し、依存サービスは遅延し、ディスクは一杯になる。問題は「障害が起きるかどうか」ではなく「障害が起きたときにどれだけ優雅に対処できるか」だ。

## カオス実験の設計思想

カオスエンジニアリングを実践する際に重要なのは、「ランダムに壊す」のではなく**仮説に基づいた実験**として設計することだ。

Chaos Monkeyが有名になったため「インスタンスをランダムに落とすツール」というイメージが定着しているが、Netflixが強調しているのはむしろ科学的な手法だ。

実験は次のステップで設計する：

1. **定常状態の定義**：システムが正常に動作しているとき、どのメトリクスがどの範囲にあるか。レイテンシ、エラーレート、スループットなどの基準を明確にする。

2. **仮説の立案**：「依存するマイクロサービスAが50msの遅延を抱えても、ユーザーへのレスポンスタイムは500ms以内を維持するだろう」という形で仮説を立てる。

3. **最小限の「乱流」を注入**：仮説を検証するために必要な最小限の障害だけを注入する。

4. **定常状態からの逸脱を観測**：仮説通りの挙動をしたか。予期しない副作用が起きたか。

5. **実験結果から学ぶ**：成功しても失敗しても、システムについての知識が増える。

この手順は科学的実験と同じ構造を持つ。「壊す」のが目的ではなく「学ぶ」のが目的だ。

## 注入すべき「乱流」の種類

カオス実験で注入する障害は大きく4つに分類できる。

**リソース枯渇**：CPU使用率を意図的に高める、メモリを消費させる、ディスクをフルにする。スロットリングやOOMキラーの動作を確認できる。

**ネットワーク障害**：レイテンシの追加、パケットロス、帯域制限、特定のエンドポイントとの接続断。マイクロサービス間の依存性と回路遮断器（Circuit Breaker）の動作を検証できる。

**インフラ障害**：インスタンスの強制停止、コンテナのkill、アベイラビリティゾーンの切り離し。オートスケールやフェイルオーバーの実効性を確かめる。

**アプリケーション層の障害**：特定のAPIに意図的に異常なレスポンスを返させる、タイムアウトを模倣する。依存コンポーネントが正しくフォールバックするかを検証する。

```bash
# tc (traffic control) による遅延注入の例
tc qdisc add dev eth0 root netem delay 200ms 50ms
```

このように、カオスツールはインフラレベルのものからアプリケーション層まで多岐にわたる。Chaos MeshやLitmus、AWS Fault Injection Simulatorなど、管理されたカオス注入を支援するツールが整備されている。

## 「安全なカオス」の実現

本番でいきなりカオス実験を行うことはリスクが高い。実践的には段階的なアプローチが重要だ。

**まずはステージング環境で**：本番に近い構成のステージング環境でカオス実験を行い、実験設計とモニタリングの方法を磨く。

**ゲームデイの実施**：チーム全員が揃った「ゲームデイ」と呼ばれる演習を計画的に実施する。障害を注入し、チームが実際にどう対応するかを練習する。非技術的な学習——コミュニケーションフローや意思決定プロセス——も得られる。

**爆発半径の制御**：最初は影響範囲を最小限に絞る。単一のインスタンス、単一のリージョン、一部のユーザーにのみ影響するよう設計する。

**ロールバック計画の準備**：実験を即座に停止できるkill switchを必ず用意する。実験が想定外の方向に進んだら、すぐに中断できる体制が前提だ。

## カオスエンジニアリングが明らかにするもの

実際にカオス実験を行うと、テストでは見つかりにくかった問題が浮かび上がる。

**Circuit Breakerの設定ミス**：依存サービスが落ちても回路が開かず、タイムアウトが連鎖してスレッドプールが枯渇する。

**タイムアウト設定の不備**：HTTPクライアントにタイムアウトが設定されていない、または不適切に長い。

**リトライの相互作用**：リトライロジックが複数のレイヤーに存在し、障害時に指数的な負荷が発生する（Thundering Herd問題）。

**ステートフルな依存性**：「ステートレスな設計」のつもりだったが、特定のセッションストアやキャッシュに依存していた。

これらは「正常系のテスト」では発見できない。カオスエンジニアリングはシステムの暗黙の仮定を明示化する道具だ。

## QA思想としての位置づけ

カオスエンジニアリングを「SREの道具」として捉えがちだが、QAの観点からも重要な意味を持つ。

従来のQAは「機能が正しく動くか」を問う。カオスエンジニアリングは「システム全体の振る舞いが期待通りか」を問う。この違いは、単体機能のテストと非機能要件（可用性・耐障害性・回復性）のテストの違いに対応する。

「99.9%の可用性を担保する」という要件は、テスト仕様書に落とすのが難しい。しかしカオスエンジニアリングの実験結果は、その要件が実際に満たされているかどうかを具体的に示す。

## おわりに

カオスエンジニアリングの本質は「壊すこと」ではなく、**「壊れ方を知ること」**にある。システムが壊れるタイミング、壊れ方、回復のスピードを理解することで、真の意味での信頼性を高めることができる。

「障害は起こらないもの」という前提を「障害は必ず起こるもの」という前提に変えたとき、設計の優先順位は変わる。カオスエンジニアリングはその前提転換を体験から学ぶための実践的な方法論だ。
