## ファイル構造は設計の言語だ

Next.js App Routerでは、`app/` ディレクトリ配下のフォルダ構造がそのままURLのルートに対応する。これは直感的で分かりやすいが、アプリケーションが成長するにつれ「URLの都合」と「UIの都合」が一致しなくなることがある。

例えば、`/dashboard`、`/settings`、`/profile` は異なるURLだが、いずれも認証済みユーザーだけが見る画面で、同じサイドバーレイアウトを共有したい。しかし一方、`/login` や `/signup` は全く異なるレイアウトで、サイドバーは不要だ。

この問題を解決するために、App RouterはRoute Groupsという仕組みを提供している。本稿では、Route GroupsとLayout設計の組み合わせが、どのようにコードの意図を明確にし、メンテナンス性を高めるかを考えていく。

## Route Groupsとは何か

Route Groupsは、フォルダ名を括弧で囲む `(folderName)` という記法によって作られる。括弧付きのフォルダはURLパスには影響しない。つまり、ルーティング上は「存在しない」フォルダとして扱われる。

```
app/
├── (auth)/
│   ├── login/
│   │   └── page.tsx
│   └── signup/
│       └── page.tsx
└── (dashboard)/
    ├── layout.tsx
    ├── dashboard/
    │   └── page.tsx
    └── settings/
        └── page.tsx
```

この構造では、`/login`、`/signup`、`/dashboard`、`/settings` というURLはすべてそのままだが、内部的には `(auth)` グループと `(dashboard)` グループに分けて管理できる。

## Layoutの継承とRoute Groupsの関係

App Routerにおけるレイアウトの仕組みは「入れ子（ネスト）構造」を基本としている。親ディレクトリのlayout.tsxは、そのディレクトリ配下のすべてのページに適用される。

Route Groupsを使うと、この継承構造をURLと分離して設計できる。`(dashboard)/layout.tsx` は、`/dashboard` や `/settings` というURL のページに対してサイドバーやヘッダーを提供しながら、`(auth)` 配下のページには一切影響しない。

```tsx
// (dashboard)/layout.tsx
export default function DashboardLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  return (
    <div className="flex h-screen">
      <Sidebar />
      <main className="flex-1 overflow-y-auto p-6">{children}</main>
    </div>
  );
}
```

このファイルを見た開発者は、「ここに書かれたレイアウトは、(dashboard)グループのページにのみ適用される」という意図を即座に読み取れる。ファイル構造が設計意図を語っている状態だ。

## 複数の`layout.tsx`と`template.tsx`の使い分け

App Routerには `layout.tsx` と `template.tsx` という2種類のファイルがある。

**layout.tsx** はページ遷移をまたいでも状態を保持するレイアウトだ。サイドバーのスクロール位置やドロップダウンの開閉状態が、ページ遷移後も維持される。ユーザーが同じセクション内でナビゲートする場合の体験として自然だ。

**template.tsx** はページ遷移のたびに新しいインスタンスが作られる。つまり状態はリセットされる。アニメーションをページ遷移ごとに発火させたい場合や、特定のページ間では状態を引き継がせたくない場合に有効だ。

この選択はUIの振る舞いの設計に直結する。「このレイアウトのUI状態をどこまで保持するか」を明示的に表現できる。

## Parallel RoutesとIntercepting Routes——さらに複雑な要件へ

Route Groupsを理解すると、App Routerのより高度な機能——Parallel Routes（`@folder` 記法）やIntercepting Routes（`(...)folder` 記法）の設計が見えてくる。

**Parallel Routes** は、同一ページ内に複数の「スロット」を持たせる仕組みだ。ダッシュボードのメインコンテンツと右サイドのパネルを、それぞれ独立したルートとして管理できる。条件によって異なるコンポーネントを差し込むモーダルや、サイドバーとメインエリアが独立してナビゲートできるレイアウトに使われる。

**Intercepting Routes** は、別のURLへの遷移を「横取り」して、モーダルとして表示する仕組みだ。Instagramの投稿をフィード上でクリックするとモーダルで開き、URLは投稿ページのURLに変わる——というUXを実現する際に使われる。URLは正規のページに変わるが、UIとしてはモーダルで見せるという体験設計が可能になる。

## 設計のアンチパターン：構造のために構造を作る

Route GroupsやLayoutの機能が増えると、「機能的に可能だから複雑な構造を作る」というアンチパターンに陥りやすい。

以下のサインが見えたら、設計を見直す時機だ。

- Route Groupsのネストが3段以上になっている
- `layout.tsx` がページのほぼすべてをラップしているが、中身がほとんど `{children}` だけ
- どのlayoutがどのページに適用されているかを把握するのに時間がかかる

シンプルな原則は「レイアウトの分割は、UIの明確な違いがある場所でだけ行う」だ。認証の有無、ナビゲーションパターンの違い、ユーザーの役割（管理者 vs 一般ユーザー）といった実質的な違いがある場合にRoute Groupsで分割し、そうでなければフラットな構造を維持する方がシンプルで読みやすい。

## ファイル構造をコミュニケーションツールとして使う

最終的に、Route GroupsとLayout設計の本質はコードの「意図の表現」にある。

ファイル構造を見たとき、チームメンバーが「なぜここにlayout.tsxがあるのか」「このRoute Groupはどのような意図で切られているのか」を、コードを読まずに理解できる状態が理想だ。

App Routerが提供するこれらの仕組みは、URLという技術的制約とUIという体験設計を分離し、それぞれの都合に応じて整理する力を与えてくれる。ファイルの置き場所が、設計の言語として機能するとき——コードベースは小さなチームでも大きなチームでも、同じ速度で前進できるようになる。
