## Flutterが「クロスプラットフォーム」であることの意味

Flutterは単一のDartコードでiOS・Android・Web・デスクトップのUIを構築できる。しかし「UI以外の機能」——カメラ、Bluetooth、バックグラウンド処理、OS固有のAPI——はプラットフォームごとにネイティブコードで実装されている現実がある。

Flutterエコシステムにはpub.devに大量のプラグインが存在し、多くの場合はそれを使えば済む。しかしプラグインが存在しない機能を使いたいとき、既存プラグインの動作を自分のアプリに合わせてカスタマイズしたいとき、またはパフォーマンスやセキュリティの理由でネイティブの実装を直接制御したいとき——Platform Channelを理解していないと対応できなくなる。

本記事では、Platform Channelの仕組みと設計の考え方を整理する。

## Platform Channelの仕組み

FlutterのDart層とネイティブ層（iOS: Swift/Objective-C、Android: Kotlin/Java）は別のランタイムで動く。この2つをつなぐ通信基盤がPlatform Channelだ。

Flutterのアーキテクチャには「エンジン」と呼ばれるC++の中間層があり、Platform Channelはこのエンジンを介してメッセージを非同期でやり取りする。DartとネイティブコードはJSON互換のデータ（文字列、数値、リスト、マップ）を送受信する形で連携する。

Flutterが提供するChannelは3種類ある。

**MethodChannel**: 最もよく使われる。Dart側からメソッド名と引数を送り、ネイティブ側で処理して結果を返す双方向の非同期呼び出し。

**EventChannel**: ネイティブ側からDart側にストリームでデータを流し続ける用途。センサーの値やBluetooth接続状態の変化など、継続的なイベント通知に使う。

**BasicMessageChannel**: 双方向のメッセージパッシング。構造化データのやり取りに使えるが、MethodChannelとEventChannelで賄えない特殊なケース向け。

## MethodChannelの基本的な使い方

最もシンプルなMethodChannelの例として、「デバイスの温度を取得する機能」を実装する場面を考える（pub.devにちょうどいいプラグインがないとする）。

Dart側では、`MethodChannel`を定義してネイティブのメソッドを呼び出す。

```dart
class DeviceInfoService {
  static const _channel = MethodChannel('com.example.myapp/device_info');

  Future<double?> getBatteryTemperature() async {
    try {
      final temperature = await _channel.invokeMethod<double>('getBatteryTemperature');
      return temperature;
    } on PlatformException catch (e) {
      debugPrint('Failed to get temperature: ${e.message}');
      return null;
    }
  }
}
```

チャンネル名は逆ドメイン記法（com.example.myapp/channel_name）で一意にする慣習がある。複数チームが開発するアプリでは命名規則を統一しておかないと衝突が起きやすい。

Android側（Kotlin）では、`MainActivity.kt`またはPlugin内でChannelの受け口を実装する。

```kotlin
class MainActivity: FlutterActivity() {
  override fun configureFlutterEngine(flutterEngine: FlutterEngine) {
    super.configureFlutterEngine(flutterEngine)
    MethodChannel(flutterEngine.dartExecutor.binaryMessenger, "com.example.myapp/device_info")
      .setMethodCallHandler { call, result ->
        when (call.method) {
          "getBatteryTemperature" -> {
            val temperature = getBatteryTemperatureFromSystem()
            if (temperature != null) result.success(temperature)
            else result.error("UNAVAILABLE", "Temperature not available", null)
          }
          else -> result.notImplemented()
        }
      }
  }
}
```

## 設計上の注意点

### チャンネルを薄いインターフェースに保つ

Platform Channelは「DartとネイティブのAPIをつなぐ薄い橋」として設計するのが原則だ。ビジネスロジックをネイティブ側に書くと、そのコードはFlutter側からテストできなくなる。「ネイティブ側にしかできない処理」だけをネイティブ側に置き、ロジックはDart側に寄せる。

例えば「カメラで撮影してOCRを実行し、テキストを返す」という機能を一つのChannelメソッドとして実装するのは避けたほうがいい。「撮影して画像データを返す」と「画像データからテキストを抽出する」を分離すれば、OCRの実装を将来別のAPIに差し替えやすくなる。

### エラーハンドリングを明示的に設計する

`PlatformException`はネイティブ側が`result.error()`を呼んだときに発生する。コード、メッセージ、詳細情報の3つのフィールドを持つが、エラーコードの体系を事前に設計しないと、Dart側でエラーの種類を判別できなくなる。

ネイティブ側のエラーコードをenumや定数として定義し、Dart側でも同じ定数を参照する形にすることで、プラットフォーム間のエラーインターフェースを一貫させられる。

### メインスレッドの扱いに注意する

iOS/Androidともに、UI操作はメインスレッドでしか行えないというルールがある。Platform Channelのコールバックはメインスレッドで実行されるため、長時間かかる処理（ファイルI/O、ネットワーク通信）をChannelのハンドラ内で同期的に実行するとUIがフリーズする。

ネイティブ側では`DispatchQueue.global().async`（iOS）や`AsyncTask`/Coroutine（Android）を使い、処理を非同期にしてから`result.success()`を呼ぶ設計が必要だ。

## FlutterPlugin化のタイミング

アプリ内でPlatform Channelの実装が複雑になってきたとき、またはほかのプロジェクトでも再利用したいとき、Flutterのプラグイン構造（flutter_plugin template）に移行することを検討する。

プラグインはDart API、iOS実装（Swift）、Android実装（Kotlin）を一つのパッケージとして管理する構造だ。公開しない場合でも、プラグイン構造にすることでDart層とネイティブ層の責任が明確に分離され、テストもしやすくなる。

`flutter create --template=plugin my_plugin`でプラグインのスケルトンが生成される。

## FideMethodChannelとモックテスト

Dart側のテストでは、実際にネイティブを呼び出さずにChannelのレスポンスをモックしたい。`TestDefaultBinaryMessengerBinding.instance.defaultBinaryMessenger.setMockMethodCallHandler`を使うと、MethodChannelに対するモックレスポンスを設定できる。

```dart
TestDefaultBinaryMessengerBinding.instance.defaultBinaryMessenger
    .setMockMethodCallHandler(
  const MethodChannel('com.example.myapp/device_info'),
  (MethodCall methodCall) async {
    if (methodCall.method == 'getBatteryTemperature') {
      return 35.5;
    }
    return null;
  },
);
```

このアプローチで、Dart側のロジックをネイティブ実行環境なしにunit testできる。Platform Channelを薄いインターフェースに保つ設計の重要性がここでも現れる。ビジネスロジックがネイティブに入り込むほど、テスト可能な範囲が狭まる。

## pub.devのプラグインをラップする場合

既存プラグインを使いつつ、アプリ固有の設定やエラーハンドリングを追加したい場合、プラグインをそのまま使わずに薄いラッパーを挟む設計が有効だ。これはDIの観点でも重要で、プラグインへの依存を直接アプリコードに散在させると、プラグインを差し替えるときに影響範囲が広がる。

## まとめ

Platform Channelは、FlutterのDart世界とプラットフォームのネイティブ世界をつなぐ橋だ。基本的なMethodChannelの使い方はシンプルだが、設計の質はチャンネルを「薄いインターフェースに保つか」「エラーコードを体系化しているか」「非同期処理を正しく扱っているか」で決まる。pub.devのプラグインで足りないとき、または既存プラグインをより細かく制御したいときにPlatform Channelの知識が直接役立つ。ネイティブとDartの境界をどこに引くかという判断が、アプリ全体の保守性に影響する。
