## モックは「嘘」を教える

テストを書くとき、私たちはしばしばこんな判断をする。「DBはモックにしよう。外部APIは差し替えよう。Redisはインメモリで代用しよう」——。

これは悪い判断ではない。テストが速く、壊れにくくなる。しかし、モックには見えにくいコストがある。

**モックは本物ではない。**

モックしたPostgresは、本物のPostgresが返す微妙なエラーコードを再現しない。モックしたRedisは、`EXPIRE`コマンドの挙動のエッジケースを知らない。モックしたKafkaは、パーティションのオフセット管理の複雑さを持たない。

結果として、テストはグリーンなのに本番で壊れる——という事態が起きる。これは「モックが悪い」というより、「モックが表現できる範囲を超えた問題をテストしていた」ということだ。

Testcontainersは、この問題に対する一つの答えを提示する。「モックをやめて、本物の依存関係をコンテナで使え」という発想だ。

---

## Testcontainersとは何か

TestcontainersはJVM向けに生まれたライブラリで、今ではGo・Python・Rust・.NET・Node.jsなど多くの言語に移植されている。テスト実行時にDockerコンテナを起動し、テスト完了後に自動的に破棄する。

コードで書くとこうなる（Node.js版の例）：

```typescript
import { PostgreSqlContainer } from "@testcontainers/postgresql";
import { drizzle } from "drizzle-orm/node-postgres";

describe("UserRepository", () => {
  let container: StartedPostgreSqlContainer;
  let db: ReturnType<typeof drizzle>;

  beforeAll(async () => {
    container = await new PostgreSqlContainer("postgres:16-alpine").start();
    db = drizzle(container.getConnectionUri());
    await runMigrations(db);
  });

  afterAll(async () => {
    await container.stop();
  });

  it("ユーザーを保存して取得できる", async () => {
    const repo = new UserRepository(db);
    await repo.save({ id: "u1", name: "田中" });
    const user = await repo.findById("u1");
    expect(user?.name).toBe("田中");
  });
});
```

これだけで、テストごとに本物のPostgreSQLが起動・停止する。`pg-mem`のようなインメモリ代替ではなく、本物のPostgres 16が動く。

## モックと本物の間にある「信頼のギャップ」

テスト戦略を考えるとき、よくテストピラミッドが引用される。ユニットテストを多く、E2Eテストを少なく、統合テストをその中間に——という構造だ。

しかしこのピラミッドが前提としているのは、「各レイヤーが正しく機能する」という信頼だ。ユニットテストが「このメソッドはこう動く」を保証し、統合テストが「これらのコンポーネントが正しく連携する」を保証する。

問題は、統合テストでモックを使いすぎると、その「連携」の部分が十分にテストされないことだ。

たとえば、こんなケースを考えてほしい：

- PostgreSQLのJSON列に対して特定のクエリを書いた
- ユニットテスト（モックあり）：パス
- 本番デプロイ後：構文エラーでクラッシュ

原因は、使ったSQLがPostgreSQL 14では動くが、本番で使っているPostgreSQL 12では動かなかった——というバージョン差異だった。モックはバージョンの違いを知らない。本物のコンテナは知っている。

Testcontainersは「ユニットテストとE2Eの間のギャップ」を埋める道具として機能する。

## 設計思想：速さと信頼性のトレードオフを正しく見積もる

Testcontainersを使う上で最初にぶつかる疑問は「遅くなるのでは？」だ。

確かに、コンテナの起動には数秒かかる。しかしこの「コスト」を正しく見積もる必要がある。

**コンテナを再利用する**

Testcontainersには`Reuse`機能がある。テスト実行間でコンテナを再利用することで、起動オーバーヘッドを一度だけにできる。ローカル開発では特に効果的だ。

**並列実行との組み合わせ**

各テストスイートが独立したコンテナを持つ設計にすれば、テストスイートを並列実行できる。合計時間は増えなくても、ウォールクロック時間は短縮される。

**「どこまでモックするか」を意識的に決める**

全ての依存関係をTestcontainersで置き換える必要はない。「本物との差異がバグを生む可能性が高いもの」を優先する。具体的には：

- DBクエリが複雑なもの（JOIN、CTE、プロシージャなど）
- バージョン依存の挙動があるもの（RedisのACL、Elasticsearchのマッピングなど）
- 副作用が外部サービスに及ぶもの（Kafka、S3互換ストレージなど）

逆に、シンプルなCRUDのみで差異が生じにくい部分は、引き続きモックで十分なことが多い。

## 実践的なパターン：Shared ContainerとTest-per-Container

テストの設計において、コンテナ共有の単位が重要になる。

**Shared Container パターン**

テストスイート全体で1つのコンテナを共有する。高速だが、テスト間でデータが汚染するリスクがある。各テストが適切なセットアップ・クリーンアップを行う規律が求められる。

**Test-per-Container パターン**

テストごとに独立したコンテナを起動する。遅いが、テスト間の独立性が完全に保たれる。CIでの並列実行と組み合わせると現実的な速度になる。

多くのプロジェクトでは、「テストスイート単位でコンテナを共有し、各テスト実行前にトランザクションをロールバック（あるいはテーブルをTRUNCATEする）」というミドルウェアが最もバランスが良い。

## CI/CDとの統合

Testcontainersの大きな強みは、CI環境でもそのまま動く点だ。GitHubActions、CircleCI、GitLab CIなど、DockerをサポートするCI環境であれば追加設定なしに動作する。

ただし、いくつかの注意点がある。

**Dockerデーモンの確認**

CI環境によっては、Docker-in-Docker（DinD）設定が必要な場合がある。GitHub ActionsのUbuntuランナーはデフォルトでDockerが使えるため、ほとんどの場合は問題ない。

**イメージのキャッシュ**

コンテナイメージのプル時間を短縮するため、CI側でDockerレイヤーキャッシュを設定する。GitHubActionsでは`docker/build-push-action`のキャッシュ設定が参考になる。

**並列実行とポートの競合**

複数のジョブが同時に実行される場合、Testcontainersはランダムなポートをバインドするため、ポート競合は基本的に発生しない。この点はモックベースのテストと比較したときの優位点でもある。

---

## いつTestcontainersを選ぶか

Testcontainersはすべてのプロジェクトに必要ではない。選択のガイドラインを整理しておく。

**使う価値が高いとき：**

- SQLの複雑さが高く、ORM抽象化だけでは不安なとき
- 本番DBのバージョンや設定が重要なとき（たとえばPostgreSQLの`pg_trgm`拡張を使う場合）
- メッセージブローカー（Kafka、RabbitMQ）の動作をテストしたいとき
- 外部サービスのローカル版（MinIO、Wiremockなど）を使うとき

**モックで十分なとき：**

- 依存関係が標準的なCRUDのみで、差異が生じにくい
- テストの速度が最優先で、コンテナ起動のオーバーヘッドが許容できない
- 依存関係のAPIが安定していて、ライブラリのモックが十分に信頼できる

---

## まとめ：「本物に近い」ことが信頼を生む

テストは「コードが動く証拠」だ。しかしモックが多ければ多いほど、その「証拠」は実態から離れていく。

Testcontainersは、テストの「信頼半径」を広げるツールだ。ユニットテストの速さとE2Eテストの現実性の間で、現実的なコストで統合テストを書けるようにする。

全てをTestcontainersに置き換える必要はない。しかし「ここのテストはモックで本当に信頼できるか？」と問い直すきっかけとして、Testcontainersの存在を知っておくことは価値がある。

本物の依存関係は、本物の信頼を生む。それがTestcontainersの根本にある設計思想だ。
