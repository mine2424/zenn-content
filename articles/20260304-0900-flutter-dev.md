## はじめに

Flutterアプリを複数人で開発していると、「誰のマシンでもビルドが通るか」「リリースのたびに手動でApp Storeにアップロードしているか」という問題が出てくる。

手動リリースは属人化の温床だ。特定のメンバーしかリリースできない状態は、チームのボトルネックになるだけでなく、ヒューマンエラーのリスクも高い。

CI/CDを整備することで、「コードをプッシュすれば自動でビルド・テスト・配布まで走る」という状態を作れる。この記事では、GitHub ActionsとFastlaneを組み合わせたFlutterのCI/CDパイプライン設計の考え方を整理する。

---

## CI/CDの責務を分けて考える

CI（継続的インテグレーション）とCD（継続的デリバリー/デプロイメント）は異なる責務を持つ。

**CI（継続的インテグレーション）の役割：**
- コードの正しさを検証する（テスト・静的解析）
- ビルドが通ることを確認する
- すべてのPRで実行する

**CD（継続的デリバリー）の役割：**
- 配布可能なアーティファクト（IPA・APK）を生成する
- ストアやテスト配布環境にアップロードする
- mainブランチへのマージやタグ作成をトリガーにする

この責務の分離が設計の出発点だ。CIとCDを同じワークフローに詰め込むと、PRのたびに署名処理や配布が走るという無駄が生じる。

## GitHub Actionsのワークフロー設計

FlutterのCIは、大きく以下のジョブで構成する。

```yaml
# .github/workflows/ci.yml
name: CI

on:
  pull_request:
    branches: [main, develop]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: subosito/flutter-action@v2
        with:
          flutter-version: '3.27.0'
          cache: true

      - name: Install dependencies
        run: flutter pub get

      - name: Analyze
        run: flutter analyze

      - name: Test
        run: flutter test --coverage
```

`subosito/flutter-action` はFlutterのセットアップを自動化してくれるアクションで、`cache: true` にするとFlutter SDKのダウンロードをキャッシュし、実行時間を大幅に短縮できる。

iOSのビルドには `macos-latest` ランナーが必要になる。Androidのビルドは `ubuntu-latest` で動くが、iOSは macOS でしかビルドできないという制約がある。この非対称性がFlutterのCI設計を少し複雑にする要因だ。

## Fastlaneの役割

**Fastlane**はモバイルアプリのデプロイ自動化ツールだ。App Store Connectへのアップロード、証明書管理、バージョン番号更新など、手動でやると面倒な作業をスクリプト化できる。

GitHub Actionsがインフラ（どこで実行するか）を担うのに対して、Fastlaneはアプリ固有の作業（何を実行するか）を担う。この役割分担が重要だ。

基本的なFastfileの構造：

```ruby
# fastlane/Fastfile
default_platform(:ios)

platform :ios do
  lane :beta do
    # バージョン番号をCIのビルド番号で更新
    increment_build_number(
      build_number: ENV["GITHUB_RUN_NUMBER"]
    )

    # ビルド
    build_app(
      scheme: "Runner",
      export_method: "ad-hoc"
    )

    # Firebase App Distributionに配布
    firebase_app_distribution(
      app: ENV["FIREBASE_APP_ID"],
      groups: "testers"
    )
  end
end
```

Fastlaneの強みは、このレーンを `fastlane ios beta` というコマンド一つで手元でも実行できること。CI環境特有の謎エラーに悩む前に、ローカルで同じ処理を確認できる。

## 証明書管理：matchを使う

iOSの署名処理は、Flutterに関わらずモバイル開発で最も複雑な部分の一つだ。Fastlaneには **match** という証明書管理ツールが含まれており、これを使うことで証明書をGitリポジトリ（またはGoogle Drive）に暗号化して保管し、チームで共有できる。

matchの思想は「証明書は一人が管理するのではなく、チームで共有するリソースにする」だ。

```ruby
lane :setup_certs do
  match(
    type: "adhoc",
    readonly: true,    # CIではreadonly（既存の証明書を使うだけ）
    git_url: ENV["MATCH_GIT_URL"]
  )
end
```

CI環境では `readonly: true` を必ず指定する。CIが証明書を勝手に再生成してしまうと、チームで共有している証明書との整合性が崩れる。

## シークレット管理の考え方

CI/CDで最も気をつけるべきは**シークレット（機密情報）の扱い**だ。

- App Store ConnectのAPIキー
- matchの暗号化パスフレーズ
- Firebase のサービスアカウントJSON

これらをコードに含めることは絶対に避ける。GitHub Actionsでは `Settings > Secrets and variables > Actions` でシークレットを登録し、ワークフロー内で `${{ secrets.MY_SECRET }}` として参照する。

ただし注意点がある。シークレットを `echo` でログに出力したり、アーティファクトに含めてしまうミスが起きやすい。シークレットをファイルとして保存する場合は、ジョブ終了後に必ず削除するステップを追加する。

```yaml
- name: Cleanup
  if: always()   # 失敗時も必ず実行
  run: rm -f secrets.json
```

## ワークフローを段階的に構築する

いきなり完全なCI/CDを作ろうとすると、証明書・環境変数・Fastlaneのデバッグが同時に押し寄せて途方に暮れる。段階的に整備する順序が大切だ。

1. **まずCIだけ**: `flutter analyze` と `flutter test` が自動で走る状態を作る
2. **次にビルド確認**: PRでIPA/APKがビルドできることを確認する（配布はまだしない）
3. **テスト配布を自動化**: mainマージ時にFirebase App Distributionへ配布する
4. **ストア配布を自動化**: タグ作成時にApp Store Connect / Google Play Consoleへ送る

各段階で動く状態を確認しながら進めることで、どこで問題が起きているかを特定しやすくなる。

---

## まとめ

FlutterのCI/CDは「GitHub Actions（インフラ担当）＋Fastlane（アプリ作業担当）」という役割分担で設計するとシンプルになる。

設計の核心は3つだ。

1. CIとCDの責務を分離し、必要なタイミングで必要な処理だけ走らせる
2. matchで証明書をチームで共有し、属人化を排除する
3. シークレット管理を徹底し、機密情報がコードやログに残らないようにする

リリース作業が自動化されると、エンジニアはリリースに怯えなくなる。安心してコードを書き、安心してリリースできる環境がチームの開発速度を上げる。
