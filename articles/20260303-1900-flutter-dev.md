## はじめに

Flutterのナビゲーションは、アプリが成長するにつれて複雑さが増していく問題の代表格だ。

`Navigator.push`で始まったシンプルなナビゲーションが、画面数が増えるにつれてネストしたルート、認証ガード、ディープリンク対応、タブナビゲーションの管理と複雑になっていく。

`go_router`はこれらの複雑さを「宣言的ルーティング」で整理するためのパッケージだ。しかし、使い始めると「Navigator 2.0のラッパーだとは聞いたが、何がどう変わるのか」という疑問が生まれることが多い。

本記事では、`go_router`が解決しようとしている問題の本質から整理する。

## 命令的ナビゲーションの問題

まず`go_router`が解決しようとしている問題を理解するために、従来の命令的ナビゲーション（Navigator 1.0）の課題を整理する。

**状態としてのスタック管理の難しさ**: `Navigator.push/pop`は画面スタックを命令的に操作する。現在のスタック状態を把握しにくく、特定の画面に直接飛んだり、スタックを特定の状態に設定したりすることが難しい。

**ディープリンクの実装複雑さ**: 外部からURLで特定の画面に遷移する場合、スタックを適切に組み立てる処理を手動で書く必要があった。認証状態によって挙動を変える場合はさらに複雑になる。

**Web対応の困難さ**: FlutterのWebアプリでは、ブラウザの戻るボタンやURL変更に対応する必要がある。命令的ナビゲーションではこの対応が煩雑だった。

## 宣言的ルーティングの考え方

`go_router`は「アプリの全ルートを宣言的に定義する」というアプローチをとる。

ルーターを定義するとき、「このURLパスはこの画面に対応する」という対応関係をツリー構造で表現する。

```dart
final GoRouter router = GoRouter(
  routes: [
    GoRoute(
      path: '/',
      builder: (context, state) => const HomeScreen(),
      routes: [
        GoRoute(
          path: 'profile/:userId',
          builder: (context, state) {
            final userId = state.pathParameters['userId']!;
            return ProfileScreen(userId: userId);
          },
        ),
        GoRoute(
          path: 'settings',
          builder: (context, state) => const SettingsScreen(),
        ),
      ],
    ),
  ],
);
```

この定義によって、`/profile/123`というパスにアクセスしたとき、`HomeScreen`を下に持ちつつ`ProfileScreen`が表示されるという**スタックの構造が自動的に決まる**。

命令的ナビゲーションでは、この「スタック構造の構築」を毎回手動でやっていた。宣言的なルート定義がこれを自動化する。

## ネストされたナビゲーションを整理する

タブバーを持つアプリでよく見られる、「タブごとに独立したナビゲーション履歴を持たせたい」というパターン。`go_router`ではShell Routeという概念でこれを表現する。

`ShellRoute`はサブルートを持ち、そのサブルート内のナビゲーションを独立した`Navigator`で処理する。タブバーのUIを保ちながら、各タブが独自のスタックを持てる。

このアーキテクチャをコードで書き下ろすより、「タブAのNavigatorとタブBのNavigatorは独立していて、タブを切り替えても各タブの履歴は保持される」という概念を掴むことが大切だ。

## リダイレクトで認証ガードを実装する

`go_router`の強力な機能の1つが`redirect`だ。

ルートに設定した`redirect`関数は、そのルートにナビゲーションが起きるたびに評価され、必要に応じて別のパスに転送できる。

```dart
redirect: (BuildContext context, GoRouterState state) {
  final isLoggedIn = authState.isLoggedIn;
  final isLoginPage = state.matchedLocation == '/login';
  
  if (!isLoggedIn && !isLoginPage) {
    return '/login';
  }
  if (isLoggedIn && isLoginPage) {
    return '/';
  }
  return null; // null = リダイレクトなし
},
```

認証状態が変わったとき（ログアウトなど）、`notifyListeners`と組み合わせることでルーターが自動的にリダイレクトを再評価し、適切な画面に遷移させる。

「認証ガードをどこに書くか」という問題の答えが、`redirect`として一箇所に集約されるのはメンテナンス性の大きなメリットだ。

## パスパラメータとクエリパラメータの扱い

`go_router`ではURLのパラメータを型安全に扱える仕組みがある。

パスパラメータ（`/user/:id`の`:id`部分）は`state.pathParameters`から取得できる。クエリパラメータ（`/search?q=flutter`の`q`）は`state.uri.queryParameters`から取得できる。

ただし、どちらも`String`型で取得されるため、型変換の処理は自前で書く必要がある。これを解消するために`go_router_builder`というコード生成ツールが用意されており、型付きのルート定義を自動生成できる。

型安全なルーティングを重視するなら`go_router_builder`の採用を検討する価値がある。ただし、コード生成の複雑さが加わるため、プロジェクトの規模と要件に合わせて判断する。

## エラーハンドリングとフォールバック

存在しないルートにアクセスしたときの処理も、`go_router`では一元管理できる。

```dart
final GoRouter router = GoRouter(
  errorBuilder: (context, state) => ErrorScreen(error: state.error),
  routes: [...],
);
```

`errorBuilder`に404相当のページを設定することで、定義されていないパスへのアクセスをキャッチできる。Webアプリとして公開する際には必須の設定だ。

## go_routerを使うべきタイミング

`go_router`の採用を検討すべき状況：

- ディープリンクやWeb対応が必要
- 認証ガードや権限チェックが複数ルートに必要
- タブナビゲーションと画面内ナビゲーションが混在している
- チームで「どの画面からどの画面へ遷移できるか」を明示したい

逆に、画面数が少なく単純なナビゲーションのみのアプリなら、`Navigator.push/pop`の直接利用で十分な場合もある。

`go_router`はナビゲーションの複雑さを整理するためのツールだが、それ自体の学習コストも存在する。導入のメリットとコストを天秤にかけて判断することが大切だ。

## まとめ

`go_router`が解決しようとしているのは「ルーティングの複雑さをコードに散在させない」という問題だ。

- ルート定義を宣言的に一箇所に集める
- ディープリンクとスタック構築を自動化する
- `redirect`で認証ガードを一元管理する
- ShellRouteでネストされたナビゲーションを整理する

ツールの使い方を覚えることより、「URLとスタックとUIの状態を一貫して管理する」という設計思想を掴むことが、`go_router`を使いこなす鍵だ。
