## はじめに——「状態管理」を選ぶとはどういうことか

Flutterアプリを作るとき、避けて通れない問いがある。「状態管理をどうするか」だ。

StatefulWidget、Provider、Riverpod、GetX、MobX——選択肢は豊富で、それぞれに哲学がある。その中でも長年にわたって業務アプリケーション開発で支持されてきたのが **Blocパターン** だ。

Blocを選ぶ理由は「機能が多いから」でも「スターが多いから」でもない。**状態遷移を明示的に設計できる**という思想そのものが、アプリの複雑さが増したときに力を発揮するからだ。

この記事では、BlocとCubitの設計思想、使い分け、そして「なぜ明示性が保守性につながるのか」を掘り下げていく。

---

## BlocパターンとはReactiveな設計思想である

Blocの語源は **Business Logic Component** だ。UIから業務ロジックを切り離し、入力（Event）と出力（State）を明確に定義する構造を指す。

```
UI → Event → Bloc → State → UI
```

この一方向データフローは、Reactの思想ともFluxアーキテクチャとも共鳴する。重要なのは、**UIが「何を求めているか」を表明し、Blocが「どう応えるか」を決定する**という役割分担だ。

Bloc/Cubitパッケージはこの設計を実装するためのフレームワークを提供しているが、本質はパッケージではなく「状態遷移を明示的に管理する」という設計の思想そのものにある。

---

## CubitとBlocの違い——シンプルさと表現力のトレードオフ

`flutter_bloc`パッケージにはCubitとBlocという2つのクラスが存在する。

**Cubit** はシンプルな状態管理クラスだ。状態を変更するメソッドを直接呼び出し、`emit()`で新しい状態を発行する。コードが簡潔で、状態の種類が少ないユースケースに向いている。

**Bloc** はイベント駆動型の設計を強制する。UIからはEventオブジェクトを投げ、Blocの内部でそのEventに対するハンドラが状態遷移を決定する。Cubitより冗長だが、状態遷移のトリガーが何であるかが明示的になる。

```dart
// Cubitのシンプルな例
class CounterCubit extends Cubit<int> {
  CounterCubit() : super(0);

  void increment() => emit(state + 1);
  void decrement() => emit(state - 1);
}
```

```dart
// BlocのEvent駆動な例
sealed class CounterEvent {}
class IncrementPressed extends CounterEvent {}
class DecrementPressed extends CounterEvent {}

class CounterBloc extends Bloc<CounterEvent, int> {
  CounterBloc() : super(0) {
    on<IncrementPressed>((event, emit) => emit(state + 1));
    on<DecrementPressed>((event, emit) => emit(state - 1));
  }
}
```

この差は小さいように見えるが、アプリが複雑になるにつれて意味を持ち始める。

## 「Eventを使う」意味——何が起きたかを記録する

Cubitはメソッドを直接呼び出す。これは直感的だが、**なぜその状態変化が起きたのか**というコンテキストが失われやすい。ログを見たとき「`emit(Loading())`が呼ばれた」ことはわかるが、それがどのユーザー操作やシステムイベントに起因するかは追いにくい。

BlocのEventは違う。`UserTappedSubmit`、`ApiResponseReceived`、`RetryRequested`——イベント名はそのまま「何が起きたか」を語るドメイン言語になる。

デバッグ時にBlocObserverのログを見ると、状態遷移の流れが事件の記録のように読める。これは「Observable性」とも言える特性で、チームで開発しているときのコードレビューやバグ追跡で威力を発揮する。

業務アプリの複雑な注文フローや、複数ステップのウィザードUIでは特に有効だ。`OrderSubmittedEvent`が来てから`PaymentProcessingState`→`OrderConfirmedState`という遷移が明文化されることで、QAチームも仕様と実装の整合性を確認しやすくなる。

## Stateの設計——sealed classで網羅性を保証する

Blocの真価はStateの設計にある。Dart 3以降の**sealed class**との組み合わせが特に強力だ。

```dart
sealed class ProductState {}

class ProductInitial extends ProductState {}
class ProductLoading extends ProductState {}
class ProductLoaded extends ProductState {
  final List<Product> products;
  ProductLoaded(this.products);
}
class ProductError extends ProductState {
  final String message;
  ProductError(this.message);
}
```

UIでsealed classに対してswitch式を使うと、コンパイラがすべてのケースを処理しているかを保証してくれる。

```dart
BlocBuilder<ProductBloc, ProductState>(
  builder: (context, state) => switch (state) {
    ProductInitial() => const SizedBox.shrink(),
    ProductLoading() => const CircularProgressIndicator(),
    ProductLoaded(:final products) => ProductList(products: products),
    ProductError(:final message) => ErrorView(message: message),
  },
)
```

これは単なる文法の工夫ではない。「このStateが来たときのUIを定義し忘れる」というバグを**コンパイル時に防ぐ**仕組みだ。

状態の追加が容易になる点も見逃せない。新しい`ProductRefreshing`ステートを追加したとき、switch式を書いているすべてのUI箇所でコンパイルエラーが出る。対応漏れがコンパイラによって検出される——これがsealed classとBlocの組み合わせがもたらす「網羅性の保証」だ。

## CubitとBlocの使い分け指針

実際のアプリ設計では、すべてをBlocにする必要はない。CubitとBlocを状況に応じて使い分けることが現実的だ。

**Cubitが向いているケース:**
- トグルスイッチ、カウンター、タブ選択など単純なUI状態
- ローカルな状態で、外部トリガーが一種類しかない
- プロトタイプや小規模画面の迅速な実装

**Blocが向いているケース:**
- 複数の入力ソース（ユーザー操作・APIレスポンス・タイマー）が存在する
- 同じ種類の操作でも文脈によって異なる処理をする必要がある
- デバッグ・ログ・テストで状態遷移のトレーサビリティが重要
- チームで開発していてコードの「読み手」を意識したい

混在させることへの心理的抵抗を持つ人もいるが、設計の一貫性はパッケージのクラス名ではなく「状態遷移の責務をどこに置くか」という思想によって確保される。

## BlocObserver——アプリ全体の状態遷移を横断的に観察する

`BlocObserver`はBlocライブラリの地味だが重要な機能だ。すべてのBlocとCubitの遷移・エラー・クローズを横断的に傍受できる。

開発中はこれをログ出力に使い、状態遷移の流れをリアルタイムで観察できる。本番環境ではFirebase CrashlyticsやSentryへのエラー上流として機能する。

特定の状態遷移が意図した順序で起きているか、エラー状態に遷移したBlocはどれかをプロダクションレベルで追跡できるのは、「状態を隠さない」というBlocの哲学の直接的な恩恵だ。

## テスタビリティ——Blocがテストを「書きやすく」する理由

BlocとCubitは純粋な入出力の設計になっているため、テストが自然と書きやすくなる。`bloc_test`パッケージは`build-act-expect`パターンを提供し、状態遷移のシナリオを宣言的に記述できる。

- ある初期状態からイベントを投げた結果、どの状態が順番に発行されるか
- エラー時に適切なState（`ProductError`）に遷移しているか
- 副作用（APIコール）が期待どおりに呼ばれているか

これらをモックと組み合わせてテストできる。UIテストではなくロジックのテストに集中できるため、テスト実行が高速で安定している。

業務アプリのように「ユースケースの組み合わせが多い」状況では、このテスタビリティの差は蓄積されていく。

## 設計の落とし穴——Bloc疲れを起こさないために

Blocパターンに慣れてくると、すべてをBlocで管理しようとする「Bloc疲れ」が起きることがある。いくつかの落とし穴を押さえておきたい。

**1. EventとStateの粒度を細かくしすぎる**
`ButtonHovered`や`TextFieldFocused`のような細かすぎるEventを作り始めると、管理コストが爆発する。EventはUIの意図を表すものであり、実装の詳細を追うものではない。

**2. StateにUIロジックを入れる**
Stateはデータの形であるべきで、「次のボタンを表示するかどうか」といったUI判断をStateに持たせるとUIとロジックが再び混ざり始める。

**3. BlocのネストとBlocの依存**
Blocから別のBlocを直接呼び出す設計は避けるべきだ。依存が必要な場合はRepository層や`BlocListener`を経由して疎結合を保つ。

これらは設計の問題であり、パッケージの問題ではない。Blocはルールを強制してくれるが、設計の質はエンジニアの判断に委ねられている。

---

## まとめ——「明示性」が長期の保守を支える

Blocパターンの核心は「明示的な状態遷移」という思想だ。

Cubitのシンプルさは開発速度をあげ、BlocのEvent駆動は状態変化の文脈を記録する。sealed classとの組み合わせは網羅性をコンパイル時に保証し、BlocObserverは横断的な観察を可能にする。

小規模なアプリでは過剰設計に見えることもある。しかしチームが増え、機能が積み重なり、「このバグがなぜ起きたか」を追う必要が生まれたとき、「状態遷移が記録されている」ことの価値が浮かび上がる。

RiverpodでもGetXでも解決できる問題は多い。それでもBlocが選ばれ続ける理由は、**複雑さと向き合う覚悟を設計で表明する**という姿勢にある。

Flutterアプリが「大きくなっても読めるコードベース」であり続けるために、状態遷移の設計をどこに投資するかは一度じっくり考える価値がある。
