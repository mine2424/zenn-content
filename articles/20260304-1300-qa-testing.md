# k6で「性能」を定義する——負荷テスト設計の思想と実践

「このシステムは性能が良い」と言うとき、その「性能」は何を指しているのだろうか。応答速度？スループット？同時接続数？クラッシュしないこと？

性能テストでよくある失敗は、テストの目的が曖昧なまま実行してしまうことだ。ツールを動かした満足感はあるが、「何が分かったのか」「何を改善すべきか」が不明瞭なまま終わってしまう。

k6は、Grafana Labsが開発したモダンな負荷テストツールだ。JavaScriptでシナリオを記述でき、CLI・CI/CDとの親和性が高い。しかし道具の前に思想がある。本稿では、k6を用いた性能テスト設計の考え方を整理する。

## 性能テストの4種類を理解する

性能テストは目的によって種類が異なり、それぞれ異なる問いに答えるものだ。

**負荷テスト（Load Test）**: 想定される通常〜ピーク負荷でシステムが正常に動作するかを確認する。「通常100RPS、ピーク時300RPSの環境で動くか」という問いに答える。

**ストレステスト（Stress Test）**: 限界を超えた負荷をかけて、システムがどのように壊れるか、回復するかを観察する。「限界はどこか」「限界を超えたときに優雅に劣化するか」を問う。

**スパイクテスト（Spike Test）**: 急激なトラフィック増加に対するシステムの挙動を確認する。セール開始やバイラルコンテンツによる突発的なアクセス増を想定する。

**ソークテスト（Soak Test）**: 長時間（24時間〜数日）にわたって低〜中程度の負荷をかけ続け、メモリリークやリソース枯渇を検出する。

どのテストを実施するかは、プロダクトのリスクプロファイルによって決まる。ECサイトならスパイクテストが重要だし、バッチ処理システムならソークテストが有効だ。

## k6の設計哲学

k6のシナリオはJavaScriptで記述する。シンプルなHTTPリクエストから複雑なユーザーフローまで、コードとして表現できる。

```javascript
import http from 'k6/http';
import { check, sleep } from 'k6';

export const options = {
  stages: [
    { duration: '2m', target: 100 },  // ランプアップ: 0→100VU
    { duration: '5m', target: 100 },  // 安定負荷: 100VU
    { duration: '2m', target: 0 },    // ランプダウン: 100→0VU
  ],
  thresholds: {
    http_req_duration: ['p(95)<500'],  // 95パーセンタイルが500ms以内
    http_req_failed: ['rate<0.01'],    // エラー率1%未満
  },
};

export default function () {
  const res = http.get('https://example.com/api/products');
  check(res, {
    'status is 200': (r) => r.status === 200,
    'response time OK': (r) => r.timings.duration < 500,
  });
  sleep(1);
}
```

ここで重要なのは`thresholds`だ。テストを「通過/失敗」で判定できるようにする。単に数字を集めるだけでなく、**合格基準を定義すること**がk6設計の肝だ。

## SLO（サービスレベル目標）から逆算する

良い性能テストはSLO（Service Level Objective）から設計される。

SLOとは「このサービスはどれくらい速く、どれくらい信頼性があるべきか」という定量的な目標だ。たとえば「APIレスポンスタイムの99パーセンタイルが1秒以内、可用性99.9%」のように定義する。

このSLOがあれば、テストのthresholdsは自然に決まる。逆にSLOなしにthresholdsを決めようとすると「500msって、なんとなくそれくらいかな」という感覚値になってしまう。

SLOはプロダクトマネージャーやビジネスサイドとの合意事項でもある。「このAPIが1秒を超えたらユーザーはページを離脱するデータがある」という根拠から導くことで、技術目標がビジネス目標と繋がる。

## Virtual User（VU）の考え方

k6ではVirtual User（仮想ユーザー）という概念でロードを表現する。VUは「同時に操作しているユーザー数」に相当するが、RPS（Requests Per Second）との関係を理解しておく必要がある。

リトルの法則: L = λ × W（L=平均同時ユーザー数、λ=スループット、W=平均応答時間）

応答時間が100msのAPIで100RPSを出したいなら、L = 100 × 0.1 = 10VU必要になる。しかし応答時間が1sになると、同じ100RPSには100VU必要だ。

この関係を理解せずにVU数だけ増やしても、期待通りのRPSにならないことがある。性能テストで「VUを増やしてもRPSが増えない」という現象はよく起き、それ自体がシステムのボトルネックを示すサインだ。

## シナリオを現実に近づける

性能テストの信頼性は、シナリオが現実のユーザー行動をどれだけ反映しているかによって決まる。

現実のユーザーは、1秒に1000リクエストを一斉に投げたりしない。ページを読んで、ボタンを押して、結果を見て、次の操作をする——そこには自然なリズムがある。

k6でこれを表現するのが`sleep()`だ。リクエスト間に現実的な待機時間を挟むことで、Think Time（ユーザーが操作を考える時間）を模倣できる。

また、すべてのユーザーが同じパスを辿るわけではない。商品一覧から詳細を見るユーザーが70%、検索で直接商品に飛ぶユーザーが30%、というような重み付けをシナリオに組み込むことで、より現実的な負荷分布が得られる。

## ボトルネックの読み方

テスト結果の数字を見るだけでなく、何がボトルネックかを特定する思考が必要だ。

**レスポンスタイムの内訳を見る**: k6はDNS解決、TCP接続確立、TLSハンドシェイク、サーバー処理時間（TTFB）、データ転送時間を分けて計測する。TLSに時間がかかっているなら接続プールの問題かもしれないし、TTFBが長いならサーバー側の処理が遅い。

**エラーパターンを追う**: エラー率が一定のしきい値で急増するなら、コネクションプールの枯渇が疑われる。徐々に増えるならリソースリークかもしれない。

**外部システムを疑う**: データベース、外部API、キャッシュ——これらが性能の律速になることが多い。k6でWebアプリをテストしながら、DBのスロークエリログを同時に観察する視点が重要だ。

## CI/CDへの組み込み

k6の大きな強みは、コマンドラインで動作しCI/CDに統合しやすいことだ。

PRマージ前に軽量な負荷テストを自動実行し、性能劣化を早期に検出できる。「昨日までは200msだったAPIが、このPRで300msになった」という変化を、ステージング環境に反映する前にキャッチできる。

ただしCI上での負荷テストは過負荷にならないよう注意が必要だ。ステージング環境がスケールアウト可能かどうか、テスト用データベースが本番と同等の規模か、といった環境条件が結果に大きく影響する。

## 性能テストは「会話」だ

最終的に性能テストは、エンジニアリングとビジネスの間の対話ツールだ。

「このシステムは毎秒1000リクエストを処理できます」という数字は、「ピーク時のユーザー数を10倍にできます」というビジネス価値に翻訳できる。「95パーセンタイルが500ms以内」という技術指標は、「ほぼ全てのユーザーが快適に操作できる」というUX保証に繋がる。

数字を計測する前に「何を証明したいか」を明確にする——性能テスト設計の本質はそこにある。

## まとめ

- 性能テストは目的（負荷・ストレス・スパイク・ソーク）によって設計が変わる
- thresholdsにSLOを反映させ、合格/失敗を明確に定義する
- VU数とRPSの関係（リトルの法則）を理解する
- シナリオはThink Timeや操作パターンの重み付けで現実に近づける
- 数字の背後にあるボトルネックを特定する思考が重要
- CI/CDに統合して性能劣化を継続的に検出する
