## はじめに

フォームはWebアプリケーションのなかで最も「見た目よりも複雑」なUIコンポーネントの一つだ。入力値の管理、バリデーション、エラーメッセージの表示、送信処理——これらを整合性を保ちながら実装することは、やってみると思いのほか難しい。

React Hook FormとZodの組み合わせは、この複雑さに対する現時点での有力な回答の一つだ。しかし「この2つを組み合わせればよい」という知識だけでは、複雑なフォームに対処できない。本稿では設計の考え方を中心に整理する。

## なぜフォームは難しいのか

フォームの難しさは「状態の複雑さ」に起因する。典型的なフォームが管理する状態を列挙すると、その多さに気づく。

- 各フィールドの現在値
- フィールドが一度でも操作されたか（touched/dirty状態）
- バリデーションエラーの有無と内容
- フォーム全体の送信可否
- 送信中の非同期状態
- 送信後のサーバーエラー

これをuseStateで素直に管理しようとすると、フィールドが増えるたびに状態管理コードが膨張し、再レンダリングも増える。React Hook Formはこの問題を**アンコントロールドコンポーネントとrefによる状態管理**という設計で解決している。

## React Hook Formの設計思想

React Hook Formの核心は「Reactの再レンダリングを最小化する」ことだ。

多くのフォームライブラリはフィールドの値をReactの状態として管理する。つまりユーザーがキーを押すたびに状態が更新され、コンポーネントが再レンダリングされる。入力フィールドが10個あれば、1文字入力するたびに10個分の仮想DOMが再評価される。

React Hook FormはDOMを直接参照することで、多くの操作をReactの外側で処理する。バリデーションエラーが発生したときや送信時など、必要なタイミングだけReactの世界に通知する。この設計によりパフォーマンスの問題を構造的に回避している。

`useForm`フックが返す`register`関数がその仕組みの中心だ。この関数をinput要素に適用すると、refを通じてDOMを直接管理する登録が行われる。

```tsx
const { register, handleSubmit, formState: { errors } } = useForm<FormData>();

// register が返すオブジェクトはref/onChange/onBlur等を含む
<input {...register("email")} />
```

## Zodが担う「スキーマとしてのバリデーション」

ZodはTypeScriptファーストなスキーマバリデーションライブラリだ。Zodの重要な特徴は、バリデーションルールとTypeScript型が**同じソースから生成される**ことだ。

通常、フォームのデータ型とバリデーションルールは別々に書くことが多い。型はTypeScriptインターフェースで定義し、バリデーションは別途実装する。するとデータ型が変わったとき、バリデーションも変更を要する。この「2つを常に同期させる」作業がバグの温床になる。

Zodのスキーマから`z.infer<typeof schema>`でTypeScript型を導出することで、型とバリデーションが常に一致することを保証できる。

```tsx
const contactSchema = z.object({
  email: z.string().email("有効なメールアドレスを入力してください"),
  message: z.string().min(10, "10文字以上入力してください").max(1000),
  category: z.enum(["general", "support", "billing"]),
});

type ContactForm = z.infer<typeof contactSchema>;
// ContactForm は { email: string; message: string; category: "general" | "support" | "billing" } と等価
```

## 2つを繋ぐ：zodResolverの役割

`@hookform/resolvers`パッケージが提供する`zodResolver`は、ZodのスキーマをReact Hook Formのバリデーションとして接続するアダプターだ。

```tsx
import { zodResolver } from "@hookform/resolvers/zod";

const { register, handleSubmit, formState: { errors } } = useForm<ContactForm>({
  resolver: zodResolver(contactSchema),
});
```

これだけで、ZodのバリデーションロジックをReact Hook Formのlifecycleに統合できる。フィールドのフォーカスを外したとき（onBlur）や送信時にZodのスキーマでバリデーションが走り、エラーはform stateに格納される。

## 設計上の重要な判断ポイント

### バリデーションのトリガータイミング

`useForm`の`mode`オプションは、いつバリデーションを実行するかを制御する。

- `onChange`：入力のたびに検証。即時フィードバックは良いが、入力途中の不完全な値にエラーが出てストレスになりやすい
- `onBlur`：フォーカスが外れたときに検証。多くのケースでバランスが良い
- `onSubmit`（デフォルト）：送信時のみ検証。シンプルなフォームに向く
- `onTouched`：最初のblur後はonChangeでも検証。onBlurより積極的なフィードバック

「ユーザーが一度触れたフィールドだけバリデーションを行う」という挙動がUX的に自然なことが多く、`onTouched`や`onBlur`が選ばれやすい。

### サーバーエラーとの統合

Zodのクライアントバリデーションを通過しても、サーバーサイドでエラーが返ることがある（メールアドレスの重複など）。React Hook Formの`setError`を使うと、サーバーエラーを特定フィールドのエラーとして統合できる。

```tsx
const onSubmit = async (data: ContactForm) => {
  const result = await submitForm(data);
  if (result.error?.field === "email") {
    setError("email", { message: result.error.message });
  }
};
```

これにより、クライアント・サーバー両方のエラーを統一的なインターフェースで扱える。

### スキーマの分割と再利用

アプリケーションが成長すると、フォームごとにスキーマを一から書くのは非効率だ。Zodのスキーマはオブジェクトとして組み合わせられるため、共通フィールドを切り出して再利用できる。

ユーザー情報フォームと注文フォームが両方メールアドレスを持つなら、emailのバリデーションは共通スキーマとして定義する。これにより「バリデーションルールを1箇所で管理する」原則を守れる。

## Next.js App Routerとの文脈

Next.js App RouterではServer Actionsとの組み合わせも重要になる。Server Actionsを使う場合、フォームのデータはサーバー側に送られる。Zodのスキーマをサーバー側でも同じように使うことで、クライアントとサーバーで一貫したバリデーションを実現できる。

`useActionState`（旧`useFormState`）とReact Hook Formを組み合わせる場合、クライアントバリデーションをReact Hook Formで行い、Server Actionの結果をsetErrorに反映するパターンが機能する。ただし「クライアントで事前検証 → サーバーで本検証」という二重バリデーションの設計は、一部重複するため管理コストを意識する必要がある。

## おわりに

React Hook Form + Zodの組み合わせが広まった理由は、それぞれが設計上の明確な問題を解決しているからだ。React Hook Formは「再レンダリングを最小化したフォーム状態管理」を、Zodは「型とバリデーションの単一ソース」を提供する。

ただし、これらのライブラリを使えば自動的に良いフォームになるわけではない。どのタイミングでバリデーションするか、サーバーエラーをどう扱うか、スキーマをどう分割・再利用するか——これらの設計判断は依然として開発者に委ねられている。ライブラリは道具であり、設計の責任は変わらない。
