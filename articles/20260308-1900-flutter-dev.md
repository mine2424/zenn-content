## Platform Channelの「型のない世界」

FlutterからネイティブコードにアクセスするためのPlatform Channelは、強力な仕組みだ。しかし素のMethodChannelには、大きな落とし穴がある。それは**型安全性がない**という点だ。

MethodChannelを使ったコードを思い浮かべると、Dart側では`dynamic`や`Map<dynamic, dynamic>`を扱い、iOS側ではSwiftの`Any?`を、Android側ではJavaのObjectを扱うことになる。引数の名前を文字列で指定するため、タイプミスはコンパイル時に検出されない。実行時にクラッシュして初めてミスに気づく。

```dart
// 素のMethodChannel：型が失われる
final channel = MethodChannel('com.example/battery');

// 文字列での呼び出し：タイプミスはコンパイル時に気づけない
final result = await channel.invokeMethod('getBatteryLevel');
final level = result as int; // キャストが必要
```

この「型のない橋渡し」を解決するのが**Pigeon**だ。

---

## Pigeonとは何か

PigeonはFlutter公式が提供するコード生成ツールだ。Dartでインターフェースを定義すると、そのインターフェースを実装するためのDart・Swift（Kotlin）の定型コードを自動生成してくれる。

設計の流れはこうだ。まずDartファイルでAPIの仕様（どんなメソッドがあり、何を受け取り、何を返すか）を定義する。`flutter pub run pigeon`コマンドを実行すると、Flutter側のDartコード、iOS用のSwiftコード、Android用のKotlinコードが生成される。

型定義はPigeonが管理するため、DartとネイティブのAPIが自動的に一致した状態を保てる。

## インターフェース定義の書き方

Pigeonの定義ファイルは通常のDartコードだが、生成のためのアノテーションを使う。

```dart
// pigeons/battery_api.dart

import 'package:pigeon/pigeon.dart';

@ConfigurePigeon(PigeonOptions(
  dartOut: 'lib/src/battery_api.g.dart',
  swiftOut: 'ios/Runner/BatteryAPI.g.swift',
  kotlinOut: 'android/app/src/main/kotlin/com/example/BatteryAPI.g.kt',
  kotlinOptions: KotlinOptions(package: 'com.example.app'),
))

// データクラスの定義
class BatteryInfo {
  BatteryInfo({required this.level, required this.isCharging});
  final int level;
  final bool isCharging;
}

// ネイティブ側が実装するAPI（Flutter → Native）
@HostApi()
abstract class BatteryHostApi {
  BatteryInfo getBatteryInfo();
  @async
  BatteryInfo getBatteryInfoAsync();
}

// Flutter側が実装するAPI（Native → Flutter）
@FlutterApi()
abstract class BatteryFlutterApi {
  void onBatteryLevelChanged(BatteryInfo info);
}
```

`@HostApi`はFlutterからネイティブに呼び出すAPIの定義だ。ネイティブ側（SwiftまたはKotlin）でこのインターフェースを実装する。

`@FlutterApi`はネイティブからFlutterへのコールバックのための定義だ。Flutter（Dart）側でこれを実装し、ネイティブから呼び出す。

## 生成されたコードを使う

コード生成後は、生成されたクラスを使うだけだ。

**Dart側（呼び出し）：**

```dart
// 生成されたAPIクラスを使う
final api = BatteryHostApi();

// 型安全に呼び出せる：引数も戻り値も型が保証される
final info = await api.getBatteryInfoAsync();
print('バッテリー: ${info.level}% (充電中: ${info.isCharging})');
```

文字列でメソッド名を指定する必要がない。`invokeMethod`も`as`キャストも不要だ。IDEの補完が効き、メソッド名を間違えるとコンパイルエラーになる。

**Swift側（実装）：**

```swift
// BatteryAPIを実装するクラス
class BatteryPlugin: NSObject, BatteryHostApi {
  func getBatteryInfo() throws -> BatteryInfo {
    UIDevice.current.isBatteryMonitoringEnabled = true
    let level = Int(UIDevice.current.batteryLevel * 100)
    let isCharging = UIDevice.current.batteryState == .charging
    return BatteryInfo(level: Int64(level), isCharging: isCharging)
  }

  func getBatteryInfoAsync(completion: @escaping (Result<BatteryInfo, Error>) -> Void) {
    // 非同期処理の例
    DispatchQueue.global().async {
      // ... 非同期でバッテリー情報を取得
      completion(.success(BatteryInfo(level: 85, isCharging: false)))
    }
  }
}

// AppDelegate.swiftで登録
BatteryHostApiSetup.setUp(binaryMessenger: controller.binaryMessenger, api: BatteryPlugin())
```

Swiftのメソッドシグネチャは生成されたコードに従うため、Dartの定義と必然的に一致する。

## 素のMethodChannelとの比較

なぜPigeonを使うのか、もう少し具体的に整理する。

**型安全性。** 引数と戻り値の型が保証される。実行時の`MissingMethodException`や`ClassCastException`のリスクが大幅に減る。

**インターフェースの明確化。** 定義ファイルを読めば、Flutterとネイティブ間でどんなやり取りが発生するかが一目でわかる。DartとSwift/Kotlinの両方をよく理解しなくても、定義ファイルが「橋渡し仕様書」として機能する。

**コードの自動同期。** 定義を変えれば、生成コマンドで全プラットフォームのコードが更新される。「Dart側は直したけどSwift側を忘れていた」という事故が防ぎやすい。

一方、素のMethodChannelの方が適している場面もある。

単純なワンオフの連携（1メソッドだけ呼ぶ）には、Pigeonのセットアップは大げさかもしれない。また、イベントストリーム（EventChannel）はPigeonではまだ限定的なサポートで、継続的なストリームには直接EventChannelを使う方が自然な場合もある。

## データクラスの設計で気をつけること

Pigeonで定義できるデータクラスにはいくつかの制約がある。

Dartの通常のクラスと違い、Pigeonのデータクラスは継承が使えない。コンポジション（他のデータクラスをフィールドとして持つ）は可能だ。

```dart
// ネストされたデータクラス
class Location {
  Location({required this.latitude, required this.longitude});
  final double latitude;
  final double longitude;
}

class DeviceStatus {
  DeviceStatus({required this.battery, required this.location});
  final BatteryInfo battery;
  final Location location;
}
```

nullable（`?`付き）フィールドも使えるが、Swiftの`Optional`やKotlinの`nullable`として生成されるため、ネイティブ側での扱いに注意が必要だ。

## 既存プロジェクトへの段階的な導入

既にMethodChannelで実装されている箇所をPigeonに移行したい場合、一度にすべてを変える必要はない。

まずは新規に追加するネイティブ連携からPigeonを使い始めるのが現実的だ。既存のMethodChannelは動いているなら、無理に移行するメリットよりもリスクの方が大きい場合もある。

既存コードをPigeonに移行する動機としては、「そのチャンネルが複雑化してきた（メソッドが5つ以上になってきた）」「型エラーによるバグが発生した」「iOSとAndroidで仕様の食い違いが起きた」——このあたりが現実的なトリガーになる。

---

## まとめ

PigeonはPlatform Channelの「型のない世界」に対する実践的な答えだ。Dartでインターフェースを定義し、コードを生成することで、Flutterとネイティブ間の連携に型安全性とAPIの明確さをもたらす。

素のMethodChannelが「動けばよい」のに対し、Pigeonは「正しく動き、変化に耐え、チームが理解できる状態を維持する」ための設計ツールだ。

プラグイン開発や複数の機能をネイティブ連携で実装する場合、Pigeonを採用することでメンテナンス性が大きく向上する。Flutterでネイティブと深く連携するプロジェクトなら、Pigeonの導入を検討する価値は十分にある。
