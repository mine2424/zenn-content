## OGP画像の「めんどくさい」を解消する

SNSでリンクをシェアしたとき、サムネイルとして表示されるあの画像——Open Graph画像（OGP画像）の管理は、長らくWebエンジニアにとって地味に辛い問題だった。

単一の静的画像を1枚用意しておくだけなら楽だが、記事ごと・ページごとに異なるタイトルや情報を反映しようとすると途端に複雑になる。Puppeteerでヘッドレスブラウザを立ち上げてスクリーンショットを撮る方法は有名だが、ビルド時間の増大・インフラコストの増加・キャッシュ管理の煩雑さが伴う。

Next.jsの `next/og` パッケージは、こうした課題にエレガントな回答を提示している。Vercelが開発した軽量なHTML/CSS→画像変換エンジンを使い、Edge Runtimeで動的にOGP画像を生成する仕組みだ。

---

## `next/og` が採用した設計上の決断

`next/og` の内部では、**Satoriというライブラリ**が使われている。SatoriはHTML/CSS（のサブセット）をSVGに変換し、そこからPNGを生成する。注目すべきは「React要素をそのまま受け取れる」という点だ。

```tsx
// app/og/route.tsx
import { ImageResponse } from "next/og";

export const runtime = "edge";

export async function GET(request: Request) {
  const { searchParams } = new URL(request.url);
  const title = searchParams.get("title") ?? "デフォルトタイトル";

  return new ImageResponse(
    (
      <div
        style={{
          display: "flex",
          flexDirection: "column",
          background: "#0f172a",
          width: "100%",
          height: "100%",
          padding: "60px",
        }}
      >
        <p style={{ color: "#94a3b8", fontSize: 28 }}>My Blog</p>
        <h1 style={{ color: "#f1f5f9", fontSize: 64, lineHeight: 1.2 }}>
          {title}
        </h1>
      </div>
    ),
    { width: 1200, height: 630 }
  );
}
```

JSXで書いたレイアウトがそのまま画像になる。Figmaで作ったデザインをコードで再現する作業が、これだけシンプルになる。

## Edge Runtimeで動かす意味

`next/og` はEdge Runtimeを強く推奨している（`export const runtime = "edge"` を明示する）。これはパフォーマンス上の選択だ。

Node.jsランタイムでも動作するが、Edge Runtimeには明確な利点がある。起動が速く（コールドスタートがほぼゼロ）、CDNのエッジロケーションで実行できるため、ユーザーに近い場所でレスポンスを返せる。OGP画像はSNSクローラーやLinkPreviewがリクエストするため、「世界中からリクエストが来る可能性がある」というユースケースとEdge Runtimeの相性は非常に良い。

ただし制約もある。Sartoriが依存するWebフォントの読み込みや、Node.js固有のモジュール（`fs`など）はEdgeでは使えない。フォントを使いたい場合は、`fetch`でリモートのフォントファイルを取得するか、`public/fonts/`に配置してバンドルに含める必要がある。

## App Routerでの統合パターン

生成したOGP画像をページに結びつけるには、App RouterのMetadata APIを使う。

```ts
// app/blog/[slug]/page.tsx
import type { Metadata } from "next";

export async function generateMetadata({
  params,
}: {
  params: { slug: string };
}): Promise<Metadata> {
  const post = await getPost(params.slug);

  return {
    title: post.title,
    openGraph: {
      title: post.title,
      images: [
        {
          url: `/og?title=${encodeURIComponent(post.title)}`,
          width: 1200,
          height: 630,
        },
      ],
    },
    twitter: {
      card: "summary_large_image",
      images: [`/og?title=${encodeURIComponent(post.title)}`],
    },
  };
}
```

`generateMetadata` は非同期関数なので、DBやCMSからデータを取得してメタデータを動的に構築できる。静的サイトでも、動的ルートでも、同じパターンが使える一貫性が美しい。

## キャッシュ設計——「生成コスト」をどう扱うか

動的生成には当然、毎回のリクエストでCPUを使うコストが伴う。同じパラメータへのリクエストが繰り返されるなら、キャッシュが重要になる。

Vercelにデプロイする場合、`ImageResponse` はデフォルトでCDNにキャッシュされる。`Cache-Control` ヘッダーを適切に設定することで、生成済み画像を再利用できる。

```ts
return new ImageResponse(jsx, {
  width: 1200,
  height: 630,
  headers: {
    "cache-control": "public, immutable, no-transform, max-age=31536000",
  },
});
```

コンテンツが変化しない（タイトルが固定の記事ページなど）なら、積極的にキャッシュする。頻繁に変わるページ（ダッシュボードの統計情報など）なら短いTTLか `no-cache` にする。この判断は通常のAPIキャッシュ設計と同じ考え方で臨める。

## デザインの自由度と制約を理解する

Satoriが対応しているCSSは**CSSのサブセット**だ。すべてのプロパティが使えるわけではない。

使えるもの:
- Flexboxレイアウト（gridは未対応）
- 基本的なテキストスタイル（`font-size`, `color`, `line-height`）
- `border-radius`, `background`, `opacity`
- `object-fit` による画像配置

制限があるもの:
- CSS Grid
- `transform` の一部
- 複雑な `clip-path`
- ブラウザ固有フォントの自動フォールバック

これを「制約」として嫌うか、「明確な境界線」として受け入れるかで使い心地が変わる。実際のところ、1200×630pxの静的画像を作るためのレイアウトに、CSS Gridやcomplexなtransformが必要になることはあまりない。必要なレイアウトのほとんどはFlexboxで表現できる。

## フォントの扱い方

日本語コンテンツを扱うとき、フォント設定は避けて通れない。システムフォントへのフォールバックは期待できないため、明示的にフォントファイルを用意する必要がある。

```ts
const fontData = await fetch(
  new URL("/fonts/NotoSansJP-Bold.ttf", process.env.NEXT_PUBLIC_SITE_URL)
).then((res) => res.arrayBuffer());

return new ImageResponse(jsx, {
  width: 1200,
  height: 630,
  fonts: [
    {
      name: "NotoSansJP",
      data: fontData,
      style: "normal",
      weight: 700,
    },
  ],
});
```

Google FontsのURLからfetchする手もあるが、外部依存が増える。`public/fonts/`に配置してバンドルに含めるか、Vercel Blob / CDN経由で配信するほうが安定する。

フォントファイルのサイズも考慮が必要だ。日本語フォントは全漢字を含めると数MBになる。Edge Runtimeのメモリ上限（通常128MB）を意識しながら、サブセット化されたフォントを使うのが現実的な選択肢になる。

## `next/og` を使うべきタイミング

すべてのプロジェクトに `next/og` が必要なわけではない。使う価値があるのはこんなケースだ。

**向いているケース**:
- ブログや技術記事サイトで、記事ごとに異なるOGP画像を出したい
- ユーザーのプロフィールや統計をOGP画像に表示するSaaSアプリ
- ECサイトで商品名・価格・画像をOGPに反映したい
- デザインシステムを持つチームで、一貫したOGPデザインをコードで管理したい

**向いていないケース**:
- 全ページ共通の静的OGP画像で十分なシンプルなサイト
- 非常に複雑なデザイン（凝ったアニメーションやCSS Gridを多用するレイアウト）
- OGP画像の生成コストが予算に対して大きすぎる低トラフィックサイト

---

## 設計の本質：「コードがデザインを語る」

`next/og` が提示する設計思想のポイントは、**OGP画像の仕様をコードで表現できる**ということだ。

これまでOGP画像はPhotoshopやFigmaでデザインされ、PNGとして書き出され、リポジトリに静的ファイルとして管理されることが多かった。バリエーションを増やすには手作業が必要で、デザインの変更をコードに追随させることは難しかった。

JSXでレイアウトを書き、TypeScriptで型を付け、GitでバージョンをトラックできるOGP画像は、開発のワークフローに自然に溶け込む。デザインの変更がコードのPRとして追跡でき、CIで自動的に反映される。

「画像」という扱いにくいアセットを「コード」の世界に引き込む——これが `next/og` が実現しようとした本質的な価値だと思う。

---

## まとめ

- `next/og` はEdge RuntimeでJSX→OGP画像をリアルタイム生成する
- Satoriを内部で使い、ReactコンポーネントをそのままOGP画像として書ける
- App RouterのMetadata APIと組み合わせることで、ページごとの動的OGPが簡単に実現できる
- CSSはサブセットのみ対応。日本語コンテンツはフォントファイルの準備が必要
- キャッシュ設計を適切に行えば、生成コストは現実的な範囲に収まる

「OGP画像を作る」という地味な作業を、コードで管理できる一等市民に変える——そういう視点でツールを評価するのが、フロントエンドエンジニアとしての成熟なのかもしれない。
