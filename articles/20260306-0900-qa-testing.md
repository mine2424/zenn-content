# OpenAPIバリデーションで「仕様と実装のズレ」を自動検出する設計思想

APIには「仕様書」と「実装」の2つが存在する。そして多くのチームでは、この2つが少しずつズレていく。

「このフィールドはnullableのはずだった」「エラーレスポンスの構造が変わっている」「必須パラメーターが増えた」——こうした「仕様と実装のズレ」は、気づかぬうちに積み重なり、APIの利用者（フロントエンドや外部クライアント）に予期しないバグをもたらす。

この記事では、OpenAPIバリデーションを使って「仕様と実装のズレ」を自動検出するための設計思想を整理する。

## 問題の本質：ドキュメントは腐る

多くのチームはAPIの仕様書をOpenAPI（Swagger）形式で管理している。しかしこれは、ドキュメントを別管理する「Code First」の文化では特に、実装が変わってもドキュメントが追従しないという問題を生む。

仕様書は一度書かれると、なかなか更新されない。レビューのチェックリストに「仕様書を更新したか」と書いてあっても、タイトなスケジュールの前では後回しにされがちだ。

**しかし本当の問題はここではない。** ドキュメントの正確性を「人間のレビュー」に頼っている点だ。

テストコードがあれば、実装の振る舞いが変わったときに自動で気づける。OpenAPIバリデーションは、「APIの仕様書」を自動テストの一部として機能させるアプローチだ。

## OpenAPIバリデーションとは何か

OpenAPIバリデーションには大きく2つの方向性がある。

### リクエスト/レスポンスの実行時バリデーション

実際のAPIリクエスト・レスポンスを、OpenAPI仕様書と照合して「仕様通りか」を検証する。

たとえば`express-openapi-validator`（Node.js）を使うと、ミドルウェアとして組み込むだけで、すべてのリクエスト・レスポンスを仕様書と照合し、不一致があればエラーを投げる。

```typescript
import OpenApiValidator from 'express-openapi-validator';

app.use(
  OpenApiValidator.middleware({
    apiSpec: './openapi.yaml',
    validateRequests: true,
    validateResponses: true,
  })
);
```

開発環境・テスト環境でこれを有効にすると、仕様書との乖離がすぐに検出できる。

### スキーマの静的バリデーション

OpenAPI仕様書自体に矛盾や誤りがないかを静的に検証するアプローチ。`@redocly/cli`や`spectral`などのLintツールが使える。

これはCI/CDに組み込むことで、仕様書の品質を継続的に保証できる。

## 「Contract」としてのOpenAPI

より本質的な思想として、OpenAPIをAPIの「契約書（Contract）」として扱うことが重要だ。

コントラクトテストの文脈では、APIの提供者（プロバイダー）と消費者（コンシューマー）の間に「約束」を定義し、その約束が守られているかを継続的に検証する。

OpenAPIはまさにこの「約束」を記述するための形式だ。

**仕様書駆動開発（Design-First）** の場合、先にOpenAPIを書いてから実装することで、「仕様が唯一の真実」という状態が作れる。実装は仕様に従うだけになり、バリデーションで仕様との乖離を検出できる。

**実装駆動（Code First）** の場合も、実装からOpenAPIを自動生成（SwaggerやNestJSのデコレーターを活用）することで、常に「実装と仕様が同期した状態」を維持できる。

いずれにしても、**仕様書と実装が同期している状態を「デフォルト」にする**ことが設計の核心だ。

## テスト戦略への組み込み方

OpenAPIバリデーションは、既存のテスト戦略にどう組み込むべきか。

### 統合テストへの組み込み

APIの統合テスト（E2Eテストの一部）を実行する際、レスポンスを手動でアサーションするのに加えて、OpenAPIバリデーションを自動実行する。

```typescript
// テスト内でOpenAPIバリデーターを使う例
const response = await request(app).get('/users/123');

// 通常のアサーション
expect(response.status).toBe(200);
expect(response.body.id).toBe(123);

// OpenAPIバリデーション（仕様との一致確認）
const validator = new OpenApiValidator(openApiSpec);
validator.validateResponse(response, '/users/{id}', 'get', 200);
```

これにより、「動いているが仕様と異なる」状態を検出できる。

### CI/CDへの組み込み

以下のチェックをCIパイプラインに加えることで、仕様の品質を継続的に担保できる。

1. **仕様書のLint**: `spectral lint openapi.yaml`で仕様書の品質チェック
2. **モックサーバーによる事前検証**: APIが実装される前から、仕様書に基づくモックサーバーでフロントエンドのテストを実行
3. **差分検出**: `openapi-diff`で仕様書の破壊的変更（Breaking Changes）を検出

特に差分検出は強力で、「既存のクライアントを壊すような変更」をPRの段階で検知できる。

## よくある落とし穴

### 「仕様書が古い」問題の解決を後回しにしない

既存のコードベースにOpenAPIバリデーションを導入すると、大量のバリデーションエラーが出ることがある。これを見て「後で直す」と先送りしても、結局直されない。

アプローチとして、まず`validateResponses: false`でリクエストのみバリデーションし、段階的に対象を広げていくのが現実的だ。

### nullable と optional の混同

OpenAPIにおいて`nullable: true`と`required`の関係は混乱しやすい。フィールドが存在しない（undefinedの）ケースと、`null`が入るケースは別物だ。この定義のズレが、実装とのミスマッチの温床になる。

## まとめ

OpenAPIバリデーションの本質は、「仕様書を生きたドキュメントにする」ことだ。人間がレビューするのではなく、ツールが自動で仕様と実装の一致を確認する仕組みを作る。

「ドキュメントは腐る」という前提を受け入れ、腐らない仕組みを設計する。それがAPIの品質を長期的に守る唯一の方法だ。
