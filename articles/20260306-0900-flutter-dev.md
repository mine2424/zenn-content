# FlutterのFlavorで「環境分離」を設計する——開発・本番を安全に管理する

モバイルアプリ開発において、「本番APIに開発中のコードが繋がってしまった」「テスト中にリアルユーザーデータを上書きした」という事故は、珍しいことではない。

開発・ステージング・本番という複数の環境を安全に管理するために、Flutterには「Flavor（フレーバー）」という仕組みがある。Flavorを適切に設計することで、環境ごとの設定を安全に分離し、誤接続や事故を防ぐことができる。

この記事では、FlutterのFlavorの設計思想と、実践的な構成方法を整理する。

## なぜ環境分離が必要か

そもそも、なぜ「環境」を分けるのか。

**開発環境（development）**: エンジニアが日々開発・デバッグする環境。ログが詳細に出る、デバッグツールが有効、APIはローカルやdevelopment用サーバーを指す。

**ステージング環境（staging）**: QAや受け入れテストを行う環境。本番に近い設定だが、データは本番と分離。App Distributionや TestFlightで配布することが多い。

**本番環境（production）**: 実際のユーザーが使う環境。ログを最小限に、デバッグツールは無効、本番APIに接続。

この3つを「コードの条件分岐」で管理しようとすると破綻する。

```dart
// 悪い例：コードの条件分岐で環境を管理
final apiUrl = kDebugMode 
  ? "https://dev-api.example.com" 
  : "https://api.example.com";
```

この方法は、`kDebugMode`がビルドモード（debug/release）に紐づいており、「staging用にrelaseビルドしたい」「developmentのreleaseビルドでデバッグしたい」といったニーズに対応できない。Flavorはこの問題を正面から解決する。

## Flavorの基本構造

Flavorは、iOSのScheme（Xcodeの設定）とAndroidのProduct Flavor（Gradleの設定）を、Flutterの世界から統一的に管理する仕組みだ。

```
app
├── flavor: development
│   ├── App ID: com.example.app.dev
│   ├── App名: "MyApp Dev"
│   └── Icon: 開発用アイコン（赤いバナーなど）
├── flavor: staging
│   ├── App ID: com.example.app.staging
│   ├── App名: "MyApp Staging"
│   └── Icon: ステージング用アイコン（黄色いバナー）
└── flavor: production
    ├── App ID: com.example.app
    ├── App名: "MyApp"
    └── Icon: 通常のアイコン
```

App IDが異なるため、同じデバイスに複数のFlavorのアプリを同時にインストールできる。これにより、開発版と本番版を同じ端末で並べて確認することが可能だ。

## 設定値の管理：環境ごとのConfigクラス

Flavorを導入したら、次は「環境ごとの設定値」をどう管理するかだ。よく使われるパターンは、環境別の設定クラスを用意し、エントリポイントで初期化する方法だ。

```dart
// lib/config/app_config.dart
class AppConfig {
  final String apiBaseUrl;
  final String appName;
  final bool enableAnalytics;
  final bool enableDebugBanner;

  const AppConfig({
    required this.apiBaseUrl,
    required this.appName,
    required this.enableAnalytics,
    required this.enableDebugBanner,
  });

  static const development = AppConfig(
    apiBaseUrl: 'https://dev-api.example.com',
    appName: 'MyApp Dev',
    enableAnalytics: false,
    enableDebugBanner: true,
  );

  static const staging = AppConfig(
    apiBaseUrl: 'https://staging-api.example.com',
    appName: 'MyApp Staging',
    enableAnalytics: false,
    enableDebugBanner: false,
  );

  static const production = AppConfig(
    apiBaseUrl: 'https://api.example.com',
    appName: 'MyApp',
    enableAnalytics: true,
    enableDebugBanner: false,
  );
}
```

エントリポイントは、Flavorごとに別ファイルを用意する。

```dart
// lib/main_development.dart
void main() {
  runApp(const MyApp(config: AppConfig.development));
}

// lib/main_production.dart
void main() {
  runApp(const MyApp(config: AppConfig.production));
}
```

ビルド時は`flutter run --flavor development -t lib/main_development.dart`のように、Flavorとターゲットファイルをセットで指定する。

## APIキーやシークレットの扱い

環境ごとに異なるAPIキーをどう管理するかは、セキュリティ上重要な問題だ。

**やってはいけないこと**: APIキーをコードにハードコードし、Gitにコミットする。

**推奨アプローチ**:

1. **--dart-defineを使う**: ビルド時に環境変数として渡す方法。
   ```bash
   flutter build apk --flavor production \
     --dart-define=API_KEY=your_key_here
   ```
   コード内では`const String.fromEnvironment('API_KEY')`で受け取る。

2. **Secrets管理ツールとCI/CDを組み合わせる**: GitHub ActionsのSecrets等でキーを管理し、CIビルド時に`--dart-define`で注入する。

この方法なら、キーはコードに残らず、環境ごとに安全に切り替えられる。

## アイコンとApp名の分離

Flavorの実用的な恩恵のひとつが、アイコンとアプリ名を環境ごとに変えられることだ。

「今インストールされているアプリが開発版か本番版かわからない」という混乱を防ぐため、開発・ステージング版のアイコンには目立つバナーをつけることが推奨される。`flutter_launcher_icons`パッケージと組み合わせることで、Flavor別のアイコン生成を自動化できる。

視覚的な区別があることで、「誤って開発版アプリで本番データを触った」という人的ミスを防ぐ防護壁になる。

## CI/CDとFlavorの組み合わせ

Flavorの真価が発揮されるのは、CI/CDパイプラインとの組み合わせだ。

```yaml
# GitHub Actionsの例（簡略）
- name: Build Staging
  run: |
    flutter build apk \
      --flavor staging \
      -t lib/main_staging.dart \
      --dart-define=API_KEY=${{ secrets.STAGING_API_KEY }}

- name: Distribute to Firebase App Distribution
  run: |
    firebase appdistribution:distribute app-staging.apk \
      --app ${{ secrets.FIREBASE_STAGING_APP_ID }} \
      --groups "qa-team"
```

`develop`ブランチへのマージ時に自動でstagingビルドが配布され、QAチームが即座にテストできる。`main`ブランチへのマージ時にはproductionビルドが作成される——このフローがFlavorで安全に実現できる。

## まとめ

FlutterのFlavorは、「環境の違いをコードの条件分岐ではなく、ビルド設定として表現する」という設計思想に基づいている。

アプリIDの分離、アイコンの視覚的区別、シークレットの安全な注入、CI/CDとの統合——これらすべてが一体となって「誤接続事故ゼロ」の開発環境を作り出す。

環境分離は「後から入れるもの」ではなく、プロジェクト開始時に設計するものだ。早めにFlavorを導入し、安全な開発サイクルを最初から確立しよう。
