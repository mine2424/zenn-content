## すべてのニューロンを毎回使う必要はない

大規模言語モデルの発展を追ってきた人ならば、「モデルが大きくなるほど賢くなる」という経験則に馴染みがあるはずだ。パラメータ数を増やせば性能が上がる。では、パラメータを増やし続けることにどんなコストが伴うのか。

その問いへの一つの答えが、**Mixture of Experts（MoE）** というアーキテクチャだ。

従来のDense（密）モデルは、入力トークンに対してすべてのパラメータを活性化させる。10億パラメータのモデルなら、1トークンを処理するたびに10億個のパラメータが計算に参加する。これは整合的だが、非効率でもある。

MoEはその前提を崩す。「すべてのパラメータを毎回使わなくてもよい。適切な専門家だけを呼び出せばいい」という発想だ。

---

## Expertとは何か

MoEの基本構造はシンプルだ。Transformerのフィードフォワード層（FFN）を、複数の「Expert」と呼ばれるサブネットワークに置き換える。そして**Gating Network（ルーター）**が、各トークンに対してどのExpertを使うかを動的に決定する。

たとえば8つのExpertがあったとして、Gating Networkは「このトークンにはExpert 2とExpert 6を使え」と判断する。他の6つのExpertはそのトークンの処理に参加しない。この「どのExpertを使うかの選択」をスパース（疎）と呼ぶ。

```python
# 概念的なMoEレイヤーのイメージ
class MoELayer:
    def __init__(self, experts, top_k=2):
        self.experts = experts  # Expert群
        self.router = Router()  # Gating Network
        self.top_k = top_k    # 各トークンで使うExpert数

    def forward(self, x):
        # どのExpertを使うかを決定（top_k個選択）
        expert_weights, expert_indices = self.router(x, top_k=self.top_k)
        # 選ばれたExpertのみ計算
        output = sum(
            expert_weights[i] * self.experts[expert_indices[i]](x)
            for i in range(self.top_k)
        )
        return output
```

重要なのは、モデル全体のパラメータ数（**総パラメータ数**）と、1トークンを処理する際に実際に使われるパラメータ数（**活性化パラメータ数**）が分離されるという点だ。

---

## なぜこの構造が魅力的なのか

MoEの利点は「大きくて軽い」という矛盾した特性を実現できることだ。

**推論コストを抑えながらモデルを大きくできる。** Dense モデルで7Bパラメータを持つモデルと、MoEで総パラメータ46Bだが活性化パラメータは7Bというモデルを比較した場合、推論時の計算量は同程度になる。しかし後者は、多くのExpertの「知識」を持っているため、様々なタスクに対してより豊かな表現力を発揮できる可能性がある。

**専門化が起きる。** 学習が進むと、各Expertがドメインごとやトークンのタイプごとに自然と特化していく傾向がある。数学的なトークンにはExpert A、コード的な文脈にはExpert Bといった分業が自然発生する。これは設計上の意図ではなく、データから学習される構造だ。

**学習効率が向上する場合がある。** 各Expertは自分に割り当てられたトークンのみを学習するため、特定のパターンに対してより深く特化できる。

## 実際のモデルにおけるMoE

MoEは理論的な話ではなく、実際に大規模モデルに組み込まれている。

Googleの**Switch Transformer**（2021年）は、MoEをTransformerに本格的に統合した初期の重要な研究だ。top-1ルーティング（各トークンに1つのExpertのみ選択）で、当時の最大規模のモデルを学習させた。

**Mixtral 8x7B**（Mistral AI）は、オープンソースコミュニティに大きな衝撃を与えた。8つの7BパラメータのExpertを持つが、各トークンで2つのExpertのみを使用。実質的な計算量は13Bモデル程度でありながら、性能は70Bクラスのモデルと肩を並べる場面も多い。

**GPT-4もMoEを採用しているのではないか**という噂は長らくあり、多くの研究者がその可能性を指摘している（公式には確認されていない）。

---

## 解決しきれていない課題

MoEは魅力的だが、設計上の難しさもある。

**Expert偏在（load imbalance）問題。** 学習が進むと、Gating Networkが特定のExpertばかりを選ぶようになることがある。一部のExpertが過負荷になり、他は使われないという状態だ。これを防ぐため、学習時に**補助ロス（auxiliary loss）**を加えてExpertへのトークン割り当てを均等化する工夫が必要になる。

**通信コスト（分散学習時）。** MoEモデルを複数GPUで学習させる場合、Expert間のトークンのルーティングがGPU間通信を増加させる。これを「Expert Parallelism」と呼び、効率的な分散戦略が求められる。

**メモリフットプリント。** 推論時の計算量は少なくても、すべてのExpertをメモリに乗せておく必要がある。総パラメータ数が大きければ、それだけVRAMを消費する。

## 設計哲学として捉える

MoEが示す本質的なアイデアは、「どんな問題にも同じリソースを投入しなくていい」という分業の思想だ。

これはソフトウェアエンジニアリングにおけるマイクロサービスの発想と似ている。モノリシックなシステムにすべてを詰め込むのではなく、専門化されたコンポーネントが適切なタスクを担う。ただしMoEの場合、どのコンポーネントを呼び出すかはルールベースではなく、データから学習されたGating Networkが決定する。

あるいは人間の脳の比喩も使われる。特定のタスクを行うとき、脳のすべての領域が同時に最大活動するわけではない。状況に応じて関連する領域が優先的に働く。MoEはその「必要な部分だけを動員する」という効率性を、人工ニューラルネットワークで実現しようとする試みだ。

---

## Denseモデルとの選択

MoEがあればDenseモデルは不要か、という問いに対する答えは「No」だ。

Denseモデルはシンプルで安定している。学習が予測しやすく、実装が単純で、Expert偏在のような特有の問題を持たない。小規模から中規模のモデルではDenseの方が扱いやすい場面も多い。

MoEが真価を発揮するのは、「計算コストを抑えながら大規模な知識を持たせたい」という状況だ。極めて大きなモデルを動かす必要があるが、推論コストには制約がある——そういう文脈でMoEは強力な選択肢となる。

---

## まとめ

Mixture of Expertsは「すべてのパラメータを毎回使う必要はない」という逆転の発想から生まれたアーキテクチャだ。総パラメータ数と活性化パラメータ数を分離することで、大きなモデルを効率的に動かす可能性を切り開いている。

Expert偏在や分散通信の課題は残るが、Mixtralのような実用的なモデルの登場によって、MoEは理論から実践の段階に移行した。LLMの設計を考えるうえで、MoEは今後ますます重要な概念になっていくだろう。
