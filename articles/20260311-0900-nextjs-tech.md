## なぜ「モノレポ」が必要になるのか

アプリケーションが成長するにつれ、コードの管理単位についての問いは避けられなくなる。最初はひとつのNext.jsプロジェクトで完結していたものが、管理画面、ユーザー向けサイト、社内ツールと分化していくと「同じロジックをどこに置くか」という問題が繰り返し浮上してくる。

その答えのひとつがモノレポ（monorepo）だ。複数のアプリケーションやライブラリを単一のリポジトリで管理するこのアプローチは、「コードの重複を排除したい」という欲求から生まれる。しかしモノレポはただリポジトリをまとめるだけではない。それはコードの境界と共有の範囲を設計するための構造的な意思決定だ。

TurborepoはそのモノレポをNext.jsプロジェクトに持ち込むための、現時点で最も洗練されたツールのひとつだ。Vercelが開発・メンテナンスしていることもあり、Next.jsとの相性は自然に高く保たれている。

---

## Turborepoが解決する問題

モノレポの難しさは「何でも置けてしまう」ことに起因する。共有ロジックを`packages/`に置けるのはいいが、依存関係が複雑に絡み合うとビルドの順番が崩れ、変更の影響範囲が読めなくなる。

従来のツール（Lernaなど）はこの問題に「パッケージのリンク」という形で対応してきたが、ビルドの並列化やキャッシュには限界があった。

Turborepoが持ち込んだのは**タスクグラフとキャッシュ**という考え方だ。

タスク（ビルド、テスト、リント）をグラフとして定義し、依存関係を静的に解析することで「何を先に実行すべきか」を自動的に判断する。さらにタスクの出力をハッシュベースでキャッシュし、入力が変わっていなければ再実行しない。この仕組みが「モノレポの遅さ」という定評を覆した。

## 構造を設計する——appsとpackages

Turborepoプロジェクトの典型的な構造は以下のようになる。

```
my-project/
├── apps/
│   ├── web/          # ユーザー向けNext.jsアプリ
│   └── admin/        # 管理画面Next.jsアプリ
├── packages/
│   ├── ui/           # 共有UIコンポーネント
│   ├── config/       # ESLint, TypeScript設定
│   └── lib/          # 共有ロジック・ユーティリティ
├── turbo.json
└── package.json
```

重要なのは`apps/`と`packages/`の分離だ。アプリはデプロイ可能な成果物であり、パッケージはアプリに消費されるライブラリだ。この区分けを明確に保つことで、依存の方向が自然に「パッケージ → アプリ」の一方向になる。

逆方向の依存（アプリがアプリを参照する）は循環依存を生み出す温床となるため、構造で防ぐことが重要だ。

## turbo.jsonで「ビルドの意図」を宣言する

Turborepoの中核は`turbo.json`だ。ここにタスクの依存関係と出力を定義する。

```json
{
  "$schema": "https://turbo.build/schema.json",
  "tasks": {
    "build": {
      "dependsOn": ["^build"],
      "outputs": [".next/**", "dist/**"]
    },
    "dev": {
      "cache": false,
      "persistent": true
    },
    "lint": {},
    "test": {
      "outputs": ["coverage/**"]
    }
  }
}
```

`"dependsOn": ["^build"]`という記法が重要だ。`^`はトポロジカルな依存を意味する——「このタスクを実行する前に、依存しているパッケージの`build`を先に完了させろ」という指示だ。

これにより、`packages/ui`のビルドが完了してから`apps/web`のビルドが始まるという順序が自動的に保証される。開発者が手動でビルド順を管理する必要はない。

`cache: false`と`persistent: true`は開発サーバー（`next dev`）のための設定だ。開発サーバーはプロセスが常駐するもので、キャッシュの対象にならない。

## 共有パッケージの設計思想——何を共有すべきか

モノレポで最も設計の判断を要するのは「何を共有パッケージにするか」だ。

**共有に向くもの:**

- **UIコンポーネントライブラリ**: ボタン、フォームフィールドなど複数アプリで見た目を統一したいコンポーネント。ただし「汎用的すぎるコンポーネント」になりがちな罠がある
- **型定義**: APIレスポンスの型、ドメインモデルの型。複数アプリが同じバックエンドと通信するなら共有の恩恵は大きい
- **ユーティリティ関数**: 日付フォーマット、バリデーションロジックなど
- **設定ファイル**: TypeScriptの`tsconfig.json`基底設定、ESLintのベース設定

**共有を避けるべきもの:**

- アプリ固有のビジネスロジック。「共有したいから抽象化する」ではなく、「本当に同じロジックか」を問い直すべきだ
- 状態管理の具体実装。状態はアプリのライフサイクルに密接なため、共有すると意図しない結合を生む

## Next.jsとの統合で気をつけること

Turborepoの`packages/ui`にReactコンポーネントを置くとき、Next.jsのApp RouterとServer Componentsを念頭に置く必要がある。

共有コンポーネントがデフォルトでサーバーコンポーネントとして振る舞うべきか、クライアントコンポーネントとして振る舞うべきかを明示的に設計する。`"use client"`ディレクティブは共有パッケージ側で付与するケースと、消費側のアプリで付与するケースがある。

一般的な方針として、インタラクティブな要素（フォーム、ボタン）は共有コンポーネント側で`"use client"`を宣言し、データ取得を持たない表示専用コンポーネントはデフォルトのサーバーコンポーネントにしておくのが扱いやすい。

また、`next/font`や`next/image`はNext.js特有のAPIであるため、これらを共有コンポーネント内で直接使うとNext.jsに依存する共有パッケージが生まれる。「共有パッケージはNextに依存しない」という原則を守りたいなら、これらの利用はアプリ層に留め、共有コンポーネントにはプロップスとして渡す設計にする。

---

## リモートキャッシュで「チームのビルド時間」を短縮する

Turborepoのキャッシュはローカルだけでなくリモートにも保存できる。Vercelのリモートキャッシュを使えば、チームの誰かがビルドした結果を他の開発者のマシンやCIでも再利用できる。

これはモノレポが大規模になるほど効果が増す。コアパッケージを変更していないのに毎回全ビルドが走るのは非効率だ。リモートキャッシュによって「誰かが既にビルドしたなら、その結果を使う」という合理的な判断が自動化される。

---

## モノレポは「組織の境界」を映す鏡

コンウェイの法則が示すように、システムのアーキテクチャはそれを作る組織の構造を反映する。モノレポの設計も例外ではない。

誰が何を担当するか、どのチームがどのパッケージを所有するか、変更の影響範囲をどう管理するか——これらはすべて技術的な決定と組織的な決定が交差する地点だ。

TurborepoはそのモノレポをNext.jsプロジェクトで扱いやすくするための強力なツールだが、ツールが組織の混乱を解決してくれるわけではない。構造を設計する前に「なぜコードを共有したいのか」「誰が共有コードのオーナーか」を問い直すことが、長期的に健全なコードベースを保つための第一歩だ。

モノレポはコードを「まとめる」ためではなく、コードの「関係を整理する」ために使うものだ。その意識でTurborepoに向き合うとき、設計の選択肢が格段に広がる。
