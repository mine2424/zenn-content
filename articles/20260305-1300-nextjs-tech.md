## 検索エンジンはHTMLをどう読むのか

Webページを人間が読むとき、見出しや本文の視覚的なレイアウトが理解を助ける。しかし検索エンジンのクローラーにとって、HTMLはあくまでマークアップの集合体だ。「このページは料理のレシピか、企業の会社概要か、商品の詳細ページか」を正確に理解するのは、HTMLの構造だけでは難しい。

JSON-LDは、ページのコンテンツに対して「このデータは何を意味するか」というセマンティクス（意味論的情報）を付加するための仕様だ。schema.orgが定義するボキャブラリーを使い、検索エンジンが機械的に解釈できる形で構造化データを記述する。

Googleのリッチリザルト（レビュースターの表示、FAQの展開、パンくずリストのインライン表示など）は、多くの場合このJSON-LDによって実現されている。

## JSON-LDがMicrodataやRDFaより選ばれる理由

構造化データの記述形式にはMicrodata（HTML要素のattributeとして記述）やRDFaもあるが、現在のベストプラクティスはJSON-LDだ。

理由はシンプルで、HTMLコンテンツとデータ定義を分離できるからだ。MicrodataはHTMLの各要素に属性を付与するため、マークアップとデータ定義が混在し、テンプレートの保守性が落ちる。JSON-LDは`<script type="application/ld+json">`タグの中にJSONとして記述するだけなので、コンテンツのHTMLとデータの定義を独立して管理できる。

Googleも公式にJSON-LDを推奨しており、動的なデータの注入も容易だ。

## Next.js App Routerでの組み込み

Next.js App Routerでは、`generateMetadata`と並行してJSON-LDを`layout.tsx`または`page.tsx`に組み込む方法が一般的だ。

```tsx
// app/recipes/[slug]/page.tsx
export default async function RecipePage({ params }: { params: { slug: string } }) {
  const recipe = await fetchRecipe(params.slug);

  const jsonLd = {
    "@context": "https://schema.org",
    "@type": "Recipe",
    name: recipe.title,
    description: recipe.description,
    author: {
      "@type": "Person",
      name: recipe.author,
    },
    datePublished: recipe.publishedAt,
    image: recipe.imageUrl,
    recipeIngredient: recipe.ingredients,
    recipeInstructions: recipe.steps.map((step, i) => ({
      "@type": "HowToStep",
      position: i + 1,
      text: step,
    })),
    aggregateRating: {
      "@type": "AggregateRating",
      ratingValue: recipe.rating,
      reviewCount: recipe.reviewCount,
    },
  };

  return (
    <>
      <script
        type="application/ld+json"
        dangerouslySetInnerHTML={{ __html: JSON.stringify(jsonLd) }}
      />
      <RecipeDetail recipe={recipe} />
    </>
  );
}
```

`dangerouslySetInnerHTML`を使うことに不安を感じるかもしれないが、`JSON.stringify`でエスケープされた内容を注入するのでXSSリスクは低い。ただし外部から取得したデータをそのまま埋め込む場合は、信頼できるソースかどうかを確認する必要がある。

## 主なスキーマタイプとユースケース

schema.orgは800以上のタイプを定義しているが、実務でよく使うものは限られる。ユースケースに対応する主要タイプを整理する。

**Article（技術ブログ、ニュース記事）**

ブログ記事や技術ドキュメントに使うタイプだ。`headline`、`author`、`datePublished`、`dateModified`を適切に設定することで、Google Discoverへの表示機会が高まる。Zennのような技術ブログプラットフォームも内部でArticleスキーマを活用している。

**BreadcrumbList（パンくずリスト）**

サイトのナビゲーション階層を表すスキーマだ。検索結果にURLの代わりにパンくずが表示されるリッチリザルトが有効になる。ECサイトやドキュメントサイトで特に効果が大きい。

**FAQPage（よくある質問）**

QとAのペアを記述することで、検索結果ページでFAQが展開表示されることがある。サービスのFAQページや製品の詳細ページで活用される。ただし2023年以降、Googleのリッチリザルト表示対象が限定されてきているため、最新の仕様を確認しながら使う必要がある。

**Product（商品ページ）**

ECサイトの商品ページで使うタイプ。価格、在庫状況、レビューを構造化することで、Googleショッピングへの掲載機会が生まれる。`offers`と`aggregateRating`が特に重要なプロパティだ。

**SoftwareApplication（アプリ）**

モバイルアプリやSaaSのランディングページに使う。App StoreやPlay Storeのリンク、評価、対応OSを記述できる。

## 複数のJSON-LDを同一ページに共存させる

1つのページに複数の構造化データが必要なことは珍しくない。例えばECサイトの商品ページには、Productスキーマ（商品情報）とBreadcrumbListスキーマ（ナビゲーション）を両方置く必要がある。

やり方は2つある。それぞれ別の`<script>`タグとして記述するか、単一の`<script>`タグ内に配列として記述するかだ。

```json
[
  {
    "@context": "https://schema.org",
    "@type": "Product",
    "name": "サンプル商品"
  },
  {
    "@context": "https://schema.org",
    "@type": "BreadcrumbList",
    "itemListElement": [...]
  }
]
```

Googleはどちらのアプローチもサポートしているので、コードの整理しやすい方を選べばいい。

## 型安全に書くためのアプローチ

TypeScriptを使っている場合、`schema-dts`パッケージを使うとJSON-LDのスキーマに型が付く。

```bash
npm install schema-dts
```

```tsx
import type { Article, WithContext } from "schema-dts";

const jsonLd: WithContext<Article> = {
  "@context": "https://schema.org",
  "@type": "Article",
  headline: "記事タイトル",
  author: {
    "@type": "Person",
    name: "著者名",
  },
  datePublished: "2026-03-05",
};
```

スキーマの必須プロパティや型の誤りをビルド時に検出できるので、JSON-LDの記述ミスによるリッチリザルトの非表示を防ぎやすくなる。

## 検証と効果確認の方法

JSON-LDを書いたら、以下のツールで検証する習慣をつけるといい。

**Google リッチリザルトテスト**（search.google.com/test/rich-results）: ページのURLを入力するか、HTMLを貼り付けることで、Googleがどのリッチリザルトを認識したかを確認できる。エラーと警告も表示されるので、記述ミスの発見に役立つ。

**Schema.org Validator**（validator.schema.org）: schema.orgの仕様に基づいてJSON-LDの文法的な正しさを検証する。Googleの仕様とは微妙に異なる部分もあるが、基本的な検証に使える。

**Google Search Console（コアウェブバイタルと構造化データレポート）**: 本番環境でクロールされた後、「拡張」セクションで各スキーマタイプの認識状況とエラーが確認できる。効果が出るまでに時間がかかるが、実際にリッチリザルトが適用されたかを追跡できる唯一のデータソースだ。

## 「SEOのためだけ」に終わらせない

JSON-LDはGoogleに限らず、Bing、Pinterestなどのプラットフォームも活用している。また、将来的にAIエージェントがWebページを読み込んで情報を抽出するとき、構造化データはより確実に機械が理解できる情報源になり得る。

「検索エンジンのランキングを上げる施策」としてだけでなく、「Webページのデータを機械可読な形で公開する」という観点でJSON-LDを捉えると、実装の優先度の判断がしやすくなる。コンテンツの意味を明示することは、今後のWebのインフラとして重要度が増していく方向性の一部だ。

## まとめ

JSON-LDはHTMLコンテンツに「意味」を付加する仕組みだ。Next.js App Routerでは`page.tsx`に`<script type="application/ld+json">`として注入し、`schema-dts`で型安全に記述できる。スキーマタイプの選択はユースケースに依存するが、Article、BreadcrumbList、Productから始めると実装の効果を実感しやすい。Google リッチリザルトテストとSearch Consoleを使った検証サイクルを作ることで、実装の品質と効果の両方を継続的に追跡できる。
