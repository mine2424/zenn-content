## はじめに

Next.js App Routerのレンダリング戦略を考えるとき、多くの開発者はSSR（サーバーサイドレンダリング）かSSG（静的サイト生成）か、あるいはISRか、という選択を迫られてきた。

しかし、この選択は「ページ単位」でしかできなかった。ダッシュボードのようなページで「ヘッダーは静的に、コンテンツは動的に」やりたくても、ページ全体がどちらかになってしまう制約があった。

**PPR（Partial Prerendering）**はこの制約を解消する、Next.jsの新しいレンダリングモデルだ。

## PPRの基本的な考え方

PPRのコアとなるアイデアはシンプルだ。**1つのページの中で、静的な部分と動的な部分を混在させる**。

従来のレンダリングモデルでは、ページに1つでも動的なデータ取得（クッキーの読み取り、リクエスト時のデータフェッチなど）があると、ページ全体がSSRになった。パーソナライズされた情報を1つ表示したいだけで、ページ全体の初期表示が遅くなるという問題があった。

PPRはこの問題を「静的シェル」と「動的アイランド」という概念で解決する。

- **静的シェル**: ビルド時に生成され、即座にユーザーに届く部分
- **動的アイランド**: リクエスト時に処理され、Suspenseでストリーミングされる部分

ユーザーはまず静的シェルを受け取り、動的な部分がストリーミングで埋まっていく。

## 具体的な仕組み

PPRはReactの`Suspense`を活用している。`Suspense`で囲まれたコンポーネントが動的なアイランドとして扱われる。

```tsx
// app/dashboard/page.tsx
import { Suspense } from 'react'
import { StaticHeader } from './components/StaticHeader'
import { UserGreeting } from './components/UserGreeting'
import { RecentActivity } from './components/RecentActivity'

export default function DashboardPage() {
  return (
    <main>
      {/* ビルド時に静的生成される */}
      <StaticHeader />
      
      {/* リクエスト時にストリーミングされる動的部分 */}
      <Suspense fallback={<GreetingSkeleton />}>
        <UserGreeting />
      </Suspense>
      
      <Suspense fallback={<ActivitySkeleton />}>
        <RecentActivity />
      </Suspense>
    </main>
  )
}
```

`UserGreeting`がクッキーからユーザーIDを読み取っても、`RecentActivity`がDBから最新データを取得しても、ページ全体のSSR化は起きない。静的シェルはビルド時に生成済みで、動的部分だけが実行時に処理される。

## 従来のモデルとの違い

### SSG（静的生成）との違い

SSGはページ全体をビルド時に生成する。動的なデータは含めることができない（ISRで更新はできるが、リクエスト時のデータは持てない）。

PPRは静的な部分はSSGと同様にビルド時に生成しつつ、動的な部分はリクエスト時に追加できる。

### SSRとの違い

SSRはリクエストごとにページ全体を生成する。初期表示までの時間（TTFB）が長くなりがちで、サーバー負荷も高い。

PPRはビルド済みの静的シェルを即座に返せるため、TTFBは静的サイト並みになる。動的部分はその後にストリーミングされる。

### Streaming SSRとの違い

Next.js App RouterはSSRとSuspenseを組み合わせたStreaming SSRをすでにサポートしている。これとPPRは何が違うのか。

Streaming SSRは「リクエストを受けてからストリーミングを開始する」が、PPRは「静的シェルはCDNにキャッシュされていて、リクエストなしで即座に返せる」という違いがある。TTFBがゼロに近づく点がPPRの強みだ。

## パフォーマンスへの影響

PPRがもたらすパフォーマンス改善を理解するために、Core Web Vitalsの観点から考えてみる。

**LCP（Largest Contentful Paint）の改善**: ページの静的シェルがCDNから即座に返ってくるため、メインコンテンツの表示が早くなる。特にヒーローエリアが静的コンテンツの場合、LCPが劇的に改善する。

**FID / INP（インタラクション応答性）**: 静的シェルが早く届くことで、ページのhydrationも早く完了し、インタラクションへの応答が改善する。

**CLS（Cumulative Layout Shift）の注意点**: 動的部分がストリーミングで差し込まれるとき、レイアウトシフトが起きないようにSuspenseのフォールバックUIを適切に設計する必要がある。SkeletonUIを使うのが一般的なアプローチだ。

## どんなページに向いているか

PPRが最も効果を発揮するのは、**静的な要素と動的な要素が混在するページ**だ。

典型的なユースケース：
- ダッシュボード（ナビゲーションは静的、データは動的）
- Eコマースの商品詳細ページ（商品情報は比較的静的、在庫状況・ユーザーレビューは動的）
- ニュースサイト（レイアウト・広告枠は静的、パーソナライズされた推薦記事は動的）
- SaaS製品の設定ページ（UIの骨格は静的、ユーザー設定データは動的）

逆に、ページ全体が完全に静的なコンテンツであればSSGで十分だし、ページ全体がリアルタイムデータに依存するならSSRが適切だ。

## 設計上の注意点

PPRを使う際に意識すべき設計方針がある。

**Suspenseの境界設計が重要になる**: どこで静的と動的を分けるかが、パフォーマンスと開発体験に直結する。粒度を細かくしすぎると管理が複雑になり、大きくすると動的部分が増えてPPRの恩恵が薄れる。

**フォールバックUIをスケルトンで設計する**: 動的部分がストリーミングされる間、ユーザーに示すプレースホルダーが必要だ。コンテンツの形状に合ったスケルトンを用意することで、体感的なローディング速度が改善する。

**データフェッチの並列性を意識する**: 複数の動的コンポーネントがそれぞれAPIを呼ぶとき、並列実行できるように設計することでストリーミング完了を早められる。

## まとめ

PPRは「静的か動的か」という二択を解消し、**1つのページの中でベストなレンダリング戦略を部分ごとに適用できる**ようにした。

これはHTTPとReact Suspenseの組み合わせで初めて可能になったアーキテクチャだ。ページという単位でレンダリング戦略を考えるのをやめ、「どのコンポーネントが本当に動的なのか」を問い直すことが、PPR時代のNext.js設計の出発点になる。
