## はじめに

Lighthouseを「Google Chromeのデベロッパーツールで動かすやつ」として認識しているエンジニアは多い。スコアを見て「100点近くになった、よし」と満足して終わる——そのサイクルが続いている現場も珍しくない。

しかしLighthouseはCIパイプラインに組み込み、継続的に監視する「品質ゲート」として設計したとき、初めてその真価を発揮する。この記事では、Lighthouseをアドホックな計測ツールから脱却させ、QAプロセスの中核に置くための設計思想を整理する。

---

## Lighthouseが計測するのはスコアではなく「ユーザー体験の指標」だ

Lighthouseのスコアそのものに意味があるわけではない。スコアは以下の6つのカテゴリで構成される：

- **Performance**：FCP、LCP、TTI、TBT、CLSなどのWebVitals
- **Accessibility**：ARIAの適切な使用、コントラスト比、キーボード操作
- **Best Practices**：HTTPSの使用、古いAPIの不使用など
- **SEO**：クロール可能性、メタ情報の適切な設定
- **Progressive Web App（PWA）**：オフライン対応、マニフェスト設定

スコアが高いことより、「どの指標が何を意味するか」を理解することが先だ。

例えばPerformanceスコアを下げる最大要因は**TBT（Total Blocking Time）**と**LCP（Largest Contentful Paint）**だ。TBTはメインスレッドがブロックされている総時間を示し、ユーザーが操作できない時間の長さに直結する。LCPはページの主要コンテンツが表示されるまでの時間だ。

「スコアを上げる」ではなく「TBTを100ms以下に抑える」「LCPを2.5秒以内にする」という具体的な指標で会話することで、QA活動が設計改善に直結するようになる。

---

## なぜ手動実行では限界があるのか

Lighthouseを手動で動かす問題点は計測環境のばらつきだ。

同じページでも、ネットワーク状況・CPUパワー・拡張機能の有無・ブラウザのキャッシュ状態によってスコアが大きく変動する。「昨日98点だったのに今日83点になった」という報告が品質改善の議論ではなく原因調査に時間を取られる原因になる。

CI環境で実行することで、この問題は大幅に解消される。Dockerコンテナ上でChrome headlessを固定のCPU・メモリ・ネットワーク条件下で動かすことで、ベースライン比較が可能になる。

```bash
# lighthouse-ciの基本的な設定例
npx @lhci/cli autorun \
  --collect.url="http://localhost:3000" \
  --collect.numberOfRuns=3 \
  --assert.preset=lighthouse:recommended
```

`numberOfRuns=3`のように複数回計測して中央値を取ることも、計測の安定性を高める重要な設計判断だ。

## 「品質ゲート」として設計するとはどういうことか

品質ゲートとは「このラインを下回るとマージできない」という閾値の定義だ。Lighthouse CIには`assert`機能があり、各指標に対して閾値を設定できる。

`.lighthouserc.json`の例：

```json
{
  "ci": {
    "assert": {
      "assertions": {
        "categories:performance": ["error", {"minScore": 0.8}],
        "categories:accessibility": ["error", {"minScore": 0.9}],
        "first-contentful-paint": ["warn", {"maxNumericValue": 2000}],
        "largest-contentful-paint": ["error", {"maxNumericValue": 2500}],
        "total-blocking-time": ["error", {"maxNumericValue": 300}],
        "cumulative-layout-shift": ["error", {"maxNumericValue": 0.1}]
      }
    }
  }
}
```

ここで重要なのは`error`と`warn`の使い分けだ。

- `error`：PRのCIを失敗させる。マージ不可にする基準。
- `warn`：警告は出るが継続する。注意喚起レベル。

全てを`error`にすると開発速度が落ちる。全てを`warn`にすると誰も改善しない。どこを「必ず守る線」にするかの判断がQA設計の核心だ。

一般的な考え方として、Core Web VitalsのCLS（累積レイアウトシフト）とLCPはユーザー体験への影響が大きいため`error`にし、FID（First Input Delay）やアクセシビリティの細かい項目は最初`warn`から始めて徐々に引き上げる、という段階的なアプローチが現実的だ。

## ページの種類別に閾値を分ける

全てのページに同じ閾値を適用するのは無理がある。ダッシュボードのような機能が密集したページは、ランディングページと同じPerformanceスコアを求めるのは現実的でない。

Lighthouse CIでは`url`ごとに閾値を分けることができる：

```json
{
  "ci": {
    "collect": {
      "urls": [
        "http://localhost:3000/",
        "http://localhost:3000/dashboard",
        "http://localhost:3000/blog"
      ]
    },
    "assert": {
      "assertMatrix": [
        {
          "matchingUrlPattern": ".*/$",
          "assertions": {
            "categories:performance": ["error", {"minScore": 0.9}]
          }
        },
        {
          "matchingUrlPattern": ".*/dashboard",
          "assertions": {
            "categories:performance": ["warn", {"minScore": 0.7}]
          }
        }
      ]
    }
  }
}
```

トップページや記事ページのような「最初にユーザーが見るページ」は高い基準を設け、管理画面や複雑なダッシュボードは現実的な基準を設ける。この使い分けが設計の質を高める。

## 計測タイミングを設計する

「いつ計測するか」もQA設計の重要な判断だ。

**PRごとに計測する**：フィーチャーブランチがPerformanceに与える影響をマージ前に検出できる。JavaScript Bundleサイズが増えてLCPが劣化するケースを早期に発見できる。

**デプロイ後に本番計測する**：CI環境と本番環境は異なる。CDNキャッシュ・画像最適化・サードパーティスクリプトの影響は本番でしか正確に測れない。デプロイフックでLighthouseを実行し、結果をSlackやDatadogに送る設計が理想だ。

**定期的にスケジュール計測する**：サードパーティスクリプト（アナリティクスや広告）の変更がいつの間にかパフォーマンスを劣化させるケースがある。週次・日次でのバッチ計測と結果の時系列モニタリングが重要だ。

---

## スコアを追わず「問題の根本」を追う

Lighthouseの最も価値ある機能の一つは、スコアとともに提示される「改善提案（Opportunities / Diagnostics）」だ。

「render-blocking resources」「unused JavaScript」「unoptimized images」——これらの指摘は、スコアが何点かではなく「何がユーザー体験を妨げているか」を具体的に示してくれる。

QAエンジニアの役割は、スコアを報告することではなく、このDiagnosticsを開発チームと共有して「何を直すか」の議論をファシリテートすることだ。「LCPが遅い」という報告ではなく「LCPの原因がHeroイメージのlazy-load設定にある」という発見を持ち込む。

---

## まとめ

Lighthouseは「一度実行して満足するツール」でも「スコアを目標にするゲーム」でもない。CIパイプラインに組み込み、ページ種別に閾値を設計し、デプロイ後も継続計測することで、パフォーマンスを「品質の一軸」として継続的に管理できるようになる。

スコアを追うのではなく、ユーザー体験を構成する指標を追う。その設計思想がLighthouseを本当の意味で活かすための前提だ。
