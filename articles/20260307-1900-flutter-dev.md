# Flutterのシェーダーコンパイルジャンクを理解してSkSLで解決する

Flutterアプリを実機で動かしたとき、最初のアニメーション再生時だけカクつく——という経験をしたことはないだろうか。ProfileモードやReleaseモードで試すと発生するが、Debugモードでは再現しないことが多い。

これは**シェーダーコンパイルジャンク（Shader Compilation Jank）**と呼ばれる問題だ。Flutterのレンダリングパイプラインとシェーダーの関係を理解することで、この問題の本質が見えてくる。

---

## なぜシェーダーがジャンクを引き起こすのか

FlutterはSkiaまたはImpellerを使ってGPUにレンダリングを委ねる。GPUが画像を描画するには、**シェーダープログラム**（小さなGPU上で動くプログラム）が必要だ。

問題は、シェーダーはGPUにアップロードする前に「コンパイル」が必要な点だ。このコンパイルは初回描画時にGPUドライバーが行うため、**そのフレームの描画がブロックされる**。

16.67ミリ秒（60fps）の制約の中でこのコンパイルが発生すると、そのフレームは間に合わず「ジャンク（コマ落ち）」として現れる。一度コンパイルされたシェーダーはキャッシュされるため、2回目以降は問題が起きない。これが「初回だけカクつく」現象の原因だ。

Debugモードで発生しにくいのは、JITコンパイル環境のため実行パスが異なることと、Debugビルドの描画パスがシンプルなためだ。

---

## Impellerの登場と問題の変化

Flutter 3.x系から**Impeller**というレンダリングエンジンがiOS向けにデフォルトになった（Android向けも順次移行中）。

Impellerはシェーダーコンパイルジャンク問題を根本から解決しようというアプローチだ。Flutterアプリのビルドプロセスでシェーダーをあらかじめコンパイルしておき（AOTシェーダーコンパイル）、実行時のオンデマンドコンパイルを不要にする。

つまり**Impellerを使っていれば、シェーダーコンパイルジャンクは原理的に発生しない**。

しかし現実は複雑だ。Impellerは全プラットフォームで完全に有効になっているわけではなく、既存アプリでSkiaが使われているケースや、Impellerに移行したものの特定のウィジェット・アニメーションで問題が残るケースもある。そのため今もシェーダー最適化の知識は実用的な価値を持つ。

---

## SkSLとは何か

**SkSL（Skia Shader Language）**は、SkiaがGPUシェーダーを記述するために使う中間シェーダー言語だ。GLSL（OpenGL Shading Language）をベースにしている。

FlutterがSkiaを使う場合、描画命令はSkSLのシェーダープログラムにコンパイルされ、それがさらにGPUドライバー固有のバイナリにコンパイルされる。ジャンクが発生するのは、この2段階目のコンパイルがフレーム描画時に起きるためだ。

SkSLウォームアップとは、アプリ起動時にあらかじめシェーダーをコンパイルしてキャッシュしておく手法だ。事前に「このシェーダーが必要になる」とFlutterに伝えることで、実際の描画前にコンパイルを済ませておく。

---

## SkSLシェーダーウォームアップの実践

Skiaを使っているFlutterアプリでシェーダーウォームアップを行うには、まずシェーダーを「録画」する必要がある。

```bash
# Profileモードで起動し、シェーダーをキャプチャ
flutter run --profile --cache-sksl --purge-persistent-cache
```

アプリを起動してジャンクが発生しそうな箇所（アニメーション、複雑な描画）を意図的に操作する。その後アプリを終了するとシェーダーキャッシュが保存される。

```bash
# キャプチャしたシェーダーをアセットとしてバンドル
flutter build apk --bundle-sksl-path flutter_01.sksl.json
```

こうしてビルドされたアプリは、起動時にバンドルされたSkSLシェーダーを使ってウォームアップを行い、実行時のコンパイルジャンクを回避する。

---

## DevToolsでジャンクを診断する

シェーダーコンパイルジャンクを定量的に確認するには、Flutter DevToolsのPerformanceビューを使う。

`Profile` モードでアプリを起動し、DevToolsのFlame Chartを見ると、`GrDirectContext::flushAndSubmit` や `shader compile` というラベルが長いフレームで現れることがある。これがシェーダーコンパイルに費やされた時間だ。

また、FlutterのPerformance Overlayを有効にしてGPUスレッドを監視する方法もある。

```dart
MaterialApp(
  showPerformanceOverlay: true,
  // ...
)
```

赤いバーが頻繁に現れる場合は、GPUスレッドでのジャンクが疑われる。シェーダーコンパイル以外にも、重い描画処理（saveLayer、clipPath、大量のBlur）が原因となることがあるため、Flame Chartと合わせて原因を特定していく。

---

## カスタムシェーダーの活用と注意点

Flutter 3.x以降、独自の**フラグメントシェーダー**をGLSL/SkSLで書いてFlutterアプリに組み込めるようになった。グラデーション、ディストーション、エフェクトなどをGPUで直接描画できる。

```yaml
# pubspec.yaml
flutter:
  shaders:
    - shaders/my_effect.frag
```

カスタムシェーダーを使う場合、そのシェーダー自体もコンパイルジャンクの対象になりうる。アプリ起動時に意図的にシェーダーをウォームアップする処理を入れることが推奨される。

ただし、カスタムシェーダーはImpellerでの対応状況や、プラットフォームごとの差異に注意が必要だ。まずImpellerが有効かどうかを確認し、ドキュメントで最新の対応状況をチェックすることを推奨する。

---

## パフォーマンス問題全般への向き合い方

シェーダーコンパイルジャンク以外にも、Flutterのレンダリングパフォーマンスに影響する要因は多い。

`saveLayer` の多用はGPUにとって重く、できる限り避けるかRepaintBoundaryで適切に分離する。`clipPath` も同様にコストが高い。`Image.network` で大きな画像をそのまま表示するとデコードがジャンクを引き起こすことがある。

こうした問題はシェーダーとは別次元の最適化だが、DevToolsのFlame Chartで同様に可視化できる。パフォーマンス改善の第一歩は「計測して原因を特定する」こと——推測でコードを変えることは、時間を無駄にするだけでなく問題を見えにくくすることもある。

---

## まとめ：レンダリングの「時間的コスト」を意識する

シェーダーコンパイルジャンクは、GPUに処理を委ねる際の「準備コスト」が描画タイミングに集中することで起きる。この問題への対処は「コストを前倒しする」という思想——起動時やウォームアップフェーズでコンパイルを済ませ、ユーザーが体験する描画の瞬間はスムーズにする。

Impellerへの移行が進む中で、この問題は自然に解消されていく方向にある。しかし「なぜジャンクが起きるのか」「GPUとシェーダーの関係はどうなっているか」を理解することは、Flutterのレンダリングパイプライン全体を深く理解することにつながり、次の問題を解くための基礎になる。
