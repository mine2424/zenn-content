## はじめに

Next.js App Routerの登場によって、Reactのコンポーネントは「サーバーで動くもの」と「クライアントで動くもの」に明確に分かれました。この分類はパフォーマンスと開発体験に大きなメリットをもたらす一方、「どこで境界を引くか」という設計判断が常につきまといます。

「とりあえず`'use client'`をつけておけばいいや」という判断をしていると、Server Componentsのメリットを失うだけでなく、パフォーマンスの低下やデータフェッチの複雑化につながります。

この記事では、Server Componentsの根本的な思想を理解し、「境界」を正しく引くための思考法について整理します。

## なぜServer Componentsが生まれたか

Server Componentsを正しく使うには、まず「なぜこれが生まれたか」を理解することが重要です。

従来のReactアプリケーション（特にCSR: Client Side Rendering）の問題点は明確でした。

1. **クライアントに大量のJavaScriptを送る必要がある**: ライブラリも含めた全てのコードがブラウザに届く
2. **データフェッチのウォーターフォール**: コンポーネントがマウントしてからフェッチが始まるため、ローディング状態が積み重なる
3. **機密情報をクライアントに露出するリスク**: APIキーやデータベース接続情報をクライアントで使えない

Server Componentsはこれらを根本から解決するアプローチです。コンポーネントをサーバーで実行することで、クライアントへのJavaScript送信量を減らし、データフェッチをより近い場所（サーバー）で行い、機密情報を安全に扱えます。

つまりServer Componentsは「パフォーマンスのための最適化」ではなく、「正しいデータフェッチのアーキテクチャ」という本質があります。

## 「サーバー」と「クライアント」の本質的な違い

境界を正しく引くには、2つの実行環境の本質的な違いを理解する必要があります。

**サーバーコンポーネント（デフォルト）:**

- データベースやファイルシステムへの直接アクセスが可能
- 環境変数（シークレット含む）を安全に参照できる
- 実行は1回だけ（レンダリング時）
- ブラウザのAPIは使えない（`window`, `localStorage`, etc.）
- インタラクティブ性はない（イベントハンドラ、useState、useEffect不可）

**クライアントコンポーネント（`'use client'`宣言）:**

- ブラウザAPIへのアクセスが可能
- インタラクティブな操作（クリック・入力・アニメーション）ができる
- stateやエフェクトが使える
- データベースに直接アクセスできない
- JavaScriptバンドルに含まれる

この違いを見ると、判断基準が明確になります。「このコンポーネントはブラウザのインタラクションが必要か？」という問いが、境界を引く最初のフィルターです。

## 境界を引く3つの判断基準

**判断基準1: インタラクティブ性が必要か**

ボタンのクリック、フォームの入力、ホバーエフェクト——これらはブラウザのイベントシステムに依存するため、クライアントコンポーネントでなければなりません。逆に言えば、「ただ表示するだけ」のコンポーネントは原則としてサーバーコンポーネントにできます。

ナビゲーションバーの「現在のページをハイライトする」機能はクライアントが必要ですが、ナビゲーションバーのリンクリストを表示するだけなら、サーバーコンポーネントで十分です。

**判断基準2: ブラウザAPIに依存するか**

`window.scrollY`, `navigator.userAgent`, `localStorage`など、ブラウザ固有のAPIを使う場合はクライアントコンポーネントが必要です。URLパラメータの読み取り（`useSearchParams`）や、現在のルートの確認（`usePathname`）もクライアント専用フックです。

**判断基準3: リアルタイムの状態更新が必要か**

`useState`や`useEffect`を使って、コンポーネント内で状態を持ち、更新するものはクライアントコンポーネントです。サーバーコンポーネントは「1回レンダリングして終わり」なので、状態の変化に反応することができません。

## 「境界を押し下げる」という設計原則

Server Componentsを最大限活用するための核心的な設計原則があります。それは**「クライアントの境界をできるだけツリーの下層に押し下げる」**ことです。

よくある失敗パターン：

```tsx
// ❌ ページ全体がクライアントになってしまう
'use client';

export default function ProductPage({ id }) {
  const [liked, setLiked] = useState(false);
  // この行のために、重いデータフェッチもクライアント側に...
  const product = await fetchProduct(id); // これはサーバーでできるはず
  return (
    <div>
      <ProductDetail product={product} />
      <LikeButton liked={liked} onToggle={() => setLiked(!liked)} />
    </div>
  );
}
```

正しいパターン：

```tsx
// ✅ サーバーで取得し、インタラクション部分だけをクライアントに分離
// ProductPage.tsx (Server Component)
export default async function ProductPage({ id }) {
  const product = await fetchProduct(id); // サーバーで実行
  return (
    <div>
      <ProductDetail product={product} />
      <LikeButton productId={product.id} /> {/* クライアント境界はここだけ */}
    </div>
  );
}
```

`LikeButton`コンポーネントだけに`'use client'`を付け、データフェッチはサーバーコンポーネントで行う。このパターンにより、JavaScriptバンドルに含まれるコードを最小化しつつ、必要なインタラクティブ性を保てます。

## Composition Patternで柔軟性を保つ

Server ComponentsとClient Componentsを組み合わせる際、「サーバーコンポーネントをクライアントコンポーネントの子として渡す」Composition Patternが非常に有効です。

クライアントコンポーネントの中にサーバーコンポーネントを直接インポートして使うことはできませんが、`children` propsを介して渡すことはできます。

この仕組みを使うと、状態を持つラッパー（クライアント）と、データを取得する中身（サーバー）を明確に分離できます。例えばアコーディオンUIのような「開閉の状態はクライアントで管理」しつつ「中身はサーバーコンポーネント」という構成が実現できます。

## まとめ

Server Componentsの境界を正しく引くための考え方をまとめます。

- Server Componentsは「最適化」ではなく「正しいアーキテクチャ」
- 「インタラクティブ性・ブラウザAPI依存・リアルタイム状態更新」がある場合のみクライアントへ
- クライアントの境界はツリーの下層に押し下げる
- CompositionパターンでサーバーとクライアントをPropsで橋渡しする

「とりあえずuse client」という判断をやめ、「なぜここはクライアントである必要があるか」を問うことが、App Routerを正しく使うための出発点です。
